# Slice DX2 — The Agent Experience Product: THE SCENE-QUERY PROTOCOL (the read half of the SDK loop) — Design

> Autonomous-session spec. Verifiable headlessly on Windows+Vulkan (backend-agnostic pure-CPU JSON goldens — NO Mac
> bake, like `introspect/default_scene.json`). The SECOND slice of FLAGSHIP #31 (THE AGENT EXPERIENCE / AX PRODUCT,
> slice-tag **DX**). DX1 shipped the versioned `agentApi` contract; the command bus (`scene::RunCommands`, driven by
> `--commands`) already does MUTATE/CAPTURE/dump. But an agent can only `dump` the WHOLE scene — there is no
> economical, addressable READ. DX2 adds a **`query` op**: a selective, deterministic JSON read of one entity / the
> stats / the camera+lights, addressed by the same view-order index the mutate ops use — the read half of the
> request/response SDK loop. It also registers the `query` command into the DX1 `agentApi` contract, exercising the
> content-churns / shape-holds design (the contract's `contentHash` changes, `schemaVersion` stays 1). **APPENDS to
> `engine/scene/commands.{h,cpp}`** (`hf::scene`, additive — the existing 9 ops + their bytes unchanged). Branch:
> `slice-dx2`. See [[hazard-forge-ax-roadmap]], [[hazard-forge-introspect-golden-staleness]], [[hazard-forge-docs-style]],
> [[hazard-forge-build-env-vcvars]].

**Goal:** Add a `query` op to the command bus and a clean golden-able driver:
- `{"op":"query","select":"entity","entity":N,"fields":["transform","material","mesh"]?}` → that entity's requested
  components (all if `fields` omitted), addressed by view-order index.
- `{"op":"query","select":"stats"}` → the deterministic counts (entity count, mesh/texture resource counts).
- `{"op":"query","select":"entities"}` → a terse ordered list `[{index, mesh}]` of all entities.
Add `std::string RunQueries(reg, resources, queriesJson)` (returns a deterministic JSON array of `{request, response}`
objects — the clean byte-golden artifact + the file-free test entry) AND wire the `query` op into `RunCommands` (prints
its response to stdout for the live `--commands` loop). Add a `--agent-query <cmds.json> <out.json>` driver that writes
`RunQueries` output. Register the `query` command in the `agentApi.commands` array (DX1). Bake the **strict-zero**
Windows byte-golden `tests/golden/agent/query_responses.json` (+ controller-rebaked `agent_api.json` &
`default_scene.json`, which gain the `query` command + a new `contentHash`).

## Design call: a commands.cpp-local query builder (preserve the no-render-symbols contract); index-addressed; deterministic
`commands.cpp` is deliberately free of vk*/Metal/rhi symbols (commands.h:10-14) and addresses entities by VIEW-ORDER
INDEX via `EntityAtIndex` (commands.cpp:105) — the SAME order `DumpScene` / the editor use. DX2 stays inside that
contract:
- **Build the query response in `commands.cpp`**, reusing the per-entity JSON emit shape `DumpScene` already produces
  (scene_io) — do NOT pull in `editor::` or any render symbol. A `query select:entity` response is the same
  transform/material/mesh JSON object `DumpScene` emits for that entity, optionally narrowed to `fields`.
- **Index-addressed, resolved fresh** (the `EntityAtIndex` convention): `entity:N` is the Nth object in view order; an
  out-of-range / negative index is a deterministic `{"error":"out-of-range","entity":N,"count":C}` response (NOT a
  crash, NOT an abort — the command bus's established "bad command → false, keep going" semantics, commands.h:57).
- **Deterministic field order:** `fields` are emitted in a FIXED canonical order (transform, then material, then mesh)
  regardless of the request array order, so two semantically-equal requests produce byte-identical responses. Unknown
  field names are ignored deterministically (or listed once under an `"unknownFields"` array — pick one, document it).
- **`RunQueries` returns the responses as a JSON array**, pretty-printed with the SAME formatter `DumpScene` uses
  (scene_io's `%g` floats, 2-space indent) so the golden matches the house JSON style and is backend-identical.

## Design decisions (locked)
1. **APPEND to `engine/scene/commands.{h,cpp}`** (`hf::scene`):
   - The `query` op handler `OpQuery(reg, res, obj, /*out*/ std::string& response)` building the response per `select`.
   - `std::string RunQueries(ecs::Registry& reg, SceneResources& res, const char* queriesJson)` — parse a JSON array
     of query objects, run each, return a pretty JSON array of `{ "request": <the query obj>, "response": <result> }`.
     Deterministic, file-free (the unit-test entry).
   - Wire `query` into `RunCommands`'s dispatch (prints the response to stdout, like `dump`).
2. **`--agent-query <cmds.json> <out.json>`** (Windows `samples/hello_triangle/main.cpp`): build the default scene,
   call `RunQueries`, write the result to `<out.json>` (LF-clean, no BOM). Also wire the Metal `--agent-query` in
   `metal_headless/visual_test.mm` for showcase-set symmetry (bytes are backend-identical). **Grep your OWN
   visual_test.mm for `--agent-query` before reporting DONE.**
3. **Register the `query` command** in `agentApi.commands` (introspect.cpp) under capability `observe`. The
   `agentApi.contentHash` will change (content churn); `schemaVersion` MUST stay 1 (shape unchanged). This is an
   intended DX1-design validation — note the before/after `contentHash` in your report.
4. **The committed query script** `assets/agent/dx2_queries.json` — a fixed array exercising every branch:
   `select:entity` with `fields`, `select:entity` without `fields` (all components), `select:stats`,
   `select:entities`, and an out-of-range `entity` (the deterministic-error branch). The golden is `RunQueries` over
   this script against the default scene.
5. **PROOFS (exact stdout lines, fail loudly):**
   - `dx2-query: {queries:<Q>, entity:<N> fields:transform,material} response byte-golden MATCH`
   - `dx2-query select-stats: {entities:<E>, meshes:<M>, textures:<T>} (matches scene)`
   - `dx2-query determinism: two RunQueries runs BYTE-IDENTICAL`
   - `dx2-query out-of-range: query entity <BIG> -> {"error":"out-of-range"} (deterministic, no crash)`
   - Golden discipline: the implementer MAY commit `query_responses.json` (deterministic backend-agnostic Windows
     artifact). Do NOT rebake `agent_api.json` / `default_scene.json` — the CONTROLLER does (they carry the new
     `query` command + new `contentHash`). NO Metal golden.
6. **Tests — NEW `tests/agent_query_test.cpp` (pure CPU, ASan-clean), ctest +1:** `RunQueries` two-run determinism;
   `select:entity` with `fields` == the expected narrowed object; `fields` omitted == all components; `select:stats`
   counts == the scene's actual counts; out-of-range → the deterministic error object (no throw); field-order
   canonicalization (a request with `fields` in a different order yields byte-identical output). Register in
   `tests/CMakeLists.txt`.

## RHI seam additions (summary)
- **None.** DX2 is pure-CPU JSON over the existing command bus. Report the seam: `commands.{h,cpp}` additive (one new
  op + one new function); `introspect.cpp` additive (one `agentApi.commands` entry); NO shader, NO RHI, NO `hf_gen_msl`;
  all sim/render headers + ALL shaders + the 258 Metal goldens BYTE-UNCHANGED (only `agent_api.json` +
  `default_scene.json` rebake + `query_responses.json` is new).

## Out of scope (YAGNI)
Writes (the `set_*`/`add`/`remove` mutate ops already exist). Querying sim/gameplay state (the verdict world — DX5/DX6).
A query language / filters / wildcards (fixed `select` enum + an optional `fields` list only). Pagination. DX2 claims
ONLY: a deterministic, index-addressed, field-selectable JSON read protocol over the scene, byte-golden, registered as
the `query` command in the versioned contract.
