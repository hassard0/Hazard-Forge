> NO Mac bake, like the existing `introspect/default_scene.json` golden). The FIRST slice of FLAGSHIP #31 (THE AGENT

> future flagships keep adding showcases. **APPENDS to `engine/editor/introspect.{h,cpp}`** (`hf::editor`, additive —
> the existing 8 top-level keys keep their byte layout). Branch: `slice-dx1`. See [[hazard-forge-ax-roadmap]],
> [[hazard-forge-introspect-golden-staleness]], [[hazard-forge-docs-style]], [[hazard-forge-build-env-vcvars]].

**Goal:** Add `editor::DescribeAgentApi()` (a pure function) emitting an additive top-level `"agentApi"` object folded
into `DescribeEngine`, and a `--agent-api` flag that emits ONLY that block (the lightweight negotiation call). The
block is the versioned SDK contract:
```
"agentApi": {
  "schemaVersion": 1,                       // governs SHAPE ONLY (the key structure) — see THE KEY DESIGN CALL
  "capabilities": ["observe","mutate","author","replay","hot-reload"],
  "commands":  [ { "op":"...", "capability":"...", "args":[ {"name","type"} ] }, ... ],
  "contentHash": "<hex>"                     // a digest over the VOLATILE content (showcases + features + commands)
}
```
Bake the **strict-zero** Windows byte-golden `agent_api.json` + the controller-rebaked `default_scene.json` (which now
carries the `agentApi` block). → the SDK contract every later DX slice (DX2 query, DX3 author, DX4 reload, DX5 replay,
DX6 stress) registers a capability + command into.

## THE KEY DESIGN CALL (the scout's flagged RISK — resolve it HERE, in the beachhead)
The AX product's whole value is a **stable versioned contract** an agent codes against. But every golden here is a
byte-exact JSON file, and `default_scene.json` enumerates ~120 showcases + ~60 features INLINE — so every future
flagship slice that adds a showcase MUTATES the bytes, even though the *schema shape* is unchanged. If `schemaVersion`
covered content, it would have to bump on every unrelated slice and the contract would churn constantly.
- **RESOLUTION (locked):** `schemaVersion` governs the **SHAPE ONLY** — the set and structure of keys in `agentApi`
  (and the shape of a command entry: `{op, capability, args:[{name,type}]}`). It bumps ONLY when that structure
  changes (a new key, a changed arg-schema shape), NOT when a showcase/feature/command is added or removed.
- The **volatile content** (the actual showcases/features/commands lists) is summarized by `agentApi.contentHash` — a
  deterministic digest (a stable 64-bit FNV-1a over the canonical concatenation of the showcase flags + feature names
  + command ops, in their existing emit order). An agent pinned to `schemaVersion:1` codes against the fixed shape;
  it watches `contentHash` to cheaply detect content drift without re-parsing or re-pinning. Future slices append a
  showcase → `contentHash` changes + the golden rebakes → `schemaVersion` HOLDS at 1. This is the property that makes
  it a *contract* rather than a dump.
- Use the engine's EXISTING FNV helper if one is reachable (grep `Fnv`/`fnv`/`Hash64` in the introspect/render code —
  the bindless/oit goldens already FNV-hash; reuse it). The hash is over engine content already in the JSON, so it is
  deterministic and backend-agnostic (identical on Vulkan and Metal — verify once on Windows).

## Design decisions (locked)
1. **APPEND to `engine/editor/introspect.{h,cpp}`** (`hf::editor`): `std::string DescribeAgentApi()` (pure — no
   `Registry` needed; reads the static showcase/feature/command tables). `DescribeEngine` gains the `"agentApi"` key
   AFTER its existing keys (append → the prior key order/bytes are unchanged except the additive trailing key + the
   controller rebake). The block shape is EXACTLY as in Goal above.
2. **The `commands` array:** one entry per scene-command op (`engine/scene/commands.h`), `{op, capability, args}` where
   `capability` ∈ the 5 capabilities and `args` is `[{name,type}]`. For DX1, map the EXISTING ops to `observe`/`mutate`
   (the `query`/`author`/`replay`/`hot-reload` capabilities are ENUMERATED now so agents can discover them, with their
   commands landing in DX2–DX6). Listing a capability with zero commands yet is fine and intended (the contract is
   forward-declared); a later slice fills it.
3. **`--agent-api <out.json>`** (Windows `samples/hello_triangle/main.cpp`; and wire the Metal `--agent-api` in
   `metal_headless/visual_test.mm` for parity even though the bytes are backend-identical — keeps the showcase set
   symmetric; grep your OWN visual_test.mm for `--agent-api` before reporting DONE). Emits ONLY the `agentApi` block
   (strip nothing — it's the top-level object). Pure CPU, no GPU, no capture.
4. **PROOFS (exact stdout lines, fail loudly) — via a small `--agent-api`-driving check in the showcase or a proof mode:**
   - `dx1-agent-api: {schemaVersion:1, capabilities:5, commands:<C>, showcases:<S>, features:<F>} byte-golden MATCH`
   - `dx1-agent-api determinism: two --agent-api runs BYTE-IDENTICAL`
   - `dx1-agent-api capability-complete: every command op has a capability AND an arg-schema (ops:<C>)`
   - `dx1-agent-api shape-stable: schemaVersion==1; contentHash:<hex> (content-versioned, shape-frozen)`
   - Golden discipline: the implementer MAY commit `agent_api.json` (a deterministic backend-agnostic Windows
     artifact, generated from their own `--agent-api`); the controller re-verifies it byte-matches a fresh run + does
     the `default_scene.json` rebake. Do NOT commit any Metal golden (there is none for DX1).
5. **Tests — extend `tests/introspect_test.cpp` (pure CPU, ASan-clean), ctest stays at its count:** assert `agentApi`
   exists; `schemaVersion==1`; exactly the 5 capabilities in order; every `commands[i]` has a non-empty `op`,
   a `capability` in the set, and an `args` array; `contentHash` is present + non-empty + STABLE across two
   `DescribeAgentApi()` calls; and that `DescribeAgentApi()` alone == the `agentApi` sub-object of `DescribeEngine`
   (the `--agent-api` == the folded block, byte-for-byte).
6. **verify.ps1:** add a SECOND Windows-side byte-golden block (clone `:382–402`) for `--agent-api` →
   `tests/golden/introspect/agent_api.json`. NOT in `$Goldens`/`$vkShots` (those are the Mac image set).

## RHI seam additions (summary)
- **None.** DX1 is pure-CPU JSON over the existing introspect path. Report the seam: `introspect.{h,cpp}` additive (one
  new pure function + one additive top-level key); NO shader, NO RHI, NO `hf_gen_msl`; all sim/render headers + ALL
  shaders + the 258 existing Metal goldens BYTE-UNCHANGED (only `default_scene.json` rebakes + `agent_api.json` is new).

## Out of scope (YAGNI)
The query/mutate EXECUTION (DX2 adds the `query` op handler; mutate ops already exist). Author/replay/hot-reload
commands (DX3–DX6 — DX1 only ENUMERATES their capabilities in the contract). A binary/compressed protocol (JSON text,
the house format). Auth/transport (it's an in-process SDK surface). DX1 claims ONLY: a versioned, shape-frozen,
content-hashed agent-API contract block, deterministic + byte-golden, that the later DX slices register into.
