# Slice AI3 — Deterministic AI: INTEGER LINE-OF-SIGHT (the feasibility crux) — Design

> Autonomous-session spec. Verifiable headlessly on Windows+Vulkan + Mac+Metal. The THIRD slice of the DETERMINISTIC
> AI flagship (GitHub issue #28, slice-tag **AI**) — and the make-or-break CRUX. AI1 shipped the decision tree, AI2 the
> environment queries. AI3 adds **perception**: can the agent SEE the target? The engine has NO line-of-sight /
> raycast / segment-intersection test on a deterministic path (the only ray tests are the float GPU ray tracer). LOS
> must be invented as a **pure-integer, bit-identical** visibility test — because the textbook float DDA (`tMax`/`tDelta`)
> and any float parametric `t` are non-deterministic across compilers/backends. Get this wrong and the perception →
> decision chain (AI4) and the rollback divergence (AI5) are built on sand. **APPENDS to `engine/ai/ai.h`** (`hf::ai`,
> AI1/AI2 byte-frozen — append-only). Header-only, pure CPU. Branch: `slice-ai3`. See [[hazard-forge-ai-roadmap]],
> [[hazard-forge-docs-style]], [[hazard-forge-build-env-vcvars]], [[hazard-forge-metal-showcase-gate]], [[hazard-forge-gpu-tdr-chunking]].

**Goal:** Append to `engine/ai/ai.h`: `LineOfSight(from, to, blockers) -> bool` — a pure-integer test that returns
false (occluded) iff the segment `from→to` is blocked by any obstacle, true (visible) otherwise. Obstacles are integer
**AABB blockers** (`fpx::FxAabb` / a small `AiBlocker { FxVec3 min, max; }` list — the segment-vs-AABB **integer slab
test**, the cleanest deterministic formulation, NO float `t`, NO division). Plus an `AiPerception` write into the
blackboard (`canSeeTarget`, `lastSeenCell`). Add `--ai3-los-shot` (Vulkan) / `--ai3-los` (Metal) rendering a
pure-integer **visibility raster** (a grid of cells, each tested for LOS from the agent, colored visible vs occluded,
with the blockers drawn). Bake the **strict-zero** golden `ai3_los`.

## Design call: segment-vs-AABB by an INTEGER slab test (no float t, no division), tie cases pinned
The deterministic-integer LOS, in priority of robustness:
- **Segment-vs-AABB integer slab test (RECOMMENDED).** For segment `p0→p1` and box `[bmin, bmax]`, the slab test
  finds the parametric entry/exit `t` per axis. The float version divides (`t = (bmin.x - p0.x) / (p1.x - p0.x)`) and
  compares floats — BOTH non-deterministic. The INTEGER version keeps `t` as a **fraction `num/den`** and compares two
  fractions by **cross-multiplication** (`a/b < c/d` ⇔ `a*d < c*b`, with sign handling — int64 intermediates, the
  `navmesh.h:126 PointInTriXZ` int64-edge-function discipline), so there is NO division and NO float. The segment hits
  the box iff `max(entry_t per axis) ≤ min(exit_t per axis)` and the overlap is within `[0, 1]` — all integer
  comparisons. `LineOfSight` = NOT (hits any blocker). Handle the **axis-parallel segment** degenerate (a zero
  direction component → the segment is inside the slab iff `p0` is between `bmin`/`bmax` on that axis, no division).
- **Pin the tie/boundary cases deterministically:** a segment grazing a box face/edge/corner (entry_t == exit_t, or a
  `t` exactly at 0/1, or exactly through a corner) must resolve by a FIXED rule (document it — e.g. "touching the
  boundary counts as NOT blocked" or the reverse; pick one and be consistent). Two runs + both backends must agree
  because it's all integer comparison — but the RULE at the boundary must be explicit so the golden is reproducible.
- **(Alternative, document if used instead):** an integer supercover/DDA walk over the `nav::Heightfield` columns
  (`QuantizeCoord` + `FloorDiv`), stepping cell-to-cell by comparing the accumulated **integer** error
  (`err_x * |dy|` vs `err_y * |dx|`, cross-multiplied — Bresenham-supercover), blocked iff a non-walkable column is
  entered; the diagonal-corner step must be pinned. The AABB slab test is preferred for AI3 (fewer tie cases, no grid
  dependency); the heightfield walk can come later if column-occlusion is wanted.
- **Within-band caveat:** LOS is a 2D (XZ) horizontal-plane test for the NPC sample (agents + target on the ground
  plane) — a full 3D volumetric occlusion is out of scope. The blockers are axis-aligned boxes (the `FxAabb`
  primitive); arbitrary convex occluders are a future refinement. The result is EXACT integer (not within-band) — the
  determinism is strict; only the geometric model (2D, AABB-only) is simplified.
- **Docs-style:** ship generic terms ("line-of-sight", "visibility", "blocker") — no third-party branding.

## Out of scope (YAGNI)
3D volumetric occlusion (2D XZ horizontal LOS for the NPC sample). Arbitrary convex occluders (axis-aligned boxes
only). Partial visibility / soft cover percentages (binary visible/occluded). The verdict-agent composition (AI4 —
which calls `LineOfSight` agent→player). Lockstep (AI5). AI3 claims ONLY: a deterministic, bit-identical, pure-integer
line-of-sight (segment-vs-AABB slab) with the boundary/corner rules pinned, strict-zero cross-vendor.
