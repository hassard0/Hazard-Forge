> [[hazard-forge-gpu-tdr-chunking]], [[hazard-forge-introspect-golden-staleness]], [[hazard-forge-metal-showcase-gate]].

**Goal:** Implement the Vulkan acceleration-structure + inline-ray-query backend behind the RT1 seam
(`IRHIDevice::{CreateBlas,CreateTlas,SupportsHardwareRayQuery}`, `ICommandBuffer::BindAccelStructure`), add

## Design call: HW BVH as a candidate generator; OUR fx math owns the closest-hit

The determinism moat rests on NOT consuming any vendor float. `VK_GEOMETRY_TYPE_AABBS_KHR` geometry offers the app a
candidate for EVERY overlapped AABB leaf in UNSPECIFIED order, and the app MUST supply its own intersection (there
is no built-in AABB intersector). So the kernel:
1. Opens a `RayQuery` over the TLAS with `RAY_FLAG_NONE` and a large `tMax`.
2. On each `Proceed()` candidate (`CANDIDATE_PROCEDURAL_PRIMITIVE`), reads the candidate's `primIndex`
   (`CandidatePrimitiveIndex` / instance custom index), runs the FROZEN `rtrace::IntersectSphere`/`IntersectAabb`
   in fx Q16.16 against that primitive, and folds the hit into a running closest by the **total order
   `(t, primIndex)`** — **never** calling `CommitProceduralPrimitiveHit`/`CommitNonOpaqueTriangleHit` and **never**
   shrinking the query's `tMax` (so the driver keeps yielding every overlapped leaf, not just the nearest).
3. After the query drains, shades the closest hit with the FROZEN `rtrace::ShadeHitInt` (integer) → RGBA8.

Because the min is over a TOTAL ORDER of OUR fx `(t,primIndex)` keys computed from the host-snapped integer scene,
the output is **independent of the driver's traversal order, BVH layout, and vendor** — bit-identical to the SW
no-cull reference. The three hard constraints (from the roadmap, all MANDATORY):
- **(C1) Inflate every BLAS AABB by a fixed host-computed margin** so the driver's FLOAT AABB-overlap test is a
  strict SUPERSET of every true fx hit (no grazing-corner miss). Extra candidates are free — OUR min discards them.
  The margin is a fixed Q16.16 constant (e.g. `kRtAabbMargin = 1/64` world unit) added to each AABB's half-extents
  when emitting the FLOAT `VkAabbPositionsKHR` (the BLAS geometry is float AABBs; the *intersection* stays fx).
- **(C2) ALL intersection + min in fx Q16.16** (the frozen `rtrace::` routines), NEVER reading the query's float
  `t`/hit attributes. The only float in the kernel is the AABB-overlap the driver does internally (which we treat
  as a conservative candidate filter, not a result).
- **(C3) Degenerate rays rejected host-side** (`dir==0`); ties broken by `primIndex` (total order); the API's
  internal "commit closest" is NEVER consulted.

## The Vulkan-HW / Metal-CPU-SW split (the int64-Vulkan-only convention, extended to HW-RT)

`rt_query.comp.hlsl` uses BOTH HLSL inline `RayQuery` (DXC→SPIR-V `RayQueryKHR`, which `glslc`/spirv-cross cannot
lower to MSL) AND int64 fx math. So — exactly like every int64 slice (`fpx_integrate.comp`, `swraster.comp`, the
whole gjk/ccd/manifold family) — **`rt_query.comp` is VULKAN-SPIR-V-ONLY, NOT in `hf_gen_msl`.** On Metal the
`--rt2-query` showcase runs the FROZEN CPU `rtrace::RenderScene` (RT1's reference) → byte-identical to the Vulkan HW
result BY CONSTRUCTION. So:
- **Vulkan** carries the HW==SW==CPU proof on real HW (RTX 4070): HW-ray-query image `== rt_trace.comp` SW-GPU image
  `==` CPU `rtrace::RenderScene`, all memcmp strict-zero.
- **Metal** runs the CPU SW reference → the `rt2_query` golden is **strict-zero cross-vendor** (Vulkan-HW == CPU ==
  Metal-CPU).
- **Metal hardware ray tracing** (a hand-authored `shaders/rt_query.metal` `metal::raytracing::intersector` over an
  `MTLAccelerationStructure`) is a DELIBERATE, documented FOLLOW-ON slice (RT2b) — `MTLAccelerationStructure` build
  + a custom-intersection `intersector` is a substantial separate backend chunk; cramming it into RT2 would make the
  slice unverifiably large. RT2's honest claim: *deterministic HW ray tracing, proven bit-exact vs the SW reference,
  validated on Vulkan HW*; the Metal-HW twin lands next.

## Design decisions (locked)
1. **Vulkan device enabling (`vulkan_device.cpp`).** Add to the pNext feature chain (by f12):
   `VkPhysicalDeviceAccelerationStructureFeaturesKHR{ accelerationStructure=TRUE }` +
   `VkPhysicalDeviceRayQueryFeaturesKHR{ rayQuery=TRUE }`; set `f12.bufferDeviceAddress=TRUE` (additive — already
   true for bindless on this path; confirm). Enable via `enable_extension_if_present("VK_KHR_acceleration_structure")`
   + `enable_extension_if_present("VK_KHR_ray_query")` (+ their deps `VK_KHR_deferred_host_operations` if the
   validation layer demands — add if-present). **Capability-gate:** `SupportsHardwareRayQuery()` returns the ACTUAL
   queried support (both features present AND both extensions enabled). On a no-RT device it returns false → the
   showcase falls back to the SW path (RT1's `rt_trace`/CPU) so CI without RT HW still passes. Load the 5 PFNs
   (`vkCreateAccelerationStructureKHR`, `vkGetAccelerationStructureBuildSizesKHR`,
   `vkCmdBuildAccelerationStructuresKHR`, `vkGetAccelerationStructureDeviceAddressKHR`, `vkGetBufferDeviceAddress`)
   right after device creation (the push-descriptor PFN precedent); expose accessors.
2. **`CreateBlas` / `CreateTlas` (NEW `engine/rhi_vulkan/vulkan_accel.cpp` + a `VulkanAccelStructure : IAccelStructure`).**
   `CreateBlas`: from `BlasDesc` AABB-procedural geoms, emit a float `VkAabbPositionsKHR` array (each AABB inflated
   by `kRtAabbMargin` per C1), build a `VK_GEOMETRY_TYPE_AABBS_KHR` BLAS — `vkGetAccelerationStructureBuildSizesKHR`
   → allocate the accel-struct buffer + a scratch buffer → `vkCmdBuildAccelerationStructuresKHR` on a one-shot
   command buffer → `vkQueueWaitIdle` (the synchronous load-time path, like staging uploads). `CreateTlas`: one
   `VkAccelerationStructureInstanceKHR` per `TlasInstance` (identity transform for RT2; `instanceCustomIndex` =
   instanceId), build a TOP_LEVEL accel-struct the same way. Store the `VkAccelerationStructureKHR` + its device
   address. Procedural AABB geometry only (no triangles — Tier B/RT6).
3. **`BindAccelStructure` (`vulkan_command_buffer.cpp`).** Push-descriptor write at the accel slot:
   `VkWriteDescriptorSet{ descriptorType = ACCELERATION_STRUCTURE_KHR, pNext =
   &VkWriteDescriptorSetAccelerationStructureKHR{ pAccelerationStructures = &tlas } }` via `pushDescriptorFn()` at
   `VK_PIPELINE_BIND_POINT_COMPUTE`. `ComputePipelineDesc` gains an additive optional field (e.g.
   `int accelStructureBinding = -1`); when ≥0 the compute layout adds the ACCELERATION_STRUCTURE binding at that slot.
4. **`shaders/rt_query.comp.hlsl` (NEW, Vulkan-SPIR-V-ONLY — int64 + `RayQuery`).** One thread per pixel. Generates
   the SAME integer primary ray as `rt_trace.comp` (the frozen `PrimaryRay` math), opens
   `RayQuery<RAY_FLAG_NONE> q` over `gTlas` (a `RaytracingAccelerationStructure` at binding 4) with a fx-derived
   float ray (origin/dir converted to float ONLY to feed the driver's traversal; the INTERSECTION uses the fx
   ray), then `while (q.Proceed())` — for each `CANDIDATE_PROCEDURAL_PRIMITIVE`, look up the candidate's primitive
   (by `q.CandidatePrimitiveIndex()` + instance/geometry index → the SAME `gSpheres`/`gAabbs` SSBOs as RT1) and run
   the frozen fx `IntersectSphere`/`IntersectAabb` + `(t,primIndex)` min — **NEVER** `q.CommitProceduralPrimitiveHit`
   and NEVER setting a ray tMax that would cull. After the drain, `ShadeHitInt` → `gImage`. Register a DXC `:cs`
   line in `samples/hello_triangle/CMakeLists.txt` ONLY; add the Vulkan-only comment block to
   `metal_headless/CMakeLists.txt` (the `rt_trace.comp` precedent). NO `hf_gen_msl` line.
   - **The float ray for traversal:** the driver needs a float origin/dir to walk its BVH. Build it by
     `FxToFloat`-converting the fx ray (a documented, ALLOWED float — it only widens the candidate set the driver
     yields; correctness is re-checked in fx). The AABB margin (C1) guarantees the float traversal never SKIPS a true
     fx hit. This is the one place float touches the path, and it is provably safe (superset).
5. **Showcase `--rt2-query-shot <out>` (Vulkan) AND `--rt2-query` (Metal) — WIRE BOTH (grep your OWN
   `metal_headless/visual_test.mm` for `--rt2-query` BEFORE reporting DONE; [[hazard-forge-metal-showcase-gate]]).**
   A RICHER pinned scene than RT1 (e.g. a 4×4 grid of spheres at varied depths + a few AABB boxes + a ground AABB,
   ~20-30 primitives — so the BVH genuinely culls and the completeness proof is meaningfully exercised), defined by a
   new frozen `rtrace::BuildRt2Scene()`... **NO** — `rtrace.h` is byte-frozen. Define the RT2 scene in the showcase
   TU (main.cpp / visual_test.mm) using the frozen `rtrace::` primitive types (do NOT append to `rtrace.h`).
   - **Vulkan:** if `SupportsHardwareRayQuery()`: build the BLAS/TLAS from the scene's AABBs (+ the spheres' bounding
     AABBs), dispatch `rt_query.comp`, read back the HW image; ALSO dispatch `rt_trace.comp` (SW GPU) and compute the
     CPU `rtrace::RenderScene`; memcmp **HW == SW-GPU == CPU** strict; write the HW image. Else (no RT HW): write the
