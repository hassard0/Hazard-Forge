# Slice AI2 — Deterministic AI: ENVIRONMENT QUERIES (integer scoring over the navmesh) — Design

> Autonomous-session spec. Verifiable headlessly on Windows+Vulkan + Mac+Metal. The SECOND slice of the DETERMINISTIC
> AI flagship (GitHub issue #28, slice-tag **AI**). AI1 shipped the integer decision-tree + blackboard. AI2 adds the
> **environment query** layer: an NPC scores candidate positions around an anchor by pure-integer queries (distance²,
> navmesh-reachability) and picks the best — the "where should I go?" question, deterministic and bit-identical. No
> float anywhere in the generator or the scorers. This is what a decision-tree "pick a destination" action calls
> (wired in AI4). **APPENDS to `engine/anim`... NO — `engine/ai/ai.h`** (`hf::ai`, AI1 byte-frozen — append-only; now
> also `#include`s `nav/navmesh.h` READ-ONLY). Header-only, pure CPU. Branch: `slice-ai2`. See [[hazard-forge-ai-roadmap]],
> [[hazard-forge-docs-style]], [[hazard-forge-build-env-vcvars]], [[hazard-forge-metal-showcase-gate]].

**Goal:** Append to `engine/ai/ai.h`: a candidate **generator** (`GenerateRing`/`GenerateGrid(anchor, radius, count)` →
candidate world positions in Q16.16 / integer voxel coords) and an integer **scorer chain** —
`ScoreDistanceSq(cand, target)` (the Q16.16 distance², no sqrt — the `verdict::FxDist2` / `nav::ManhattanDist`
discipline) and `ScoreNavReachable(cand, agentCell, navmesh)` (is the candidate's nav poly in the agent's connected
component AND does `nav::FindPath` return a finite cost) — and `RunQuery(query, navmesh, scorers) -> EqsResult { int
bestIndex; int32_t bestScore; }` returning the best candidate by the combined integer score, **tie → lowest candidate
index** (deterministic). Add `--ai2-query-shot` (Vulkan) / `--ai2-query` (Metal) rendering a pure-integer candidate
grid colored by score with the best marked. Bake the **strict-zero** golden `ai2_query`.

## Design call: integer generator + integer scorers + lowest-index tie-break = deterministic "best position"
- **Generator (integer).** `GenerateRing(anchor, radius, count)` places `count` candidates on a ring (or
  `GenerateGrid` on a lattice) around the anchor, in FIXED order, at integer/Q16.16 coordinates. NO float trig — use a
  fixed precomputed integer direction table (the `gi`/`ik` host-baked angle LUT precedent, or a fixed set of integer
  offset vectors). Candidate order is fixed → deterministic.
- **Scorers (integer).** `ScoreDistanceSq` = `FxDist2(cand, target)` (Q16.16 squared distance, int64 intermediate
  narrowed — NO sqrt, NO float). `ScoreNavReachable` quantizes the candidate to its nav cell/poly, checks it's in the
  agent's `nav::ConnectedComponents` component, and runs `nav::FindPath` for a finite integer cost (unreachable →
  a large sentinel score so it loses). Combine scores by a FIXED integer weighting (e.g. `total = distanceSq + (reachable ? 0 : kBigPenalty)`).
- **Selection.** `RunQuery` scans candidates in FIXED order, keeps the lowest `total` (or highest — document the
  convention: "closest reachable wins" → lowest distanceSq among reachable); on a TIE, the LOWEST candidate index wins
  (the deterministic tie-break — never a float compare, never an unordered container). Bit-identical by construction.
- **The within-band caveat:** reachability inherits `nav::FindPath`'s documented scope — paths within the LARGEST
  connected component only (`navmesh.h` NAV scope), so the query scene keeps the anchor + candidates + target in one
  component. Distance is exact integer; the "best" is a deterministic min, not a continuous optimum.
- **Docs-style:** ship generic terms ("environment query", "candidate", "scorer") — no third-party branding.
