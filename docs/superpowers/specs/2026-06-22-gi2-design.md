# Slice GI2 — Deterministic Lumen-class GI: INTEGER SH ENCODE (the crux) — Design

> Autonomous-session spec. Verifiable headlessly on Windows+Vulkan AND Apple M4+Metal (pure-integer compute, NO ray
> query → **MSL-native, true GPU on BOTH backends** — the strongest proof tier; no RT hardware needed). The SECOND
> slice of FLAGSHIP #29 (DETERMINISTIC LUMEN-CLASS GI, `engine/render/gi.h`, namespace `hf::render::gi`, slice-tag
> **GI**). GI1 produced the per-probe-per-ray integer radiance (`TraceProbeRays`). GI2 **encodes** that radiance into
> a deterministic **integer spherical-harmonics** representation (3rd-order, 9 coeffs × RGB) — the strict-zero twin
> of the float DDGI `probe_sh.h`. This is **THE CRUX of the flagship**: integer SH coefficients are small signed
> fractions and the later multi-bounce loop (GI4) re-injects them, so precision and dynamic range must be managed
> exactly. The mitigation is the established "stay wide, narrow once" discipline: accumulate in an **int64**
> accumulator and quantize to Q16.16 only AFTER the single normalize — and PROVE the dynamic-range envelope on the
> CPU first. APPEND to `gi.h` (GI1 code BYTE-FROZEN). Branch: `slice-gi2`. See [[hazard-forge-gi29-roadmap]],
> [[hazard-forge-gi-roadmap]], [[hazard-forge-docs-style]], [[hazard-forge-introspect-golden-staleness]],
> [[hazard-forge-metal-showcase-gate]].

## Design call: int64 accumulator, quantize once — the crux mitigation
3rd-order SH has 9 basis functions (l=0:1, l=1:3, l=2:5). Encoding folds each probe ray's radiance into the coeffs:
`coeff[i] += radiance * Y_i(dir) * solidAngleWeight`, then a cosine-lobe convolution + a `1/N`-style normalize for
reconstruction. In float this is trivial; in **integer Q16.16** the danger is twofold — band-2 coeffs are tiny
(products of small fractions) and can **underflow toward the LSB** (coarse SH, or a zeroed contribution), while
scaling up to preserve precision risks **overflowing the ±32768 fx bound** once GI4 feeds bounces back. The fix
(the `rtrace::IntersectSphere` discriminant discipline — hold the value in `int64` Q32.32, narrow ONCE):
- **Accumulate every `radiance(Q16.16) * basis(Q16.16)` term in an `int64`** (Q32.32 per term; the 16-ray sum stays
  well within int64), apply the integer solid-angle + cosine-lobe `A_l` factors in the wide domain, and **quantize
  to Q16.16 exactly once** at the final normalize (`>> kFrac` after the single `/ totalWeight`). No per-sample
  re-quantization.
- **Pick the SH scale so the encoded coeffs and the reconstructed irradiance both stay within `[-2, 2]` in Q16.16**
  for the GI scene's `[0,1]`-radiance inputs — a documented headroom that survives GI4's bounded-gain feedback (the
  geometric series `< radiance/(1-gain)` is pinned `< 2` by the GI4 `giStrength < kOne`). Verify this bound in the
  unit test (no coeff or irradiance ever exceeds the headroom on the GI scene).
- **The basis table is host-precomputed** (compute `Y_lm` + the `A_l` cosine-lobe factors in `double` at table-
  definition time, bake Q16.16 literals) — NO `sqrt`/`cos`/`pow` on the GPU hot path; both backends read identical
  bits (the `probe_sh.h` discipline, in integer).

## Design decisions (locked)
1. **APPEND to `gi.h`** (GI1 byte-frozen):
   - `struct FxProbeSH { fx coeff[9][3]; }` (9 SH coeffs × RGB, Q16.16) — pad for std430 if the GPU mirror needs it.
   - `static constexpr fx kGiSHBasis[16][9] = { ... }` — the `Y_lm` evaluated at each of the 16 `kGiProbeDirs`,
     baked Q16.16; plus the cosine-lobe `A_l` factors as constants. Document the generator.
   - `inline FxProbeSH FxSHEncodeProbe(const GiRadiance* rays, ...)` — the **int64-accumulator** encode: per ray,
     per coeff, per channel, `acc[i][c] += (int64)rays[r].ch * (int64)kGiSHBasis[r][i]`; then apply the solid-angle +
     `A_l` factors + the single normalize in int64; **quantize to Q16.16 once**. Returns `FxProbeSH`.
   - `inline GiRadiance FxSHEvaluate(const FxProbeSH& sh, const FxVec3& normal)` — integer cosine-lobe irradiance
     reconstruction (the `probe_sh::SHEvaluate` integer twin), clamped ≥0.
   - `inline void EncodeAllProbes(grid, std::span<const GiRadiance> radiance, std::span<FxProbeSH> out)` — per probe.
   - `inline void GiSHToImage(grid, std::span<const FxProbeSH> sh, FxVec3 evalDir, std::span<uint32_t> img, w, h)` —
     the VIZ: each probe tile colored by `FxSHEvaluate(sh, evalDir)` (e.g. `evalDir = +Y`) — a smoother lighting grid
     than GI1's raw radiance. Strict integer.
2. **NEW `shaders/gi_sh_encode.comp.hlsl`** (pure integer, NO RayQuery → MSL-native): one thread per probe, reads the
   GI1 `GiRadiance` buffer (16 rays), runs `FxSHEncodeProbe` VERBATIM (int64 `long` accumulator), writes `FxProbeSH`
   to `gSH[probe]`. Register `:cs` in `samples/hello_triangle/CMakeLists.txt` AND `hf_gen_msl(gi_sh_encode.comp ...)`
   in `metal_headless/CMakeLists.txt` → **true GPU on both backends** (GPU==CPU strict, the int32-MSL-native tier).
   NOTE: if the int64 accumulator forces a Vulkan-only shader (the `glslc` int64 limit), fall back to the
   `csrq`-style Vulkan-only + Metal-CPU-ref convention — BUT prefer MSL-native: keep the accumulator in a form
   `glslc` accepts (e.g. a 2×int32 hi/lo manual widen if needed, or scale so an int32 accumulator suffices for 16
   rays) so GI2 is GPU-proven on BOTH backends. The implementer picks based on what compiles + documents which.
3. **Showcase `--gi2-shencode-shot <out>` (Vulkan) AND `--gi2-shencode` (Metal) — WIRE BOTH (grep your OWN
   `visual_test.mm` for `--gi2-shencode` BEFORE reporting DONE).** Build the GI1 scene + grid, `TraceProbeRays`
   (CPU, GI1), upload the radiance, dispatch `gi_sh_encode.comp`, read back the SH; compute the CPU `EncodeAllProbes`
   reference; memcmp GPU==CPU strict; render `GiSHToImage`. Golden = `tests/golden/metal/gi2_shencode.png`
   (CONTROLLER-baked).
4. **PROOFS (fail loudly; exact stdout lines):**
   - **(1) GPU==CPU:** `gi2-shencode: {probes:<P>, coeffs:9} GPU==CPU SH BYTE-EQUAL` (memcmp the SH buffer == 0).
   - **(2) dynamic-range envelope (the crux):** `gi2-shencode: SH in-range (maxAbsCoeff:<v> < 2.0, maxIrr:<v> < 2.0,
     nonzero-band2:true)` — every coeff AND every reconstructed irradiance stays within the Q16.16 headroom, AND a
     band-2 coeff is NON-ZERO on a directional-lit probe (proves the integer encode did NOT underflow band-2 to
     zero — the crux passes).
   - **(3) zero-radiance → zero SH:** an all-zero-radiance probe encodes to an all-zero `FxProbeSH`.
   - **(4) determinism + no-op:** GPU/CPU two-run byte-identical; `ProbeCount==0` → zero dispatch, SH buffer untouched.
   - Golden discipline: ONLY `tests/golden/metal/gi2_shencode.png`; do NOT commit it. Existing 247 goldens UNTOUCHED.
5. **Cross-backend bar:** strict **0.0000**. Mac re-renders `--gi2-shencode` vs the golden at 0.0000; the GPU==CPU
   proof holds on BOTH backends (MSL-native), so cross-vendor is strict-zero by construction.
6. **Tests — NEW `tests/gi_sh_test.cpp` (pure CPU, ASan-clean), ctest 120→121 — THE CRUX VALIDATION (CPU-FIRST):**
   the integer SH ROUND-TRIPS within a documented error band (encode a known directional radiance → evaluate the SH
   in that direction → recovers the radiance within the integer-SH band); the **dynamic-range envelope** (no coeff /
   no irradiance exceeds the `< 2.0` headroom across the GI scene; band-2 coeffs are non-zero for a directional
   input — NO underflow-to-zero); zero-radiance → zero SH; `FxSHEvaluate` is ≥0; a constant (ambient) radiance →
   only the DC (l=0) coeff non-zero (the SH orthogonality sanity); two-run determinism. **This test settles the crux
   before any GPU trust.** Clean under `windows-msvc-asan`.
7. **Introspect.** Add EXACTLY `gi2-shencode` (features) + `--gi2-shencode-shot` (showcases) + update
   `tests/introspect_test.cpp`. CONTROLLER rebakes the JSON golden (verify it stages ` M `).

## Out of scope (YAGNI)
Trilinear probe interpolation (GI3), multi-bounce (GI4), occlusion weighting (GI5), the lit GI capstone (GI6). 4th+
order SH (3rd order, like DDGI). A float SH variant (strict integer — the crux is to prove integer SH suffices; if
it ever proves too coarse for the capstone, a documented Tier-B-float SH for GI6 ONLY is the fallback, integer SH
staying the strict-zero golden). GI2 claims ONLY: a deterministic integer 3rd-order SH encode of the probe radiance,
bit-exact GPU==CPU on both backends, with a proven dynamic-range envelope (no underflow / no overflow), strict-zero
cross-vendor.
