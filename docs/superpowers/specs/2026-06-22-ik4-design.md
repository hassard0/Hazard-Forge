# Slice IK4 — Deterministic IK Control-Rig: IK ON THE SKELETON (the FK-pose → IK-corrected palette bridge) — Design

> Autonomous-session spec. Verifiable headlessly on Windows+Vulkan (RTX 4070) + Mac+Metal. The FOURTH slice of
> FLAGSHIP #32 (DETERMINISTIC IK CONTROL-RIG, slice-tag **IK**). IK1–IK3 built the deterministic IK SOLVERS (angle
> LUT, two-bone, FABRIK + look-at). IK4 is the ANIM-PILLAR BRIDGE: wire those solvers into the EXISTING skeleton +
> skinning-palette pipeline. Take a base local pose (the skeleton's bind pose / a sampled animation pose), run the IK
> solver to correct a limb SUB-CHAIN toward a world target (foot-plant / hand-reach), and emit the corrected joint
> palette through the same forward-kinematics + palette machinery the animation + ragdoll capstones use. The
> IK-corrected LOCAL POSE is bit-identical CPU/Vulkan/Metal; the float palette read-back is the documented render-only
> crossing (the JT4/PoseToPalette precedent). **APPENDS to `engine/anim/ik.h`** (`hf::anim::ik`, IK1–IK3 byte-frozen —
> append-only). Branch: `slice-ik4`. See [[hazard-forge-ik-roadmap]], [[hazard-forge-docs-style]],
> [[hazard-forge-build-env-vcvars]], [[hazard-forge-metal-showcase-gate]], [[hazard-forge-gpu-tdr-chunking]].

**Goal:** Append to `engine/anim/ik.h`: `struct IkRig` (the IK overlay description — a limb sub-chain of skeleton joint
indices + bone lengths + a pole + the chain's WORLD-space target), `SolveRigToTargets(skeleton, baseLocalPose, rig)
-> std::vector<FxQuat>` (the IK-corrected per-joint LOCAL rotations: forward-kinematics the base pose to world joint
positions in Q16.16, run the IK solver — FABRIK / two-bone — over the chain toward the target, convert the corrected
chain back to LOCAL joint rotations via `LookAtRotation`, and write them into the chain joints; the un-IK'd joints
stay EXACTLY the base pose), and `IkPoseToPalette(skeleton, correctedLocalPose) -> std::vector<math::Mat4>` (the
Q16.16→float skinning-palette read-back, the `joint::PoseToPalette` twin, render-only). Add
`shaders/ik_rig.comp.hlsl` (int64 → Vulkan-only; Metal runs the CPU `SolveRigToTargets` byte-identical). Add
`--ik4-rig-shot` (Vulkan) / `--ik4-rig` (Metal). Bake the **strict-zero** golden `ik4_rig`.

## Design call: FK the base pose, FABRIK the world positions, LookAtRotation back to local; the base-pose snap + palette read-back are the float crossings
- **The bit-exact pipeline (Q16.16, GPU==CPU):**
  1. **Base pose → world joint positions (FK).** Forward-accumulate the base LOCAL pose along the skeleton's
     topologically-sorted joints (the `joint::RagdollFromSkeleton` / `anim::PaletteFromLocalPose` single forward pass,
     MIRRORED in Q16.16): each joint's world transform = parent's world × local; the world POSITION is the chain input.
  2. **IK solve over the chain (world positions).** For each `IkRig` chain, run `FabrikSolve` (IK3) — or `TwoBoneSolve`
     (IK2) for a 2-bone limb — over the chain's world joint positions toward the world target → corrected world
     positions (the foot-plant). Pure integer; the IK1–IK3 within-band residual carries.
  3. **Corrected world positions → corrected LOCAL rotations.** For each bone in the chain, the corrected LOCAL
     rotation is the rotation that aims the bind-pose bone direction along the corrected world bone direction,
     expressed in the PARENT's world frame: `localRot = QConj(parentWorldRot) ⊗ LookAtRotation(bindBoneDir,
     correctedBoneDir) ⊗ parentWorldRot`-style composition (the implementer works the exact frame algebra — the
     constraint is bit-exact integer, GPU==CPU, reusing `LookAtRotation` (IK3) + `FxQuatMul`/`QConj`). Write the
     corrected rotations into ONLY the chain joints.
  4. **Un-IK'd joints stay EXACTLY the base pose** — a clean falsifiable invariant (IK touches only the chain joints;
     every other joint's local rotation is byte-identical to the base).
- **The float crossings (documented, OUTSIDE the bit-exact loop):** (a) the base pose comes from the skeleton bind
  pose / `anim::SampleLocalPose` (float) snapped to Q16.16 (the `RagdollFromSkeleton` bind, joint.h:546 — `FxQuatFromFloat`
  idiom); (b) `IkPoseToPalette` is the Q16.16→float `math::Mat4` read-back for the render (joint.h:630
  `PoseToPalette`). The IK SOLVE between them is bit-exact.
- **The golden** is a PURE-INTEGER 2D side-view of the IK-corrected SKELETON (the joints + bones drawn from the
  corrected world positions — a stick figure with the limb IK-corrected to plant the foot at the target),
  strict-zero cross-vendor (the integer pose drives the viz; the palette read-back is NOT in the golden).
- **The test skeleton:** a small hand-built `anim::Skeleton` (e.g. a humanoid leg: pelvis→hip→knee→ankle→foot, or an
  arm) is sufficient to prove the bridge — IK6 drives the REAL Fox skeleton render. Keep it controlled so the foot-plant
  is visually obvious.

  `QuatFromAxisAngle`/`FxCross`/`FxDotV` — the solvers + helpers (reused verbatim).
- **`engine/sim/joint.h`:** `RagdollFromSkeleton` (`:546`, the float→Q16.16 bind + the forward-accumulate FK pattern to
  MIRROR), `PoseToPalette` (`:630`, the Q16.16→float palette read-back twin), `QConj` (`:248`), `FxQuatMul`
  (fpx.h:412), `FxQuatNormalize` (fpx.h:427), `FxRotate` (fpx.h:440, applying a joint rotation to a bone vector).
- **`engine/anim/skeleton.h`:** `Skeleton` / `Joint` (`:14`, parent index + topo-sorted order — the FK accumulation
  order); **`engine/anim/animation.h`:** `JointPose` (`:36`, the local r/t), `SampleLocalPose` (`:45`),

- **The IK2/IK3 slices** (`ik_twobone.comp`/`ik_fabrik.comp` int64/Vulkan-only + the showcase/proof/golden structure) —
  IK4's `ik_rig.comp` is likewise int64/Vulkan-only (the FK + solve + rotation-convert is int64).

## Design decisions (locked)
1. **APPEND to `engine/anim/ik.h`** (`hf::anim::ik`, IK1–IK3 byte-frozen): `IkRig` (chain joint indices + lengths +
   pole + world target; support ≥1 chain), `SolveRigToTargets` (FK + solve + local-rotation convert → corrected local
   pose, a `std::vector<FxQuat>` or an `IkPose` struct), `IkPoseToPalette` (the float read-back). A small `FkWorldPositions`
   helper (the Q16.16 forward-accumulate) if useful. All deterministic, header-only.
2. **NEW `shaders/ik_rig.comp.hlsl`** (int64, `:cs`): one thread runs the WHOLE `SolveRigToTargets` (FK + IK + rotation
   convert) and writes the corrected local `FxQuat`s (and/or the corrected world positions for the viz) to an SSBO.
   Register the DXC `:cs` in `samples/hello_triangle/CMakeLists.txt` ONLY; Vulkan-only comment in
   `metal_headless/CMakeLists.txt`; NO `hf_gen_msl` line. Single-thread `[numthreads(1,1,1)]` if iterative (FABRIK);
   chunk only if heavy (a short skeleton is light).
3. **Showcase `--ik4-rig-shot <out>` (Vulkan) AND `--ik4-rig` (Metal) — WIRE BOTH (grep your OWN visual_test.mm for
   `--ik4-rig` BEFORE reporting DONE).** A fixed test skeleton + base pose + a SWEEP of world targets (the foot/hand
   tracing a path, incl. an un-reachable one). Vulkan: dispatch `ik_rig.comp`, read back the corrected pose, memcmp
   GPU==CPU; render a PURE-INTEGER 2D side-view of the IK-corrected skeleton (joints + bones, the planted limb,
   for each target — a fan / sequence). Metal: the CPU `SolveRigToTargets` → the same render. Golden =
   `tests/golden/metal/ik4_rig.png` (CONTROLLER-baked).
4. **PROOFS (exact stdout lines, fail loudly):**
   - `ik4-rig: {joints:<J>, chains:<C>, targets:<N>} GPU==CPU IK-corrected pose BYTE-EQUAL` (the Vulkan readback memcmp).
   - `ik4-rig: foot-plant — chain end-effector at the world target (residual:<r> in band) over reachable sweep`.
   - `ik4-rig: un-IK'd joints == base pose EXACTLY (untouched joints byte-identical)` — the falsifiable invariant.
   - `ik4-rig: provenance — IkPoseToPalette rebuilds from the bit-exact corrected pose; two-run BYTE-IDENTICAL;
     strict cross-vendor 0.0000 at the bake`.
   - Golden discipline: do NOT commit a Mac golden (the controller bakes `ik4_rig.png`). Existing 261 goldens UNTOUCHED.
5. **Cross-backend bar:** strict **0.0000** (the pure-integer corrected-skeleton viz; Metal CPU-ref byte-identical to
   the Vulkan GPU).
6. **Tests — NEW `tests/ik_rig_test.cpp` (pure CPU, ASan-clean), ctest +1:** `SolveRigToTargets` two-run determinism;
   the foot-plant (chain end-effector at the world target within band over a reachable sweep — FK the corrected pose
   and check); the un-IK'd-joints invariant (every joint NOT in a rig chain has a local rotation byte-identical to the
   base pose); reaching a target moves the corrected pose AWAY from the base (the IK actually did something — the
   chain joints differ from base when the target ≠ the rest end-effector); `IkPoseToPalette` determinism (two calls
   byte-equal). Register in `tests/CMakeLists.txt`.
7. **Introspect.** Add EXACTLY `ik4-rig` (features) + `--ik4-rig-shot` (showcases) + update `tests/introspect_test.cpp`.
   CONTROLLER rebakes the JSON goldens (verify they stage ` M `).

## Out of scope (YAGNI)
Animation-CLIP base poses driven over time (a single fixed/sampled base pose is enough; IK6 may animate). Joint LIMITS
on the IK chain. Multiple competing rigs / blend weights between IK and FK (a future refinement — IK4 is a hard
overlay on the chain joints). The lit 3D render (IK6 — IK4's render is the 2D integer pose viz). The lockstep
(IK5). IK4 claims ONLY: a deterministic bit-identical FK-pose → IK-corrected-local-pose → palette bridge, the foot
planted within a documented band, the un-IK'd joints exactly preserved, strict-zero cross-vendor.
