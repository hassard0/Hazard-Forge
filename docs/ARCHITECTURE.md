# Hazard Forge ‚Äî Architecture

## The RHI Seam

The central architectural decision in Hazard Forge is the **Rendering Hardware Interface seam** ‚Äî a layer of pure C++ abstract interfaces in `engine/rhi/rhi.h` that separates all engine and application code from any GPU backend.

### The hard rule

> No `vk*` type, no `MTL*` type, and no backend header may appear in any above-seam module ‚Äî `engine/rhi/` (the seam itself), `engine/scene/`, `engine/math/`, `engine/render/`, `engine/material/`, `engine/ui/`, `engine/audio/`, `engine/game/`, or the rest. Backend symbols are confined to `engine/rhi_vulkan/` and `engine/rhi_metal/` respectively. (`engine/hal/` is the platform layer: it touches SDL's `SDL_Metal_*` to vend an opaque `CAMetalLayer*` as a `void*`, but pulls in no Metal/Obj-C GPU types.)

The rule is checkable with a single `grep` for `vk[A-Z]|MTL|Metal` over the above-seam dirs: every match is a comment, prose, or a substring false-positive (`kMetallic`, the `Backend::Metal` enumerator name). The only genuine backend-symbol CODE above the seam is the **two-line `rhi_factory` dispatch** (the `CreateMetalDevice` forward-declaration + the `Backend::Metal` ‚Üí `mtl::CreateMetalDevice` call) ‚Äî the single sanctioned place the factory names a backend. This baseline is re-checked each consolidation pass and enforced by convention in every slice plan.

### The seam surface (`engine/rhi/rhi.h`)

All resource types are opaque base classes:

```
IRHIDevice        ‚Äî device factory; owns swapchain, frame loop, descriptor infrastructure
ISwapchain        ‚Äî surface + images; resize
ICommandBuffer    ‚Äî per-frame draw recording (BeginRenderPass, BindPipeline, Draw, ...)
IPipeline         ‚Äî compiled graphics PSO from GraphicsPipelineDesc
IBuffer           ‚Äî vertex / index / uniform buffer
ITexture          ‚Äî device-local image + sampler descriptor (base for render targets too)
IRenderTarget     ‚Äî sampleable offscreen color (+depth) image; inherits ITexture
```

`GraphicsPipelineDesc` is a small, flat descriptor that encodes what the pipeline needs to know about shaders, vertex layout, depth, push constants, descriptor set membership, and a set of mode flags that has grown additively slice by slice ‚Äî `fullscreen`, `depthOnly`, `pbrMaterial`, `usesEnvironment`, `usesJointPalette`, `depthWrite`, `alphaBlend`, `cullNone`, `lineList`, `fragmentPushConstants`, `usesLightClusters` ‚Äî all in terms of engine-defined enums and types, no backend types. Every new flag **defaults to the value that leaves all pre-existing pipelines byte-for-byte unchanged**, which is what keeps the committed goldens stable as features land.

There is also a compute path (`IComputePipeline` + `Dispatch`) used by the GPU particle system and the **GPU-driven culling** pass, on the same seam. Newer per-frame binding entry points have been added additively to the command interface ‚Äî e.g. `BindLightClusters` (clustered/Forward+ light data) and `BindReflectionProbe(ITexture& probeAtlas)` (the baked reflection/irradiance probe atlas) ‚Äî each with a default no-op body so existing pipelines and goldens are unaffected.

Three later seam additions follow the same defaulted-no-op discipline, so backends without the path still link and pre-existing goldens are byte-for-byte unchanged:

- **Indirect draw** ‚Äî `DrawIndexedIndirect(IBuffer& args, ...)` consumes a `VkDrawIndexedIndirectCommand` / `MTLDrawIndexedPrimitivesIndirectArguments` produced on the GPU by the cull compute pass (frustum-cull + ordered compaction), so the survivor draw never round-trips to the CPU. The `vk*`/`MTL*` indirect calls live only in the backend dirs.
- **Resource-state transitions** ‚Äî `TransitionResource(state prev ‚Üí next)` lets the render-graph barrier solver request a transition declaratively. On Vulkan it lowers to a `vkCmdPipelineBarrier2` (`VkImageMemoryBarrier2` with the matching layout + stage/access masks); Metal's tracked-hazard model makes it a no-op.
- **Secondary / parallel recording** ‚Äî passes record into per-thread secondary command buffers that are replayed in deterministic creation order (`vkCmdExecuteCommands` on Vulkan; an `MTLParallelRenderCommandEncoder`'s sub-encoders on Metal). This is what makes a 1-worker render and an N-worker render byte-identical.
- **OIT revealage blend (Slice CO)** ‚Äî a single additive **pure-interface** field, `GraphicsPipelineDesc::oitRevealageBlend` (a `bool`, default `false`), in `engine/rhi/rhi.h`. It is the only knob the Weighted-Blended-OIT resolve needs from the seam: when set, the pipeline's revealage target uses the multiplicative `(0, 1‚àísrcAlpha)` blend (vs. the default additive accum blend). It carries **no backend symbol above the seam** ‚Äî the actual `VkPipelineColorBlendAttachmentState` / `MTLRenderPipelineColorAttachmentDescriptor` blend factors are set **only inside the backend dirs** (`engine/rhi_vulkan/vulkan_pipeline.cpp`, `engine/rhi_metal/metal_pipeline.mm`), and `false` reproduces every pre-CO pipeline byte-for-byte, so existing goldens are unchanged.
- **Compute barriers (Slice CS)** ‚Äî two additive **pure-interface** entry points, `ICommandBuffer::ComputeToComputeBarrier()` / `ComputeToFragmentBarrier()` (defaulted empty bodies on the base interface), express the froxel inject‚Üíintegrate‚Üíapply hazards. The real `vkCmdPipelineBarrier2` / Metal tracked-hazard barriers are recorded **only inside the backend command buffers**; the COMPUTE‚ÜíCOMPUTE‚ÜíFRAGMENT chain is SYNC-HAZARD-free under the Khronos sync-validation layer.
- **Compute-stage shadow-map sampling (Slice CX)** ‚Äî one additive **pure-interface** entry point `ICommandBuffer::BindShadowMapCompute(IRenderTarget&)` plus one additive field `ComputePipelineDesc::sampledShadowMap` in `engine/rhi/rhi.h`, the only seam knobs the volumetric-shadows inject pass needs to read the sun's CSM atlas from a **compute** shader. Both are pure interface (defaulted/empty on the base, no backend symbol above the seam ‚Äî the real binding lives only in `engine/rhi_vulkan/` + `engine/rhi_metal/`), so the shadow-pass‚Üíinject-compute read barrier is SYNC-HAZARD-clean and pre-CX goldens are byte-for-byte unchanged. The rhi_factory dispatch count is still 2.
- **Cubemap render targets (Slice DD)** ‚Äî the **cubemap render-target capability** the dynamic cubemap-capture probe needs, a set of additive **pure-interface** entry points in `engine/rhi/rhi.h`: a new opaque base `ICubemapTarget : public ITexture` (a cube render target that is *both* a 6-attachment render destination and a sampled hardware `TextureCube`), `IRHIDevice::CreateCubemapTarget(size, format)`, `BeginCubemapFace(ICubemapTarget&, face)` / `EndCubemapFace(...)` to render each of the 6 faces, the read-back pair `ReadCubemapFace(...)` / `ReadRenderTarget(...)` (the capture-correctness proof), and `ICommandBuffer::BindCubemapProbe(ITexture&)` to bind a captured cube as the reflection source. Every method has a **defaulted body** on the base interface (so backends without it still link and pre-DD goldens are byte-for-byte unchanged); the real per-face attachment + cube-sampling implementation lives **only** in the backend dirs (`engine/rhi_vulkan/vulkan_cubemap_target.*`, `engine/rhi_metal/metal_cubemap_target.*`). The render graph emits the **capture‚Üísample barrier** (the 6 face passes then a ShaderRead transition), proven SYNC-HAZARD-free under the Khronos sync-validation layer. The rhi_factory dispatch count is still 2. **Slices DE (planar reflections) and DF (CAS) added no RHI** ‚Äî DE reuses the existing `cullNone` knob for the flipped reflection winding; DF is a fullscreen post pass on the existing path.

Backends are handed out through the factory:

```cpp
// Vulkan (Windows):
auto device = hf::rhi::CreateDevice(hf::rhi::Backend::Vulkan, window);

// Metal headless (macOS, no window):
auto device = hf::rhi::mtl::CreateMetalDeviceHeadless(width, height);

// Metal windowed (macOS): built directly from a CAMetalLayer* (passed as void*), so a windowed,
// presenting Metal device can be constructed WITHOUT pulling in hal/window.h or SDL ‚Äî used by the
// SDL-free native Cocoa entry (mac_window/) that drives the interactive --fly viewport on macOS.
MetalDevice dev(caMetalLayer /* void* */, width, height);
```

`MetalDevice` therefore has three constructors ‚Äî windowed-from-`Window&` (SDL HAL), windowed-from-`CAMetalLayer*` (native Cocoa, SDL-free), and headless-offscreen ‚Äî all behind the same `IRHIDevice` seam.

Both return `std::unique_ptr<IRHIDevice>`. The caller never sees a backend type again.

## Descriptor / Binding Model

Hazard Forge uses a **frequency-based descriptor set layout** adopted in Slice D (lighting):

| Set | Update frequency | Contents |
|-----|-----------------|----------|
| 0 | Per frame | UBO: `viewProj`, directional light, view position, point lights, `lightViewProj`, camera basis for sky; shadow map depth image (binding 1) + shadow sampler (binding 2) |
| 1 | Per material / per draw | Material textures + samplers. The base set is color + normal; the **PBR** path (`BindMaterialPBR`) widens it to the full 5-texture metallic-roughness set (base / metal-rough / normal / emissive / occlusion). The **environment** (HDR IBL) binds on a dedicated slot via `BindEnvironment` so the base material layouts are unchanged. |
| 2 | Per skinned draw | Joint palette (skinning), bound via `SetJointPalette` at a dedicated vertex binding. |
| 3 | Per clustered-lit frame | The clustered/Forward+ light data (Slice AG): the per-cluster `Cluster` array (offset/count) at binding 13, the flat light-index list at binding 14, and the `GpuLight` array at binding 15. Bound via `BindLightClusters` when a pipeline sets `usesLightClusters = true`; otherwise the set is absent and pre-existing pipelines are byte-for-byte unchanged. |

GPU instancing adds a **second per-instance vertex stream** (binding 1, step-per-instance) rather than a descriptor set ‚Äî four `RGBA32_Float` attributes carry a column-major `float4x4` per instance.

The set-3 cluster set follows the **same dedicated-extra-set pattern** established by the environment (HDR IBL) set and the joint palette: a feature that only some pipelines need binds on its own slot through a single `Bind*` call, so the base material/frame layouts ‚Äî and therefore every existing golden ‚Äî are untouched. `BindLightClusters` is the first case of **graphics-stage storage buffers** on the seam: the three set-3 bindings are `STORAGE_BUFFER`s read by the fragment shader (`lit_clustered.frag`), not the uniform buffers used by set 0/1. On Vulkan they are SSBO descriptors; on Metal they map to fragment `[[buffer(13/14/15)]]` via `spirv-cross --msl-decoration-binding` (`engine/rhi_metal/metal_common.h`: `kFragClusterBuf=13`, etc.).

**Set 0** is owned by the device. The device maintains one UBO per frame-in-flight (double-buffered), each with a pre-baked descriptor set that already points at its buffer. `SetFrameUniforms(data, size)` memcpys into the current frame's UBO; `BindPipeline` auto-binds set 0 when the pipeline has `usesFrameUniforms = true`. The CPU never writes a UBO that the GPU is still reading.

**Set 1** is owned per texture / render target. Each `VulkanTexture` and each `VulkanRenderTarget` allocates one descriptor set from a device pool at creation time and updates it with its image view and the device's default sampler. `BindTexture(ITexture&)` binds that set.

**Push constants** carry the per-object model matrix (64 bytes, `mat4`). `PushConstants(data, size)` records a push to the vertex stage.

When `usesFrameUniforms = false` (the post-processing pipeline), the material set is the only set and occupies set index 0. The `BindTexture` path accounts for this shift.

## Per-frame UBO growth

The `FrameData` struct (352 bytes at the time of the skybox slice; `kFrameUboSize` was 512) has grown additively across slices:

- Slice D: `viewProj`, `lightDir`, `lightColor`, `viewPos` ‚Äî basic per-frame camera + directional light
- Slice H: `ptCount`, `ptPos[3]`, `ptColor[3]` ‚Äî colored point lights
- Slice I: `lightViewProj` ‚Äî directional shadow map light-space matrix
- Slice J: `camFwd`, `camRight`, `camUp`, `skyParams` ‚Äî camera basis for sky ray reconstruction
- Slices AD‚ÄìAF: the cascaded / spot / point shadow matrices, per-cascade split distances, and the spot/point light parameters ‚Äî the shadow set's per-frame data. This is what pushed the struct past 512 B, so **`kFrameUboSize` was bumped 512 ‚Üí 1024** (`engine/rhi_vulkan/vulkan_device.cpp` and `engine/rhi_metal/metal_device.h`, kept in sync); both backends now allocate a 1024-byte per-frame UBO, and `SetFrameUniforms` asserts the upload fits.

The `kFrameUboSize` constant is bumped whenever the struct approaches the current allocation. The C++ `FrameData` struct (mirrored byte-for-byte in `metal_headless/visual_test.mm`) and the HLSL `cbuffer FrameData` in every shader that reads it must be kept in sync by hand ‚Äî the Metal struct layout is asserted to match the Vulkan sample's. Pass-specific parameters that are NOT per-frame (bloom thresholds, SSAO kernel params, per-object material factors, glass tint/alpha) travel via **push constants**, not the UBO ‚Äî the vertex push constant carries `mat4 model + float4 material`, and the bloom/SSAO fullscreen passes use the `fragmentPushConstants` flag to read their params in the fragment stage.

## Multi-Pass Frame Structure

Frames are described by `render::RenderGraph`: passes declare imported targets (shadow map / scene color / swapchain) as reads/writes and are executed in declaration order through the RHI. The baseline lit frame is three passes; feature showcases add more passes to the same graph (a g-buffer prepass + AO + blur + composite for SSAO; a threshold + 5√ó down + up + composite chain for bloom into an `RGBA16_Float` target; a sorted alpha-blended pass after the opaque pass for transparency; a debug-line draw after opaque geometry). The capstone scene composes **seven distinct opaque pipelines plus a transparent pipeline** into one scene pass.

The baseline three-pass frame:

```
Pass 0: Shadow pass (depth-only)
  BeginShadowPass(*shadowMap)
    ‚Üí depth-only command buffer, 2048¬≤ depth image, no color attachment
    ‚Üí drawDepthOnly: shadow pipeline (depthOnly=true), PushConstants(model), DrawIndexed
  EndShadowPass ‚Üí submit + fence-wait ‚Üí shadow map transitions to SHADER_READ_ONLY

Pass 1: Scene ‚Üí offscreen render target
  BeginRenderTargetFrame(*rt)
    ‚Üí command buffer targeting the RT's color+depth
    ‚Üí sky pipeline (fullscreen, depthTest=false) ‚Üí Draw(3)   [background fill]
    ‚Üí lit pipeline: per-renderable [PushConstants, BindTexture, BindVB/IB, DrawIndexed]
  EndRenderTargetFrame ‚Üí submit + fence-wait ‚Üí RT color transitions to SHADER_READ_ONLY

Pass 2: Post pass ‚Üí swapchain
  BeginFrame ‚Üí acquires swapchain image
    ‚Üí post pipeline (fullscreen, usesTexture=true, usesFrameUniforms=false)
    ‚Üí BindTexture(*rt) ‚Üí Draw(3)
  EndFrame ‚Üí submit + present (or capture + BMP in --shot mode)
```

Passes 0 and 1 use dedicated command buffers (`rtCmd_`, `rtPool_`, `rtFence_`) to avoid contention with the per-frame swapchain recorder. Both are submitted synchronously (fence-wait) before the swapchain pass begins, so there are no semaphore dependencies to track.

### Render-graph resource-state tracker + barrier solver

`render::RenderGraph` no longer relies on each pass to hand-place its own barriers. Every imported resource carries a current **state** (e.g. `RenderTarget`, `ShaderRead`, `TransferSrc`, `Present`); each pass declares the state it needs each resource in (read vs write). Before a pass runs, the **barrier solver** diffs each resource's prior state against the pass's required state and, when they differ, emits exactly one transition via `ICommandBuffer::TransitionResource(prev ‚Üí next)`, then advances the tracked state. On Vulkan that transition becomes an explicit `vkCmdPipelineBarrier2`; on Metal the tracked-hazard model makes it a no-op. The solver is pure host logic over `rhi/` types (no backend symbols) and is unit-tested (`render_graph_test`). Crucially the auto-inserted barriers are **proven hazard-free by the Khronos synchronization-validation layer**: running every showcase under sync validation yields zero `SYNC-HAZARD-*` lines, which is the oracle that the solver places correct, sufficient, and non-redundant barriers.

### Parallel command recording (worker pool)

Within a pass, draws can be recorded **across threads** without changing the output a single bit. A worker pool fans the pass's draw list out to N per-thread secondary command buffers; each thread records its slice into its own secondary, and the pass replays the secondaries **in deterministic creation order** (`vkCmdExecuteCommands` with the array in order on Vulkan; closing an `MTLParallelRenderCommandEncoder`, which commits its sub-encoders in creation order, on Metal). Because the replay order is fixed and independent of which thread finished first, a `--mt --workers 1` render and a `--mt --workers 4` render produce a **byte-identical** image (the `mt` golden + a 1-vs-N hash test, `parallel_record_test`, both enforce this). The ordering guarantee ‚Äî not thread count ‚Äî is what preserves determinism.

### Screen-space projected decals (`engine/render/decal.h`)

Screen-space decals (Slice BH) project a texture onto whatever scene geometry lies inside an oriented box volume, **reusing the G-buffer** rather than re-rasterizing geometry. A decal is a box: `BuildDecalTransform` builds its local‚Üíworld `TRS = Translate(center) ¬∑ R(eulerXYZ) ¬∑ Scale(2¬∑halfExtents)` so the local **unit cube** `[-0.5,0.5]¬≥` maps onto the world box, and its **inverse** (world‚Üídecal) is uploaded to the shader. In the decal composite pass the deferred fragment's world position is reconstructed from depth **exactly as SSR/SSAO reconstruct view space** (`ReconstructViewPos` + the same yFlip convention), then mapped view‚Üíworld via the camera `invView`; that world point is transformed into decal-local space, and if it lies `InsideUnitBox` the decal texture is sampled at the top-down `DecalUV` (`local.xz + 0.5`, the decal projects along local `-Y`) and alpha-blended over the lit scene with a smoothstep `EdgeFade` near the box faces.

The box-projection math (`BuildDecalTransform` / `WorldToDecalLocal` / `InsideUnitBox` / `DecalUV` / `EdgeFade`) is **header-only, pure CPU, zero backend symbols**, and is **shared** by both the in-shader composite (`shaders/decal.frag.hlsl`) and the unit test (`decal_test`) ‚Äî exactly the `render/ssr.h` pattern ‚Äî so the test pins the same math the GPU runs and the CPU side and shader stay mutually consistent. Golden-captured (`--decal-shot` / `--decal`, the `decal` golden).

### Screen-space global illumination ‚Äî raw / bilateral denoise / temporal (`engine/render/ssgi.h`)

Screen-space global illumination (Slice BP) adds **one bounce of indirect diffuse lighting in screen space**, and the depth pivot here is *reuse*: `engine/render/ssgi.h` is header-only, pure CPU (no device, no backend symbols) and **builds directly on the SSR machinery** rather than duplicating it. It `#include`s `render/ssr.h` and **re-exports** `ssr::ReconstructViewPos` / `ssr::ViewToScreenUV` (`using ssr::‚Ä¶`), so the SSGI math reconstructs the view-space position + normal from the G-buffer with the **same** functions SSR's reflection ray uses (RH view space, `-Z` forward, view-space linear depth `= -vpos.z`, the `yFlip` sign `-1` on Vulkan / `+1` on Metal). For each pixel it traces `K` cosine-weighted hemisphere rays about `N`, marches each in view space against the G-buffer using the **same march + binary-search refinement as `ssr.frag`**, and on a hit samples the **already-lit HDR scene color** as that ray's incoming radiance; the accumulated mean of the `K` hit radiances is the indirect-diffuse irradiance, which the composite multiplies by albedo and **adds** to the scene. The in-shader march (`shaders/ssgi.frag.hlsl`) copies `ssr.frag`'s `ReconstructViewPos`/`ProjectToUV` verbatim, so the CPU model and the shader stay bit-consistent. `SsgiParams` (`K=16`, `marchDist`, `thickness`, `intensity`) mirrors `SsrParams`. Golden-captured (`--ssgi-shot` / `--ssgi`, the `ssgi` golden); unit-tested (`ssgi_test`) against the shared kernel.

The single-frame `K=16` gather is **noisy**, so Slice BR adds an **edge-preserving bilateral denoise** of the indirect buffer (`render/ssgi.h`: `BilateralWeight` + `SsgiDenoiseParams`). `BilateralWeight` is a single pure function returning a neighbor tap's weight relative to the center as the product of a **spatial** Gaussian (pixel-distance¬≤/`spatialSigma`), a **range** term (indirect-color difference), and a **normal** term (G-buffer normal agreement) ‚Äî so the blur smooths noise *within* a surface but does not bleed indirect light across depth/normal edges. The denoise pass outputs the edge-aware weighted mean over a radius-2 (5√ó5) window (`sum/wsum`), and the **same math is mirrored verbatim in `shaders/ssgi_denoise.frag.hlsl`**. It is deterministic (no time/RNG) ‚Äî two runs are byte-identical ‚Äî and golden-captured (`--ssgi-denoise-shot` / `--ssgi-denoise`, the `ssgi_denoise` golden). Both SSGI passes carry **zero backend symbols** and follow the established `render/ssr.h` "shared CPU math == shader math" discipline.

The spatial denoise attacks the noise *within* a frame; Slice BV adds the orthogonal axis ‚Äî **temporal accumulation** ‚Äî completing the GI trilogy `raw ‚Üí spatial-denoise ‚Üí temporal`. The `--ssgi-temporal-shot` path renders the **same** Cornell scene/camera/gather but runs a **fixed `N=8`** loop in which each iteration performs a full SSGI gather with a **different golden-angle-rotated hemisphere kernel** (the `ssgi.frag` `frame` param drives the per-frame rotation) and accumulates a **running mean** of the per-frame indirect buffers into a ping-pong HDR accumulation target before the composite multiplies by albedo and adds to the scene. Averaging eight decorrelated (golden-angle-jittered) gathers converges to a much cleaner indirect-diffuse result than the single `K=16` frame. The accumulation count `N=8` matches `taa::kAccumFrames`; because the showcase camera is **static**, the accumulation needs **no reprojection** (history-reprojection under camera motion is a future slice ‚Äî this is the fixed-camera, no-reprojection running mean). Deterministic two-run byte-identical (no time/RNG/clock ‚Äî the jitter is a fixed golden-angle sequence), golden-captured (`--ssgi-temporal-shot` / `--ssgi-temporal`, the `ssgi_temporal` golden), carrying **zero backend symbols**.

### GPU multi-draw-indirect batching (`engine/render/mdi.h`)

GPU multi-draw-indirect (Slice BM) collapses **N distinct objects into a single indirect draw call**. `render::mdi::BuildBatch` is a header-only, pure-CPU builder (no device, no backend symbols) that, from a list of `DrawObject`s, lays out the two GPU buffers one `DrawIndexedMultiIndirect(drawCount=N)` consumes:

1. An **indirect-args** buffer of N `MdiCommand` records. The record is the standard 5√óu32 `{indexCount, instanceCount, firstIndex, vertexOffset, firstInstance}` ‚Äî byte-identical (and `static_assert`-pinned at 20 bytes) to `VkDrawIndexedIndirectCommand` and `MTLDrawIndexedPrimitivesIndirectArguments`, so the packed array uploads straight into a `BufferUsage::Indirect` buffer with `stride = sizeof(MdiCommand)`. `instanceCount` is always 1 and `firstInstance` always 0.
2. A **per-draw storage** buffer of N `PerDraw` records (`{ float model[16]; float material[4]; }`, std430-friendly, 80 bytes). The MDI vertex shader (`shaders/lit_mdi.vert.hlsl`) reads `PerDraw[gl_DrawID]` ‚Äî the SPIR-V `DrawIndex` built-in, **not** `firstInstance` ‚Äî so command i and `PerDraw[i]` describe the same object and the `firstInstance` field stays free.

The seam gains two defaulted methods on `ICommandBuffer`: **`DrawIndexedMultiIndirect(argsBuffer, drawCount, stride)`** and **`BindPerDrawData(perDraw)`**. On **Vulkan** these lower to a true `vkCmdDrawIndexedIndirect(argsBuffer, 0, drawCount, stride)` plus binding the `gl_DrawID`-indexed per-draw SSBO; both default to no-ops so passes/backends without MDI are unaffected, and the **Metal golden renders the identical scene per-object** (the per-object reference path). Render-invariance is structural: because the per-object reference path pushes the *same* `PerDraw[i]` values and issues the *same* index slices, the one-MDI-call render is mathematically identical to N per-object draws, and the single builder is the sole source of truth for both arrays. The builder is deterministic (a plain ordered memcpy into two POD arrays), unit-tested (`mdi_test`), and golden-captured (`--mdi-shot` / `--mdi`, the `mdi` golden); the showcase reports `drawCalls:1, refDrawCalls:144` and `mdi==per-draw: BYTE-IDENTICAL`.

### Bindless textures (`engine/render/bindless.* + engine/rhi_vulkan/vulkan_bindless.*`)

Bindless textures (Slice BZ) are the texturing analogue of MDI and complete the GPU-driven story: instead of rebinding a per-material descriptor set before each draw, **all** scene textures are bound **once** as a single large descriptor **array** and selected per-draw by an integer `texIndex`. The above-seam half is `render/bindless.h` ‚Äî a header-only, pure-CPU **texture-index table / interner** (`BuildBindlessTable`): it assigns each distinct texture a **stable first-insertion index**, de-duplicates repeats, and is the **single source of truth** for which array slot every draw samples. It carries **zero backend symbols** and is unit-tested (`bindless_test`: interning, material-coverage, determinism, ASan-clean).

The descriptor-indexing implementation lives **only below the seam** in `engine/rhi_vulkan/vulkan_bindless.*`. The seam gains an additive, pure-interface trio: **`CreateBindlessTextureSet(span<ITexture*>)`** ‚Üí an **`IBindlessTextureSet`**, **`BindBindlessTextures(set)`**, and a `usesBindlessTextures` query. On **Vulkan** this is **true bindless**: a runtime sampled-image array (set 4) created with `PARTIALLY_BOUND` + `VARIABLE_DESCRIPTOR_COUNT` + `UPDATE_AFTER_BIND` (the `descriptorIndexing` device features enabled), bound once, with a `texIndex` push constant per draw; `shaders/lit_bindless.frag.hlsl` samples `gTextures[NonUniformResourceIndex(texIndex)]` (DXC emits `SPV_EXT_descriptor_indexing`). Both seam methods **default to no-ops**, so **Metal** inherits the bound path and renders the **identical** multi-texture scene per-material (image backend-identical) ‚Äî the Vulkan-true / Metal-bound-golden split, exactly as MDI is Vulkan-true / Metal-per-object. `lit_bindless.*` is Vulkan-only and intentionally **not** in the Metal MSL generation list.

Render-invariance is structural: `--bindless-shot` renders the bindless-array path **and** the per-material **bound** reference (same texture, same `LINEAR`/`REPEAT` sampler) and asserts the two captures are **byte-identical** (FNV), so the single index table feeds both paths. The showcase reports `textures:5, draws:100, textureBinds:1, refTextureBinds:5` and `bindless==bound: BYTE-IDENTICAL`, and is golden-captured (`--bindless-shot` / `--bindless`, the `bindless` golden).

### The GPU-driven pipeline ‚Äî combined pass + culled pass (`engine/render/gpu_driven.h` + `engine/render/gpu_culled.h`)

MDI (BM) and bindless (BZ) are the two halves of a modern GPU-driven renderer; Slices CB and CD **compose** them ‚Äî and then add the GPU as the culler ‚Äî into the complete `cull ‚Üí compact ‚Üí MDI + bindless` pipeline. Both builders are **header-only, pure-CPU, zero backend symbols**, and are each the **single source of truth** that feeds both the GPU-driven path and its per-object reference, so the two paths can never diverge.

**Fully-GPU-driven combined pass (Slice CB, `render/gpu_driven.h`).** `render::gpudriven::BuildBatch` merges the BM and BZ outputs into one record set. The combined per-draw record `GpuDrivenPerDraw` extends `render::mdi::PerDraw` (model at offset 0, `float4` material at 64) with a `texIndex` at 80, padded to **96 bytes** (std430-friendly); the fully-GPU-driven vertex shader (`shaders/lit_gpudriven.vert.hlsl`) reads `PerDraw[gl_DrawID].{model,material}` and passes `texIndex` **flat** to the fragment, which samples `gTextures[NonUniformResourceIndex(texIndex)]` (`shaders/lit_gpudriven.frag.hlsl`). So **one** `DrawIndexedMultiIndirect(N)` (the BM indirect-args buffer) **plus one** `BindBindlessTextures` (the BZ array, bound once) draw an entire **multi-material** scene ‚Äî the union of the two earlier capstones. The builder is the sole producer of the commands, the per-draw records, **and** the bindless table; `texIndex` is assigned by first-insertion interning (`render::bindless::Intern`), and the `ITexture*` is treated as an **opaque interning key** (never dereferenced), so the whole builder is unit-testable with no GPU (`gpu_driven_test`). On **Vulkan** the showcase issues one true `vkCmdDrawIndexedIndirect` + one descriptor-indexing array bind; **Metal** inherits the per-object **bound** path and renders the identical scene. Render-invariance is structural ‚Äî the per-object reference pushes the same model + material and binds the same texture each object's `texIndex` selects ‚Äî so `--gpudriven-shot` reports `drawCalls:1, textureBinds:1` (vs `refDrawCalls:100, refTextureBinds:101`) and `gpudriven==bound: BYTE-IDENTICAL`, golden-captured (`--gpudriven-shot` / `--gpudriven`, the `gpudriven` golden).

**Fully-GPU-driven-CULLED pass (Slice CD, `render/gpu_culled.h`).** This is the capstone: instead of laying out *all* objects unconditionally (CB), a **compute** pass frustum-culls the instances and **compacts the survivors** itself, then writes the **GPU-decided MDI `drawCount`** that the single indirect draw consumes ‚Äî the GPU both *decides which objects survive and draws exactly them*. `render::gpuculled::Cull` is the authoritative CPU **mirror** of `shaders/gpudriven_cull.comp.hlsl`, composing four proven slices: it runs the **identical** conservative `render::frustum::SphereOutside` test (AR) on the **identical** world bounding sphere `render::gpu_cull::InstanceWorldSphere` computes (AR), emits the survivors **in source-index order** via the **ordered single-workgroup prefix-sum compaction** (AR ‚Äî *not* an unordered `atomicAdd` append; the source-order scan is the determinism trick that makes the golden reproducible), and writes each survivor as a `render::gpudriven::GpuDrivenPerDraw` (CB) whose count is the MDI `drawCount` (BM) and whose `texIndex` selects its bindless slot (BZ). The compute pass writes the compacted survivors into the per-draw SSBO + the indirect command, then **one** `DrawIndexedMultiIndirect(drawCount = survivor count)` + **one** bindless bind render exactly them. Render-invariance is structural: the showcase also renders the same survivor set per-object **bound** (CPU frustum-cull ‚Üí draw only the K survivors, each texture bound) and asserts byte-identity; because the survivor set (the conservative sphere test) and the per-draw values are pure functions of scene + frustum and the cull math is shared with the validated AR reference, the images cannot diverge. `--gpucull-draw-shot` reports `total:144, drawn:107, cpuRef:107, drawCalls:1, textureBinds:1`, a matching `gpucull-draw-hash == ref-hash`, and `gpucull-draw==bound: BYTE-IDENTICAL`, golden-captured (`--gpucull-draw-shot` / `--gpucull-draw`, the `gpucull_draw` golden). Unit-tested (`gpu_culled_test`).

### Hi-Z occlusion culling ‚Äî completing the GPU-driven culling pipeline (`engine/render/hiz.h`)

Slice CJ adds the **occlusion** half of GPU-driven culling, completing the pipeline that AR (frustum cull ‚Üí compaction ‚Üí indirect draw) and CD (compute-cull ‚Üí MDI + bindless) began: now the GPU culls not just what's **outside the frustum** but what's **hidden behind already-drawn geometry**. A **depth pre-pass** rasterizes the opaque occluders, then `render::hiz` builds a **Hi-Z max-depth pyramid** ‚Äî a mip chain where each coarser texel holds the **maximum** (farthest) depth of the four finer texels it covers, so a single coarse texel conservatively bounds the depth of a screen region. The occlusion test for a candidate projects its world bounding box to a **screen-space AABB**, picks the Hi-Z mip whose texel size covers that AABB, and compares the candidate's **nearest** depth against the mip's max-depth: if the candidate is entirely farther than the closest occluder in that region it is **fully hidden** and culled. `render/hiz.h` is **header-only, pure-CPU, zero backend symbols** ‚Äî the pyramid reduction + the conservative screen-AABB test are the single source of truth, mirrored by the GPU path and unit-tested (`hiz_test`).

The test is deliberately **conservative** (it never culls a visible object ‚Äî only ones provably behind the depth bound), which is exactly what makes it **render-invariant**: the culled objects would have contributed **zero pixels** anyway, so the occlusion-culled render is **byte-identical to the frustum-only render**. `--hiz-cull-shot` runs both and asserts the hashes match: `total:31, frustumKept:31, occluded:24, drawn:7, cpuOccluded:24` and `hiz-cull==frustum-only: BYTE-IDENTICAL`. On **Vulkan** the candidates surviving frustum+occlusion are drawn via the CD MDI + bindless path; **Metal** renders the identical CPU-occlusion-culled survivor subset per-object bound. Golden-captured (`--hiz-cull-shot` / `--hiz-cull`, the `hiz_cull` golden). Combined with frustum culling, the GPU-driven culling pipeline is now **frustum + occlusion**.

### Clustered light culling / Forward+ (`engine/render/cluster.h`)

Slice CL is the **many-lights flagship**: a Forward+ light-culling pass that lets the lit shader touch only the lights affecting a given fragment's screen region. `render::cluster` divides the view frustum into a **`16√ó9√ó24` cluster grid** (3456 clusters ‚Äî `16√ó9` tiles across the screen, 24 exponential z-slices in depth), computes each cluster's view-space **AABB**, and `AssignLights` tests **every light's bounding sphere against every cluster's AABB** (sphere-AABB overlap), producing a per-cluster `{offset, count}` plus a flat **light-index list** that the lit pass reads (the set-3 storage buffers bound via `BindLightClusters`, exactly as the earlier `render/clustered.h` Forward+ path). The grid construction + sphere-AABB assignment is **header-only, pure-CPU, shared math, zero backend symbols** ‚Äî the single source of truth for both the GPU compute assignment and the CPU reference, unit-tested (`cluster_test`).

Determinism + render-invariance are proven two ways. First, the **GPU assignment is byte-identical to the CPU reference**: the compute pass and `render::cluster::AssignLights` produce the **same** flat index list (GPU assigned `5606` == CPU ref `5606`, with `maxPerCluster:34` inside the valid `(1, 96)` range), and the per-cluster brim buffer is two-run byte-identical (`brimByteIdentical:true`). Second, the **clustered shading is byte-identical to brute-force**: shading only each cluster's assigned lights produces the same image as shading every one of the 96 lights per fragment (`clustered==brute-force: BYTE-IDENTICAL`). The showcase reports `lights:96, clusters:3456, maxPerCluster:34, avgPerCluster:1.62, brimByteIdentical:true`; golden-captured (`--clustered-lights-shot` / `--clustered-lights`, the `clustered_lights` golden), two-run byte-identical.

### Data-driven post-process stack (`engine/render/post_stack.*`)

The post-process stack (Slice BN) makes the final post chain **data-driven**: a `PostStack` is an **ordered** list of `PostEffect`s applied in sequence to the resolved scene color. The effect kinds are **ACES `Tonemap`**, **`ColorGrade`** (per-channel lift/gamma/gain), **`ChromaticAberration`** (radial RGB split, strength in pixels), **`Vignette`**, and **`FilmGrain`** (additive luminance noise from a fixed 32-bit integer hash of the *integer* pixel coord ‚Äî no time/frame input, so golden-stable). The per-effect math lives in `engine/render/post_stack.h` and is **shared** by both the in-shader stack (`shaders/post_stack.frag.hlsl`) and the unit test (`post_stack_test`) ‚Äî the same `render/ssr.h` / `render/decal.h` sharing pattern ‚Äî so the CPU mirror and the GPU evaluate bit-for-bit identically (the grain hash uses defined 32-bit unsigned wraparound so HLSL `uint` math matches).

The shader encodes the whole chain into a **flat float push-constant stream** to honor the **128-byte guaranteed `maxPushConstantsSize` minimum**: a `float4[7]` (28 floats = 112 B) + an `int4` count (16 B) = exactly **128 bytes**. Each effect contributes `kind` followed by that kind's params, and the shader walks the stream with a running cursor, applying each effect in order ‚Äî a per-effect `vec4` array would overflow a 128-byte GPU, so the flat-stream layout is the binding-budget constraint that shapes the design. The stack is **data** (`LoadPostStack` parses it from JSON via the vendored single-header parser; an absent/empty `"effects"` array yields a pass-through, an unknown `"kind"` fails the load) so an agent can compose looks. Pure CPU, zero backend symbols; deterministic two-run byte-identical; golden-captured (`--poststack-shot` / `--poststack`, the `poststack` golden) ‚Äî the showcase's `DefaultShowcaseStack` is the 5-effect cinematic chain tonemap‚Üícolorgrade‚Üíchromatic‚Üívignette‚Üígrain (`count:5`).

### Sky / atmosphere suite ‚Äî water, depth of field, volumetric clouds, cloud shadows (`engine/render/water.h`, `engine/render/dof.h`, `engine/render/clouds.h`)

Slices CF‚ÄìCK complete the **sky / atmosphere suite** (sky + water + clouds + cloud shadows) and a companion lens effect. All three follow the established `render/ssr.h` discipline: a **header-only, pure-CPU, zero-backend-symbol** math core that is the single source of truth **shared verbatim** with the in-shader pass, evaluated **deterministically at a fixed time** (no clock / RNG), so a two-run capture is byte-identical and the Metal golden matches the Vulkan render.

- **Water (Slice CF, `engine/render/water.h`).** A **Gerstner-wave** water surface: a fixed set of summed Gerstner waves (`GerstnerDisplace` ‚Äî per-wave amplitude / wavelength / steepness / direction, evaluated at a fixed `time`) displaces a `gridN√ógridN` water mesh on the CPU and the shared shader, and shading adds a **Fresnel-weighted reflect/refract** mix (Schlick Fresnel selecting between a reflected-sky term and a refracted-depth-tinted term). Golden-captured (`--water-shot` / `--water`, the `water` golden), reporting `waves:3, time:1.3, gridN:128`, two-run byte-identical.
- **Depth of field (Slice CG, `engine/render/dof.h`).** A **thin-lens circle-of-confusion (CoC) depth gather**: `CoC(depth)` computes each pixel's blur radius from a thin-lens model (`focalDist` / `aperture` / `maxCoC`) over the reconstructed view-space depth, and a fullscreen pass gathers neighbours weighted by their CoC to composite a focus-plane-sharp / out-of-focus-blurred image. The CoC + gather math lives in the header and is mirrored in `shaders/dof.frag.hlsl`. Golden-captured (`--dof-shot` / `--dof`, the `dof` golden), reporting `focalDist:12.0, aperture:90.0, maxCoC:18.0`, two-run byte-identical.
- **Volumetric clouds (Slice CH, `engine/render/clouds.h`).** A **raymarched cumulus** layer: a fullscreen pass marches a fixed `steps` count through a cloud slab, sampling a procedural density field (`coverage`-thresholded), accumulating **Beer-Lambert** transmittance for self-shadowing and an **Henyey-Greenstein (HG) phase** term for forward sun scattering, then composites the sunlit cloud over the sky. The density / Beer-Lambert / HG math is shared with `shaders/clouds.frag.hlsl` and evaluated at a fixed `time`. Golden-captured (`--clouds-shot` / `--clouds`, the `clouds` golden), reporting `steps:64, coverage:0.42, time:2`, two-run byte-identical. - **Cloud shadows on the ground (Slice CK, `engine/render/clouds.h` ‚Äî the `CloudShadow` addition).** The cloud field casts **dappled ground shadows** by attenuating the **direct sun** on the lit scene. `CloudShadow(worldPos, sunDir, time)` marches a fixed `steps` **sun ray** from each shaded world point up through the same `coverage`-thresholded procedural cloud slab, accumulating **Beer transmittance** along the ray ‚Äî the fraction of sunlight that survives the cloud ‚Äî which scales the direct-light term in the lit pass. It reuses the same density field as the CH cloud raymarch (one source of truth for both the sky clouds and their ground shadows), is header-only / pure-CPU / zero-backend-symbol, shared verbatim with the lit shader, and evaluated at a fixed `time`. This is the fourth member of the suite (sky + water + clouds + **cloud shadows**).

### Ground-truth AO, froxel volumetric fog (+ clustered-light injection + volumetric shadows), auto-exposure, contact shadows (`engine/render/gtao.h`, `engine/render/froxel.h`, `engine/render/auto_exposure.h`, `engine/render/contact_shadows.h`)

**Cross-backend FP discipline (what makes the GPU==CPU bit-exactness real).** A naive GI compute pass would diverge from a CPU reference because GPUs evaluate transcendentals (`sin`/`cos`/`sqrt`) and fused multiply-adds with different rounding than the host. The GI arc sidesteps that entirely: every transcendental is **host-precomputed once** (the Fibonacci-sphere ray directions, the SH sample directions, the `SHBasis9` weights, the per-sample solid-angle weights) and **uploaded as exact bits** that both the GPU pass and the CPU reference read identically, and every accumulation uses **`fma`** so the multiply-add rounds the same way on both sides. The result: the per-probe SSBO the GPU writes is **bit-identical** (`memcmp`) to the CPU reference *and* bit-identical across Vulkan and Metal ‚Äî the same contract the `gpu_cull` gpu==cpuRef count proof established, now extended to floating-point GI buffers.

No new interface method.

- **Vis-buffer render (Slice DW, `engine/render/visbuffer.h`).** The DS‚ÄìDV cluster MDI draw rasterizes `PackVisId(clusterID, triID) = (clusterID << 7) | triID` into an `R32_Uint` target (depth-tested so the front-most surface wins per pixel; `SV_PrimitiveID` supplies the triangle ID). The written value is a flat integer ‚Äî no interpolation/FP/blend ‚Äî so it is inherently bit-exact cross-vendor. **Proof B (bit-exact, vendor-robust, via raw `ReadRenderTarget` memcmp ‚Äî never depends on matching the hardware rasterizer's edge rules):** every written texel decodes to a valid survivor (`clusterID < survivorCount`, `triID < that cluster's triCount`) and the distinct clusterIDs present are exactly the on-screen subset of the CPU cull's survivors; two renders are byte-identical; and at deterministically-chosen cluster-interior pixels the GPU ID equals the CPU-predicted covering cluster. The image golden CPU-colors the read-back IDs by `hashColor(clusterID)` ‚Üí identical on both backends by construction. Golden `visbuffer` (the clustered spheres as per-cluster ID patches). (Cross-backend caveats handled: `SV_PrimitiveID` requires MSL 2.2 on Metal ‚Äî isolated per-shader so other goldens are untouched; and the integer-target clear uses a `0xFF000000` background sentinel that is float32-lossless and decodes outside the valid ID range, since `0xFFFFFFFF` reinterpreted through the float clear path becomes NaN‚Üí0.)
- **Deferred material resolve (Slice DX, `engine/render/visresolve.h`).** A fullscreen pass **texel-fetches** the integer vis-buffer (`Texture2D<uint>.Load`, `OpImageFetch` / MSL `texture.read`, no sampler ‚Äî reusing the existing `BindTexture`, no new RHI), reconstructs the covering triangle's vertices from the cluster/index/vertex SSBOs, computes a **flat** geometric normal + Lambert, and shades ‚Äî the spheres rendered via the vis-buffer instead of a forward lit pass. **Honest proof scope:** a byte-identical "resolve == forward render" proof is *not* cross-vendor-feasible (hardware-rasterizer barycentric/fill-rule divergence the FP discipline can't close ‚Äî even flat shading depends on which triangle covers each edge pixel). So DX proves **ID-provenance** (the resolve reads the correct IDs at interior pixels ‚Äî integer equality inherited from DW) + **GPU==CPU resolve-math** bit-exact (the flat-normal + Lambert at interior pixels matches the `ResolveFlatShade` CPU mirror, `std::fma` + host-precomputed light), with cross-backend guaranteed by the same-shader **image golden** and a forward-render **smoke bound** (resolve-vs-forward ‚â§ ~1 LSB, explicitly not 0.0000). Golden `visresolve` (the flat-shaded faceted spheres lit from the vis-buffer). A runtime cluster-DAG simplifier is the deferred stretch.
- **Software rasterization (Slices SW1‚ÄìSW2, `engine/render/swraster.h` + `shaders/swraster.comp.hlsl`).** The alternative path that *produces* the visibility buffer for sub-pixel / tiny triangles where a hardware rasterizer is inefficient: a deterministic **integer edge-function** scan-converter that packs `depth|id` (depth high bits, the same `PackVisId` low bits) into a per-pixel buffer and resolves the front-most surface by a **`min` over the 32-bit key**. SW1 is the pure-CPU reference (`swraster.h`): int64 edge functions, a watertight **top-left fill rule** (a shared cluster edge covers each boundary pixel exactly once ‚Äî no gap, no double-cover), and host-snapped fixed-point screen verts. SW2 is the GPU compute version (`swraster.comp.hlsl`, one thread per triangle) writing the same packing into a `RWStructuredBuffer<uint>` via **`InterlockedMin`** ‚Äî **no new RHI** (the existing `InterlockedAdd` storage-buffer family; no storage image, no 64-bit atomic). **The make-or-break proof:** the GPU vis-buffer is BIT-IDENTICAL to the CPU `swraster.h` reference at *every* pixel (full-frame `ReadBuffer` memcmp) ‚Äî guaranteed because the GPU consumes the same integer screen verts (zero GPU FP), runs the verbatim integer edge math (vendor-independent coverage), and the `InterlockedMin` is commutative (thread-race cannot change the result). This is *stronger* than the hardware vis-buffer's interior-only oracle ‚Äî there is no edge fuzz. Goldens `swraster` (CPU) and `swraster_gpu` (a clustered sphere, watertight, depth-occluded, plus a sub-pixel triangle the hardware path would miss). (Cross-toolchain note: the int64 edge math compiles under DXC for the Vulkan SPIR-V path but not glslang's HLSL frontend, so `swraster.comp` is Vulkan-only and Metal renders the identical result via the CPU `swraster.h` path ‚Äî the `cluster_cull.comp` convention; the Vulkan side carries the GPU==CPU proof.) Slice SW4 then **feeds the SW vis-buffer into the deferred resolve**: a fullscreen int32 "unpack" blit (`swraster_resolve_blit.frag`) strips the depth high-bits and writes the low-16 `visId` into the `R32_Uint` target the DX resolve already reads, so `visresolve.frag` is reused byte-for-byte ‚Äî completing a full software-raster ‚Üí deferred-shade pipeline (golden `swraster_resolve`, a Lambert-lit software-rasterized sphere; the integer vis-buffer + blit are cross-backend bit-identical, the float Lambert resolve is the same per-backend-golden + interior-bit-exact bar as DX). The HW-large/SW-small hybrid dispatch is the remaining deferred stretch.

### Virtual shadow maps (`engine/render/vsm.h`, namespace `hf::render::vsm`)

Slices VA‚ÄìVC add a **clipmap-organized directional Virtual Shadow Map** ‚Äî UE5's Nanite-scale shadowing reduced to its byte-exact-provable core: a huge virtual shadow resolution sparsely backed by physical pages, allocated only where the visible receivers need them. It reuses the existing shadow-atlas RHI (`CreateShadowMap` + `SetViewport` tiling + `SetShadowMap`) and the flat-SSBO + readback discipline wholesale, so the whole arc needs **no new RHI**. The page table, allocation, and per-pixel page selection are all integer-deterministic ‚Üí bit-exact GPU==CPU; the shadowed image is a normal lit golden.

- **Page table + page-needed marking (Slice VA, `engine/render/vsm.h`).** A clipmap of `levels` rings, each covering `level0WorldExtent ¬∑ 2^L` world units across `virtualPagesPerSide¬≤` pages, centered on the camera. A compute pass marks which virtual pages the visible receivers need: per receiver, `level = SelectClipmapLevel(distToCamera)` ‚Äî selected by an **integer threshold-ladder** against host-precomputed `level0WorldExtent¬∑2^L` bounds, *not* a GPU `log2`, so the level (and thus the whole resident set) is bit-identical cross-backend ‚Äî then project to `(px,py)` and mark `resident[pageId]=1`. **Proof:** the GPU-marked resident set equals the CPU `vsm.h` reference byte-for-byte (`ReadBuffer` memcmp, integer set, no FP); two runs identical; `markingEnabled=false` ‚Üí empty. Golden `vsm_pages` (a top-down clipmap viz, resident pages `hashColor`-lit over concentric rings, CPU-colored from the bit-exact set ‚Üí identical both backends by construction).
- **Physical-page depth rendering (Slice VB, `engine/render/vsm.h`).** A deterministic allocator assigns each resident virtual page a physical atlas tile (an integer virtual‚Üíphysical indirection table, bit-exact), and each resident page's shadow casters are rendered into its tile via the CSM `SetViewport` atlas loop, each with that page's clipmap-level ortho sub-frustum (`PageWorldOrtho`). **Proof:** the indirection table is two-run byte-identical and matches the CPU `AllocatePhysicalPages` reference; `markingEnabled=false` ‚Üí empty atlas. (Since a `CreateShadowMap` depth atlas has no color image for `ReadRenderTarget`, the VB *golden* renders depth-as-grayscale into a color RT ‚Äî a viz-only trick; the actual shadow sampling in VC uses a real depth atlas.) Golden `vsm_atlas` (the physical tiles, each holding its page's caster-depth silhouettes).
- **Lit-pass indirection sample (Slice VC, `shaders/lit_vsm.frag.hlsl`).** A sibling lit pass shadows each receiver pixel through the VSM: compute its clipmap level + virtual page (the VA math copied verbatim), look up the physical tile in the indirection SSBO, project into that page's light-space, and sample the tile's depth with tile-clamped 3√ó3 PCF ‚Äî the standard CSM depth-compare semantics, with the tile origin fetched from the page table instead of a cascade index. The VSM atlas binds as a depth-only shadow map via the existing `SetShadowMap`; the indirection SSBO via the existing `BindLightClusters` fragment-storage path ‚Äî **no new RHI**. **Make-or-break no-op:** `vsmEnabled=0` forces the shadow factor to a literal `1.0` ‚Üí the render is BYTE-IDENTICAL to the unshadowed lit; **page-lookup proof:** at interior receiver pixels the `(level, pageId, tile)` the shader resolves equals the CPU `vsm.h` reference (integer, bit-exact); `vsmEnabled=1` casts coherent shadows (differs from A), and a single-clipmap-level VSM is within a small epsilon of the existing CSM shadow (an equivalence smoke, not a byte match). Golden `vsm_shadow` (the lit scene with VSM-cast shadows). Per-page caching is the deferred stretch.

### Runtime virtual texturing (`engine/render/vt.h`, namespace `hf::render::vt`)

Slices VT1‚ÄìVT5 are a complete **Runtime Virtual Texturing** core (UE5's literal RVT): a virtual texture far larger than VRAM, sampled through a virtual‚Üíphysical page indirection table driven by per-pixel **feedback**. It is structurally the **same machinery as the VSM clipmap arc applied to texture pages** ‚Äî a virtual page table ‚Üí a feedback/marking pass ‚Üí physical tile allocation + indirection ‚Üí (later) a sampled lookup ‚Üí per-page caching ‚Äî and reuses the VSM page-table/allocator discipline near-verbatim, so the whole arc needs **no new RHI**. UV-space page math has no clipmap-snap/world-projection, making RVT structurally *simpler* than VSM; the page IDs / residency sets / indirection tables / procedural page content are pure integer, so **every slice of the VT1‚ÄìVT5 core is cross-backend bit-identical** (the strict zero-differing-pixel posture, not the float-smoke bar) ‚Äî only the deferred bilinear/3D-textured refinement introduces float.

- **Page-needed feedback / marking (Slice VT1, `engine/render/vt.h`).** The virtual texture is a **mip pyramid**: `PageId(mip, px, py) = mipPageOffset(mip) + py¬∑pagesPerSide(mip) + px`, where `pagesPerSide(m) = virtualPagesPerSideMip0 >> m` and the per-mip offset is a host **prefix-sum** of `pagesPerSide(k)¬≤` (unlike VSM's flat `level¬∑vpps¬≤`, since mips differ in size) ‚Äî a bijection with `UnpackPageId`. A compute pass marks which pages a set of (UV, mip) sample-requests need: `VtPageId` quantizes `floor(u¬∑pagesPerSide(mip))` ‚Üí `feedback[pageId]=1` (an order-independent integer set, no atomics). The one float‚Üíint boundary (the UV quantize) is eliminated entirely by **host-snapping** each request to integer `(mip,px,py)` before upload (the `swraster.h` discipline), so the GPU does zero floating point and the feedback set is trivially bit-exact. Mip selection uses the VSM integer threshold-ladder `SelectMipLevel` (no `log2`). **Proof:** the GPU feedback set equals the CPU `MarkFeedbackPages` reference byte-for-byte (`ReadBuffer` memcmp); `feedbackEnabled=false` ‚Üí empty; two runs identical. Golden `vt_feedback` (a per-mip page grid, resident pages `hashColor(pageId)`-lit ‚Äî CPU-colored from the integer set ‚Üí identical both backends by construction).
- **Physical tile-pool allocation + indirection (Slice VT2, `engine/render/vt.h`).** A finite physical tile pool (`VtTilePool`, `tilesPerSide¬≤` tiles) and a **deterministic ascending-priority allocator** (`AllocatePhysicalTiles`, the VSM `AllocatePhysicalPages` analog): walk pageId ascending, each resident page takes the next sequential tile while the pool has room, else `kNoTile` (non-resident or overflow). Ascending pageId is mip-major = **finest-mip-first** priority, so the nearest/highest-detail pages win the pool when it overflows. Allocation is inherently sequential, so the GPU mirror is a single-thread compute scan (`shaders/vt_alloc.comp.hlsl`) ‚Äî **no new RHI**. **Proof:** the GPU indirection table equals the CPU `AllocatePhysicalTiles` reference byte-for-byte; `allocated = min(resident, capacity)`, tile indices unique and in-range, the first `capacity` resident pages by pageId are the allocated ones; `allocEnabled=false` ‚Üí all `kNoTile`; two runs identical. Golden `vt_alloc` (the per-mip indirection viz: allocated pages tile-colored, overflow-resident a dim color, non-resident dark ‚Äî bit-identical both backends).
- **Procedural page generation into the physical atlas (Slice VT3, `engine/render/vt.h` + `shaders/vt_pagegen.comp.hlsl`).** GENERATE each allocated page's content into its physical tile ‚Äî the analog of VSM-VB's physical-page render, writing procedural texture color instead of caster depth, kept pure-integer. `PageTexel(pageId, mip, lx, ly)` is a deterministic per-texel generator (an FNV/avalanche hash of `(pageId, mip)` for the per-page base color, modulated by an 8√ó8 integer checkerboard + a coarse gradient over the page-local `(lx, ly)`; every channel floored into `[0x40,0xFF]` so it never collides with the `kAtlasClear = 0xFF101010` sentinel). A compute pass (one thread per atlas texel, a race-free map via a host-built `tilePageId` reverse table) fills the physical atlas SSBO; pure shifts/xor/add/mul ‚Üí no float. **Proof:** the GPU atlas equals the CPU `GeneratePhysicalAtlas` reference byte-for-byte; exactly the allocated tiles are non-clear and each is fully written; `genEnabled=false` ‚Üí all clear; two runs identical. Golden `vt_atlas` (the physical atlas: each allocated tile a distinct tinted patterned page, unallocated dark ‚Äî bit-identical both backends).
- **Material-pass sample through the indirection (Slice VT4, `engine/render/vt.h` + `shaders/vt_sample.comp.hlsl`).** The round-trip that makes RVT a usable texture: `SampleVirtualTexel(vx, vy, mip)` computes the virtual page ‚Üí looks up `indirection[pageId]` ‚Üí offsets into the physical atlas via `PhysicalTileOrigin` ‚Üí reads the texel. **Design call: NEAREST (integer) sampling**, so the sample path stays bit-exact and the golden is cross-backend bit-identical ‚Äî bilinear filtering and perspective-correct UV on real geometry are an explicit deferred float refinement (the round-trip is proven first, then the filter). A compute pass reconstructs the virtual texture's mip-0 image in virtual-UV space (one thread per virtual texel); a non-resident page samples `kVtMiss`. **Proof:** the GPU image equals the CPU `ReconstructVirtualImage` reference byte-for-byte; every resident virtual texel reads back exactly what VT3 wrote (`== PageTexel`, the indirection is a correct virtual‚Üíphysical map); page-lookup matches the CPU at interior texels; `sampleEnabled=false` ‚Üí all miss; two runs identical. Golden `vt_sample` (the reconstructed virtual texture: resident pages textured at their UV location, non-resident pages the miss color ‚Äî visually distinct from VT3's physical-atlas layout, bit-identical both backends).
- **Per-page caching across frames (Slice VT5, `engine/render/vt.h` + `shaders/vt_cachegen.comp.hlsl`).** Skip regenerating physical tiles whose page content is unchanged ‚Äî the analog of VSM-VD's `PageContentKey`/`VsmPageCache`. `PageContentKey(pageId, mip, contentVersion)` is an FNV hash of the page's content identity; a per-tile cache stores `(key, valid)`; a tile regenerates iff its key changed. `PageTexel` becomes `PageTexelV(‚Ä¶, contentVersion=0)` (version 0 reproduces VT3's bytes exactly ‚Äî the VT3/VT4 goldens are the guard). The cache-aware compute sibling early-returns for cached tiles, leaving the **persistent** atlas SSBO untouched. **Proof (the VSM-VD SHA-equality lineage, here direct `memcmp`):** Pass 2 re-runs the same scene ‚Üí all cache hits, atlas byte-identical to Pass 1 (cached == fresh); Pass 3 bumps one page's content version ‚Üí exactly one tile regenerates and the partial-cached atlas is byte-identical to a full uncached regen (the cache is transparent); two runs identical. Golden `vt_cache` (the cache-status viz: green-washed cached tiles + the one red regenerated tile + dark unallocated ‚Äî bit-identical both backends). The whole VT1‚ÄìVT5 core adds **no new RHI** and every slice is cross-backend bit-identical; the bilinear-filter / textured-3D-geometry float refinement is the deferred next step.

### GPU isosurface meshing (`engine/render/mc.h`, namespace `hf::render::mc`)

Slices MC1‚ÄìMC3 begin a **GPU Marching-Cubes isosurface mesher** ‚Äî extract a triangle mesh from a scalar (voxel/SDF) field, the foundation of real-time procedural & destructible geometry (a capability with no UE5 core-flagship equivalent). It decomposes into exactly the integer primitives this engine has golden-proven cross-backend, so the whole core is **cross-backend bit-identical** (the strict zero-differing-pixel bar) and adds **no new RHI**. The SDF field is generated host-side by a deterministic pure function (the `heightmap.h::Height` discipline ‚Äî no RNG/clock/disk).

- **Per-cell case classification (Slice MC1, `engine/render/mc.h` + `shaders/mc_classify.comp.hlsl`).** A `VoxelField` of `int32` quantized scalars; a locked 0‚Äì7 corner numbering (`kCornerOffset`) + 0‚Äì11 edge numbering. A compute pass classifies each voxel CELL into its 8-bit Marching-Cubes case index: `CaseIndex = Œ£ ((corner[i] > isovalue) ? 1 : 0) << i` ‚Äî eight pure integer compares, no transcendental, the cross-backend crux. **Proof:** the GPU case-index buffer equals the CPU `ClassifyCells` reference byte-for-byte; hand-checked known cells (`empty=0x00`, `full=0xFF`, `corner0=0x01`); disabled ‚Üí all-zero; two runs identical. Golden `mc_classify` (a grid of Z-slices through a sphere field, `hashColor(caseIndex)` per surface cell ‚Äî the surface cells form concentric rings, bit-identical both backends).
- **Per-cell triangle count (Slice MC2, `shaders/mc_count.comp.hlsl`).** The canonical 256-case triangle table (`kTriTable`, Bourke/Lorensen-Cline ‚Äî it matched the locked corner/edge numbering directly) and its derived `kTriCount`. The compute pass looks up each cell's triangle count and accumulates the grand total via `InterlockedAdd` ‚Äî an integer atomic that is commutative ‚Üí order-independent ‚Üí deterministic ‚Üí cross-backend bit-identical (the autoexposure-histogram precedent). **Proof:** per-cell counts AND the atomic total equal the CPU reference (5240 triangles for the test sphere); the table is self-consistent (`kTriCount` derived = `kTriTable`, empty/full = 0); disabled ‚Üí zero; two runs identical. Golden `mc_count` (the surface cells colored by triangle count).
- **Prefix-sum compaction + triangle emission (Slice MC3, `shaders/mc_scan.comp.hlsl` + `shaders/mc_emit.comp.hlsl`).** A single-thread exclusive prefix-sum of the per-cell counts (the VT2 single-thread-scan pattern ‚Äî the allocation order is sequential, so one thread keeps it GPU==CPU bit-exact) gives each cell its write offset; then one thread PER CELL emits its triangles into preallocated vertex+index SSBOs at disjoint ranges (race-free, no atomics). Vertices are cell-edge **midpoints** in half-grid integer units (`EdgeMidpoint` = the two corner grid-coords summed ‚Üí no division, no float); a triangle soup with identity indices (cross-cell vertex dedup deferred). **Proof:** the GPU prefix-offsets AND the full vertex+index buffers (15720 verts for the sphere) equal the CPU `MarchCells` reference byte-for-byte; a hand-checked cell's first vertex; disabled ‚Üí empty; two runs identical. Golden `mc_emit` (a deterministic 2D orthographic projection of the generated mesh ‚Äî the sphere surface as a filled-disk point cloud, bit-identical both backends). The whole MC1‚ÄìMC3 core reuses the autoexposure `InterlockedAdd`, the VT2 single-thread scan, and the swraster fixed-point lattice.
- **Fixed-point interpolated vertices (Slice MC4, `EdgeInterp`).** Replace the cell-edge midpoint with the ACTUAL isosurface crossing: `t = (iso - s0)/(s1 - s0)` computed as a TRUNCATING integer divide into a `1/256` edge lattice (the swraster `kSub` trick) ‚Äî `t = ((iso - s0) * kSub) / (s1 - s0)`, clamped, `vertexPos = Ca*kSub + (Cb - Ca)*t`. The same truncating integer divide on C++/HLSL/MSL stays bit-identical (int32, no overflow for this field), so the smooth-surface mesh is still GPU==CPU byte-exact and cross-backend zero-diff. **Proof:** the interpolated vertex+index buffers equal the CPU reference byte-for-byte (15720 verts); every vertex lies on its edge and on the isosurface to lattice precision (`max |field - iso| ‚â§ 1`). Golden `mc_interp` (a smoother sphere projection than `mc_emit`). MC1‚ÄìMC4 are the strict cross-backend bit-identical integer core.
- **Lit render + smooth normals (Slices MC5‚ÄìMC6).** MC5 RENDERS the extracted mesh as a lit 3D surface through the EXISTING terrain lit-mesh pipeline (no new shader, no new RHI): the fixed-point mesh ‚Üí float world positions (`vert/kSub`) + flat per-face normals ‚Üí `BindVertexBuffer`/`DrawIndexed`. MC6 swaps the flat normals for per-vertex SDF FIELD-GRADIENT normals (`-‚àáf` via central differences) for smooth Gouraud shading. These two are the arc's FIRST FLOAT slices ‚Äî a perspective-lit raster render is cross-vendor float, so they are held to the **float bar** (the visresolve/SW4 discipline): the golden is baked on Metal, the gate is Metal-render == Metal-golden DIFF 0.0000 (deterministic) + mesh provenance (the rendered geometry IS the MC4 bit-exact mesh) + visual parity, with the Vulkan-vs-Metal delta the documented cross-vendor smoke (NOT zero, the engine's ~55‚Äì60-mean float-render baseline). Goldens `mc_render` (faceted lit sphere) + `mc_normals` (smooth lit sphere). This completes the 6-slice GPU Marching-Cubes flagship ‚Äî a voxel/SDF field meshed on the GPU (bit-exact) and rendered as a lit 3D surface, with zero new RHI across the whole flagship.

### Deterministic fixed-point physics (`engine/sim/fpx.h`, namespace `hf::sim::fpx`)

Slices FPX1‚ÄìFPX6 are a **complete deterministic fixed-point rigid-body simulator** (flagship #6) ‚Äî the first physics in the engine that is BIT-IDENTICAL across Vulkan/Metal/compilers AND frame-to-frame/run-to-run reproducible. The existing `engine/physics/` is float (held to a "visually identical at rest" bar); `engine/sim/fpx.h` is an additive, parallel, Q16.16-fixed-point reimplementation. The motivation is concrete and a genuine capability gap: the engine already ships snapshot replication + client-prediction, but float physics diverges across machines, so true cross-platform **lockstep / rollback netcode** (peers re-simulate from inputs alone to identical state) is impossible with it ‚Äî exactly the limitation UE5's (float) Chaos has. A fixed-point solver is the missing primitive, and FPX5 lands the proof (replica==authority bit-for-bit from inputs alone, on two platforms) ‚Äî the arc's beyond-UE5 headline. The whole arc adds **no new RHI** and **no new shader beyond FPX1‚ÄìFPX4's compute passes**; the integer/compute outputs are strict cross-backend bit-identical (the zero-differing-pixel bar), and only the final FPX6 render is float (the visresolve-bar).

- **Q16.16 integrator (Slice FPX1, `engine/sim/fpx.h` + `shaders/fpx_integrate.comp.hlsl`).** A `fx = int32` Q16.16 scalar with `fxmul(a,b) = (int64(a)*b) >> 16` (int64 intermediate); `FxVec3`; `FxISqrt`/`FxLength` (integer floor-sqrt copied from `mc.h::ISqrt` ‚Äî no `std::sqrt`, no `<cmath>` on the bit-exact path); `FxBody`/`FxWorld`; `IntegrateStep` (semi-implicit Euler + a ground floor-clamp). The compute pass consumes **host-snapped Q16.16 integers and does zero floating point**, so it is GPU==CPU bit-exact by construction (the swraster/VT1 host-snap discipline). **Cross-toolchain note:** `fxmul`'s int64 compiles under DXC (the Vulkan path) but glslc (the Metal HLSL‚ÜíSPIR-V‚ÜíMSL frontend) cannot parse `int64_t`, so `fpx_integrate.comp` is Vulkan-only and the Metal showcase runs the CPU `IntegrateStep` (byte-identical by construction ‚Äî the `swraster.comp` convention). **Proof:** GPU bodies == CPU reference byte-for-byte after K steps; disabled no-op; two runs identical; a hand-checked closed-form fall. Golden `fpx` (the settled bodies, CPU-colored from the integer read-back ‚Üí identical both backends).
- **Integer broadphase (Slice FPX2, `shaders/fpx_pair_{count,scan,emit}.comp.hlsl`).** An integer-AABB sweep generates the deterministic candidate collision-pair list: `AabbOverlap` is six **int32 compares** (no int64, no products) ‚Äî so unlike FPX1 these shaders MSL-generate natively and run as a true GPU pass on both backends. The count‚Üíprefix-sum‚Üíemit compaction reuses the MC3/VT2 pattern (per-body count, a single-thread exclusive scan, per-body emit at the offset). **Proof:** the per-body counts, prefix offsets, and the full `(i,j)` pair list all equal the CPU `BuildPairs` reference byte-for-byte. Golden `fpx_pairs` (the pair-adjacency matrix ‚Äî a banded structure of the grid neighbors).
- **PBD collision response (Slice FPX3, `shaders/fpx_solve.comp.hlsl`).** A position-based-dynamics positional solver resolves ground + sphere-sphere contacts over the FPX2 pairs: move bodies apart along the contact normal by their inverse-mass-weighted share of the penetration, K Gauss-Seidel iterations in a fixed order. `fxdiv = (int64(a) << 16) / b` (truncating Q16.16 divide) and `FxNormalize` (via `FxISqrt`) keep the impulse-free math fully integer. The solver is **single-thread** (`[numthreads(1,1,1)]`) ‚Äî sequential Gauss-Seidel is order-dependent, so one thread guarantees bit-exactness (the VT2 single-thread pattern); it is int64 ‚Üí Vulkan-only + the Metal showcase runs the CPU `StepWorld`. **Proof (the make-or-break):** the bodies after K integrate+solve steps equal the CPU reference byte-for-byte; the residual-overlap count is deterministic (bit-exact, not necessarily zero after K iterations); a hand-checked contact splits by the exact Q16.16 inverse-mass shares; two runs identical. Golden `fpx_solve` (the settled bodies).
- **Integer quaternion orientation (Slice FPX4, `shaders/fpx_orient.comp.hlsl`).** Promotes the sim to full 6-DOF: `FxQuat{x,y,z,w}` with `FxQuatMul` (the Hamilton product, each term an int64 `fxmul`), `FxQuatNormalize` (length via `FxISqrt` of the int64 Q32.32 sum-of-squares, `fxdiv` each component ‚Äî no `std::sqrt`), `FxRotate`, and `IntegrateOrientation`/`IntegrateBodyFull` (a body's rotation integrated from its angular velocity). `orient`/`angVel` are **defaulted** on `FxBody` so FPX1‚ÄìFPX3 and their GPU packs stay byte-identical. One thread per body (independent), int64 ‚Üí Vulkan-only + the Metal showcase runs the CPU `IntegrateBodyFull`. **Proof:** GPU == CPU byte-for-byte over K steps; the `|q|-1` normalization drift is ~zero; a hand-checked analytic spin. Golden `fpx_orient` (per-body orientation gizmos).
- **Lockstep + rollback (Slice FPX5, pure CPU ‚Äî `FxCommand`/`ApplyCommand`/`SimTick`/`SnapshotWorld`/`RestoreWorld`/`RunLockstep`/`RunRollback`).** The arc's headline, and a capability UE5's float Chaos cannot provide. ### Shadow-atlas tiling (`SetViewport`)

The expanded shadow set (Slices AD‚ÄìAF) renders **many** light-space depth maps per frame ‚Äî N directional cascades, a spot map, and the six faces of a point light's cube ‚Äî without allocating N separate depth textures. Instead the depth pass clears one shadow **atlas** once, then for each sub-map calls a new seam entry point, `ICommandBuffer::SetViewport(x, y, w, h)`, to restrict rasterization to that tile before drawing the geometry for that cascade/face. The lit pass samples the right tile by transforming into the corresponding light-space matrix and offsetting into the atlas region.

`SetViewport` is a defaulted no-op on the base interface, so passes and backends that don't tile an atlas are unaffected (the pre-existing single-shadow-map goldens are byte-for-byte unchanged). On Vulkan it records `vkCmdSetViewport`/`vkCmdSetScissor`; on Metal it sets the encoder viewport + scissor rect. The CSM frustum-split + per-cascade ortho fit, the spot `spotViewProj`, and the point-light 6-face cube view-proj + dominant-axis face/tile mapping are all pure, header-only, unit-tested math (`render/csm.h`, `render/spot.h`, `render/point_shadow.h`) ‚Äî the goldens prove the GPU side, the unit tests prove the math.

## Headless Capture and Golden-Image Testing

Headless capture is a first-class RHI feature, not a test harness hack.

`IRHIDevice::CaptureNextFrame()` arms a flag. ### Shaders are generated, not hand-written

The Metal shaders are **generated from the shared HLSL sources at build time** ‚Äî there is no hand-written MSL to drift from the canonical shaders. For each shader the `metal_headless` build runs HLSL ‚Üí SPIR-V (`glslc -x hlsl`) ‚Üí MSL (`spirv-cross --msl --msl-decoration-binding`), emitting `*.gen.metal`, which `visual_test` compiles at runtime via `newLibraryWithSource:`. The `--msl-decoration-binding` flag maps each resource's SPIR-V binding directly to its Metal `[[buffer/texture/sampler(n)]]` index; the engine's Metal binding constants (`engine/rhi_metal/metal_common.h`) are chosen to match. The only Metal-specific shader adjustments are guarded by `#ifdef HF_MSL_GEN` (two texture-origin V-flips), so the Vulkan SPIR-V is byte-identical.

## Vulkan Backend Notes

- **Vulkan version:** 1.3 dynamic rendering (`VK_KHR_dynamic_rendering`); no `VkRenderPass` / `VkFramebuffer` objects.
- **Synchronization:** Vulkan Synchronization 2 (`vkCmdPipelineBarrier2` / `VkImageMemoryBarrier2`). A `TransitionImage` helper in `vulkan_device.cpp` centralizes the barrier pattern.
- **Memory:** Vulkan Memory Allocator (VMA 3.3) for all images and buffers. Staging uploads use a host-visible VMA buffer, `vkQueueWaitIdle` after the copy (synchronous load-time path).
- **Frames in flight:** `kFramesInFlight = 2` (double-buffered). Per-frame fences + semaphores in `frames_[frameIndex_]`. Per-frame UBOs indexed by `frameIndex_`.
- **Shader pipeline:** HLSL ‚Üí SPIR-V via DXC (`-spirv`, invoked by a CMake custom command at build time). Shaders are loaded as `.spv` files at runtime via `IShaderModule`.
- **Depth:** swapchain-sized `VK_FORMAT_D32_SFLOAT` image owned by the device, recreated on resize.

## Scene Layer

`engine/scene/` sits above the RHI and below the application:

- `vertex.h` ‚Äî canonical `Vertex { pos[3], color[3], uv[2], normal[3], tangent[3] }` (stride 56, tangent added for normal mapping) and `MeshVertexLayout()`; a `SkinnedVertex` (stride 88, + joints/weights) and `SkinnedMeshVertexLayout()`; `InstanceTransformLayout()` (stride 64) for the per-instance stream.
- `transform.h` ‚Äî `Transform { position, eulerRadians, scale }` with `Matrix()` ‚Üí TRS `mat4`.
- `mesh.h/.cpp` ‚Äî `Mesh` owns `IBuffer* vertices` and `IBuffer* indices`; static factories `Cube(device)`, `Plane(device)`, `Sphere(device)`.
- `renderable.h` ‚Äî `Renderable { Mesh*, ITexture*, Transform }`; `scene_io` (load/dump) and `commands` (undoable edits) round-trip the scene; `instance_grid` builds the deterministic instancing field.
- `streaming.h/.cpp` ‚Äî distance-based scene / asset **streaming** (Slice BD), below.

The scene layer ‚Äî and the `asset`, `anim`, `physics`, `material`, `ui`, `audio`, `terrain`, `game`, `runtime`, `editor`, and `debug` modules above the seam ‚Äî has no knowledge of which backend is active. They call `IRHIDevice::Create*` through the seam, which is exactly why the backend-agnostic subset compiles into `hf_core` and runs clean under AddressSanitizer in the pure unit tests. This is the same code the Metal headless target uses to render its reference images.

### Scene / asset streaming (`engine/scene/streaming.*`)

`scene::StreamingWorld` (Slice BD) is **pure CPU above the seam** ‚Äî zero `vk*`/`MTL*`/`mtl::`/`Backend::` symbols; it depends only on `engine/math` + the stdlib and is compiled into both `hf_core` (ASan-scoped, `streaming_test`) and `hf_engine` (the `--stream-shot` showcase). A large world is divided into a fixed deterministic `N√óN` grid of **cells** laid out row-major on the `y=0` plane; `Update(cameraPos)` runs a three-step residency policy each frame:

1. **Desired residency** ‚Äî a cell within `loadRadius` of the camera wants to be Resident; a cell beyond `unloadRadius` wants to be Unloaded; cells in the **hysteresis band** (`loadRadius < dist ‚â§ unloadRadius`, with `unloadRadius > loadRadius` enforced) **keep their current state**, which is what prevents load/unload thrash as the camera jitters around a boundary.
2. **Enqueue** to-load (`Unloaded‚ÜíLoading`) and to-unload (`Resident‚ÜíUnloading`) by **ascending distance** (nearest first).
3. **Process under a per-frame budget** ‚Äî up to `loadBudgetPerFrame` loads finish (`Loading‚ÜíResident`, synchronously building the cell's procedural renderables) and `unloadBudgetPerFrame` unloads finish (`Unloading‚ÜíUnloaded`, freeing them) this frame, so a large working set streams in **over** frames rather than in one hitch.

The resident set is a **pure function** of (camera position, radii, budget, prior state). A cell's content is described as backend-free `CellRenderable` data (a primitive `Kind` + model matrix + material + palette index), and the showcase maps that to real GPU meshes/textures ‚Äî so the streaming module itself stays RHI-free and the unit test stays GPU-free. Driven by a fixed scripted camera path (no live input / RNG / clock) the per-frame `ResidentCellIds` sequence is bit-stable; the `--stream-shot` capture frame reports a fixed `frame:40, resident:24` line and is pinned by the `stream` golden. The "load" here is the **synchronous** construction of a cell's procedural renderables ‚Äî the MVP models the budget/over-frames behavior, not async disk I/O (that is a future slice).

### Animation state machine ‚Äî cross-fade FSM (`engine/anim/state_machine.*`)

`anim::StateMachine` (Slice BL) is a **parameter-driven finite state machine with cross-fade blending**, built **purely on top of** the existing skeletal-animation core (`anim::SampleAnimation` / `anim::BlendAnimations`). It is **CPU-only and produces a joint-matrix palette exactly like the existing sampler/blender already do**, so the **unchanged GPU skinning path consumes its output with no new RHI seam and no new shader** ‚Äî zero `vk*`/`MTL*` symbols.

An `AnimState` is a named clip (an index into a caller-provided `std::vector<Animation>`) with a per-state playback `speed` and a `loop` flag; a `Transition` is a parameter-threshold-gated edge (`paramIndex cmp threshold`, `Cmp::Greater|Less`) with a cross-fade `duration`, where `from == -1` means an **any-state** edge. `Update(dt)` advances a deterministic cursor: when not transitioning it accrues `stateTime += dt¬∑speed(current)` then evaluates the current state's outgoing edges (plus every any-state edge) **in the fixed order they were added ‚Äî first satisfied wins** ‚Äî and on a firing edge records the target, seeds `transitionTime = 0`, and starts both states' local times advancing; while transitioning both local times advance, `transitionTime += dt`, and at `transitionTime ‚â• duration` the transition **completes** (current = target). `Evaluate()` turns the cursor into a palette: not transitioning ‚Üí `SampleAnimation(current)`; transitioning ‚Üí **`BlendAnimations(from, to, weight = transitionTime/duration)`**, reusing the established blend convention.

The FSM output (current state, transition progress, blend weight, final palette) is a **pure function of (state graph, scripted parameter timeline, fixed-dt sequence)** ‚Äî no input / RNG / wall-clock ‚Äî so it is bit-stable run-to-run and across backends. Unit-tested (`anim_fsm_test`) and golden-captured (`--anim-fsm-shot` / `--anim-fsm`, the `anim_fsm` golden), scripted to a fixed mid walk‚Üírun cross-fade: `state:walk->run, blend:0.53, speed:0.925, step:37`.

## Terrain Layer (`engine/terrain/`)

`engine/terrain/heightmap.*` (Slice BF) is a **procedural terrain generator** that is pure CPU (`engine/math` + the scene `Vertex`/index types + the stdlib only) with **no RHI or backend symbols**. Crucially it is compiled into `hf_core`/`hf_engine` **and** into the standalone `metal_headless` target **from the same translation unit**, so the Windows/Vulkan and Apple/Metal builds generate a **bit-identical** mesh ‚Äî the cross-backend golden contract (same code ‚Üí same mesh ‚Üí same pixels).

- **`Height(x, z)`** is a fixed pure function: `A1*sin(f1¬∑x)*cos(f1¬∑z) + A2*sin(f2¬∑x+P2x)*cos(f2¬∑z+P2z) + A3*ValueNoise(x¬∑nf, z¬∑nf)`, where `ValueNoise` is a hash-lattice value noise (integer-lattice corner values from a 2D integer hash, bilinearly interpolated). It uses no RNG, no clock, and no float-time input, with the constants locked in `heightmap.cpp`, so it is deterministic and locale-independent.
- **`BuildTerrain(n, worldSize, heightScale)`** builds an `n√ón` vertex grid over `[-worldSize/2, +worldSize/2]¬≤` in the XZ plane, displacing each vertex by `y = Height(x,z) * heightScale`. **Per-vertex normals come from central finite differences** of `Height` (`N = normalize(cross(dz, dx))`); UVs are the grid coordinates in `[0,1]¬≤`; tangent is `+X` (matching the `Plane`'s convention); and a deterministic height-based **vertex-color tint** (low ‚Üí grass green, high ‚Üí rock/snow) reads the relief through the **unchanged** lit shader (which multiplies texture √ó vertex color), so **no new shader is required**. Indices wind two CCW (viewed from above) triangles per quad so the surface is front-facing under back-face culling. The returned `TerrainMesh` also carries `peak` (max world-space Y) as the deterministic stat.

It is unit-tested (`terrain_test`, which re-implements the locked formula independently and checks structure / counts / index range / corner positions / displaced Y / finite-difference normals / two-call bit-identity) and golden-captured (`--terrain-shot` / `--terrain`, the `terrain` golden); the showcase reports a fixed `n:128, verts:16384, tris:32258, peak:2.0972` line. Because the same TU feeds both the Vulkan sample and `metal_headless`, the generated mesh ‚Äî and therefore the golden ‚Äî is identical on both backends.

### Streamed terrain + distance-banded LOD (`engine/terrain/terrain_stream.*`)

`terrain::TerrainStreamWorld` (Slice BJ) is the **open-world capstone** that ties Slice BD (distance-based streaming residency) and Slice BF (procedural heightmap terrain) into a streamed, LOD-selected terrain field. Like both parents it is **pure CPU above the scene** ‚Äî zero `vk*`/`MTL*`/`mtl::`/`Backend::` symbols, depending only on `engine/math` + `engine/terrain/heightmap` + the stdlib ‚Äî and is compiled into `hf_core` (ASan-scoped, `terrain_stream_test`), `hf_engine` (the `--terrain-stream-shot` showcase), **and** the standalone `metal_headless` target, so the tile meshes are bit-identical cross-backend.

- **Tile grid.** A fixed `T√óT` grid of square **tiles** covers the world, the whole grid centered on the origin; tile `(i,j)` covers a `tileSize√ótileSize` XZ region at a fixed world offset (`id = j*T + i`, row-major). Each tile is meshed by `BuildTerrainTile` over **that tile's** region using the **global** `Height(x,z)` ‚Äî so adjacent tiles meeting at equal LOD share bit-identical edge vertices (seamless; no per-tile randomness).
- **Distance-banded LOD with hysteresis.** A tile's mesh resolution is one of three discrete levels chosen from its center's distance to the camera: `LOD0` (`n=96`, near), `LOD1` (`n=48`, mid), `LOD2` (`n=24`, far). `LodFor` is the pure `‚â§`-band function; `LodNext` adds **hysteresis** ‚Äî it refines (toward LOD0) immediately as a tile approaches, but only **coarsens** once the distance is past the band edge by a margin, so a tile sitting on a boundary doesn't flicker LOD. Per-LOD vertex/index counts (`verts == n¬≤`, `indices == (n-1)¬≤¬∑6`) are pinned by the unit test.
- **Residency policy.** `Update(cameraPos)` runs the BD load/unload-radius + hysteresis-band + per-frame-budget policy (nearest-first) to pick the resident tile set, then updates each resident tile's LOD, **rebuilding its mesh** on initial load or whenever its LOD changes.

The resident tile set **and** each tile's LOD are pure functions of (camera position, radii, budget, distance‚ÜíLOD bands, prior state); driven by a fixed scripted camera path (no live input/RNG/clock) the per-frame `{resident tiles + LODs}` is bit-stable run-to-run and across backends. Unit-tested (`terrain_stream_test`: tile coverage + shared-edge seamlessness, the band/hysteresis functions, the per-LOD counts, residency+LOD determinism, refine-on-approach rebuild) and golden-captured (`--terrain-stream-shot` / `--terrain-stream`, the `terrain_stream` golden); the showcase reports a fixed `frame:45, resident:22, lod0:4, lod1:8, lod2:10` line.

## Material / Shader Graph Layer (`engine/material/`)

Phase 4 adds a **data-driven material system** that authors shaders as node graphs instead of hand-written HLSL, while preserving the "shaders are generated, not hand-written" and golden-stability invariants. It is pure host logic above the seam ‚Äî no `vk*`/`MTL*`/`Backend` symbols ‚Äî and is split into five concerns plus a build-time tool:

- **`shader_graph` (`shader_graph.h/.cpp`)** ‚Äî the pure-CPU graph model: typed nodes (`Float`/`Float2`/`Float3`/`Float4` values on the edges) wired into the PBR fragment inputs through the `PBROutput` sink. The node set is `Constant`, `UV`, `TextureSample`, `Multiply`, `Add`, `Lerp`, and `Fresnel`, the **Slice-AZ expansion** ‚Äî `Swizzle` (mask over `xyzw`/`rgba`), `MakeFloat3`, `MakeFloat4`, `Dot`, `Normalize`, `Power`, `OneMinus`, and `Saturate` ‚Äî which together cover vector (de)construction, common math, and remapping, plus the **Slice-BE `NormalMap` node**, enough to author non-trivial materials as graphs. Output types are resolved structurally (`OutputType` propagates the matched input type through `Multiply`/`Add`/`Lerp`, etc.). No GPU types; fully unit-tested (`shader_graph_test`).

  - **The `NormalMap` node + `PBROutput.normal` (Slice BE).** The `PBROutput` sink gained a **5th input slot, `normal`** (tangent-space `float3`, default `(0,0,1)` = no perturbation), **appended last** so the original four inputs (`baseColor`/`metallic`/`roughness`/`emissive`) keep their order and semantics ‚Äî any graph that leaves `normal` unconnected codegens **byte-identically** to before, which is why every pre-BE golden is unchanged. The `NormalMap` node samples a named normal-texture slot (default `"normalmap"`) at a UV input and outputs `normalize(decode(sample))` with `decode(c) = c*2-1` (the standard tangent-space unpack; the decode is shared with the codegen via `EvalNormalMap`). When (and only when) a graph drives `PBROutput.normal`, codegen emits a perturbed-normal lighting path ‚Äî the **`hfShadePBRN` variant** of the shared PBR lighting core ‚Äî which builds a TBN from the interpolated tangent/normal and rotates the decoded tangent-space normal into world space before lighting, instead of the flat-normal `hfShadePBR`. The `mat_normal` golden (`--material-normal`) pins it.
- **`codegen`** ‚Äî lowers a validated graph to HLSL that drops into the existing `lit_pbr` fragment contract (same descriptor sets / push constants), so a graph material is just another PBR pipeline as far as the render graph and goldens are concerned.
- **`material_loader`** ‚Äî parses the JSON authoring format (`assets/materials/*.mat.json`) into a `ShaderGraph`.
- **`runtime_compile` (`runtime_compile.h/.cpp`)** ‚Äî the **runtime path**: it shells out to the `dxc` compiler as a **subprocess** (temp-file IO only; HLSL ‚Üí SPIR-V), then builds an `IPipeline` from the result. It fails **safe** ‚Äî returns `nullopt` + an error on any problem, never throws across the boundary ‚Äî so a bad live edit cannot crash the host.
- **`live_material` (`live_material.h`)** ‚Äî the in-editor authoring controller: it watches a material file and re-runs the loader ‚Üí codegen ‚Üí runtime compile chain on change, hot-swapping the pipeline for live iteration.

**Two production paths, proven equal.** A **build-time codegen tool** (`tools/material_codegen`) runs during the build to bake the showcase materials into committed generated HLSL (`shaders/generated/mat_showcase*.frag.hlsl`), which compile through the normal offline shader pipeline (the fast, shipping path). The **runtime** `runtime_compile` path generates and compiles the *same* graph live (the authoring path). The two are required to render **byte-identically**: `--material-shot` (build-time) and `--material-live-shot` (runtime) produce the same SHA, and the `mat_graph` / `mat_graph2` goldens pin both. This is the material analogue of the HLSL‚ÜíMSL "generated, not hand-written" rule ‚Äî the live authoring convenience can never silently diverge from what ships.

A **multi-material scene** (`--material-multi-shot` / `--material-multi`, the `mat_multi` golden) renders three spheres in a row, each shaded by a distinct graph material (`showcase` / `showcase2` / `showcase3`) with one draw per material binding that material's pipeline ‚Äî demonstrating that several independently-authored graph materials coexist in a single frame, all through the same build-time-generated path.

### Graph introspection ‚Äî JSON / DOT (`engine/material/graph_introspect.*`)

`material::DescribeGraphJson` / `ToDot` (Slice BI) close the loop on the data-driven material pipeline: having **authored** a graph (`*.mat.json` ‚Üí `ShaderGraph`) and **rendered** it (codegen / runtime compile), introspection lets an agent ‚Äî or a future visual node editor ‚Äî **inspect** it. Over the same in-memory `Graph`/`Node`/`Edge` model the loader and codegen use, it emits a deterministic, machine-readable description of the graph: as pretty-printed **JSON** (`material`, `nodeCount`, `edgeCount`, the resolved `output` PBROutput-sink id, then per-node `id`/`type`/`params`/typed input `ports` with their source-or-default + resolved output type, then the `edges`) or as a **Graphviz DOT** digraph (one labelled node per graph node with the PBROutput sink highlighted, one labelled edge per connection) for visualization.

It mirrors the engine-state introspection (`editor::DescribeEngine`) but for material graphs, and shares its determinism discipline: **HARD RULE** no `vk*`/`MTL*`/`mtl::`/`Backend::` symbols ‚Äî it is a pure string builder over the CPU graph model (the caller writes the strings), with nodes emitted in a **stable id-ascending order**, ports in declaration order, edges in stored order, floats formatted with the same `%g`-on-the-double convention the engine introspect JSON uses, and LF newlines ‚Äî so output is byte-identical run-to-run and cross-platform. Exposed via `--material-introspect <mat.json> [out.json]`; unit-tested (`graph_introspect_test`) and pinned by a byte-exact JSON golden: `DescribeGraphJson(load(showcase3.mat.json))` byte-equals `tests/golden/material/showcase3_graph.json` (the test reads the committed golden and compares, so CI checks it too).

## Text / HUD Layer (`engine/ui/`)

`engine/ui/` (`text.h/.cpp`) is a **baked-font text renderer**: it bakes a fixed 8√ó8 monospace bitmap font (`kFont8x8`, the printable-ASCII range 0x20‚Äì0x7E, one `uint8_t[8]` per glyph) into a single RGBA atlas with `BuildFontAtlas`, and lays a string out into a batch of screen-space textured quads in NDC with `LayoutText`. It is **pure CPU with zero RHI / backend symbols** ‚Äî no `vk*`/`MTL*`/`mtl::`/`Backend::` ‚Äî and uses no external asset, clock, or RNG, so the same string always produces the same atlas and the same quad batch, and the HUD goldens match on every backend. It is compiled into **both** `hf_core` (ASan-scoped, unit-tested via `text_test`) and `hf_engine`.

The **draw side is additive**: it reuses the existing alpha-blend + sampled-texture paths through a **screen-space overlay pass** (`shaders/text.vert.hlsl` + `text.frag.hlsl`, an `alphaBlend` + `cullNone` pipeline). The atlas is a normal sampled texture and the quads are drawn over the scene *after* post-processing, alpha-blended ‚Äî so the HUD composites without disturbing any pre-existing pipeline or golden. Two showcases pin it: `--hud-shot` / `--hud` (standalone text, the `hud` golden) and `--game-hud-shot` / `--game-hud` (the game sample with its score HUD overlaid, the `game_hud` golden).

## Audio Layer (`engine/audio/`)

`engine/audio/` is a **deterministic software audio engine** ‚Äî `mixer.h/.cpp` (the mixer) + `wav.h/.cpp` (a 16-bit PCM WAV writer) ‚Äî that, like `engine/physics` and `engine/math`, is pure C++ above the seam and compiled into `hf_core` (ASan-scoped, unit-tested) and `hf_engine`. It has **no RHI/graphics-backend symbols**.

The mix path is **integer / fixed-point end to end**: `Q15` (signed 1.15 fixed-point) gains, an `int32` accumulator, and a hard clamp to `int16` ‚Äî there is **no `float`/`double` anywhere** in `Render()`'s sample loop. This is deliberate: floating-point mixing can differ in the last bit across compilers (MSVC vs Apple clang), which would break a byte-exact cross-platform WAV golden, whereas the integer path is **bit-identical on every platform and every run**. Voices are sine (integer phase accumulator ‚Üí fixed `int16` wavetable), square (sign of the phase half), or noise (a per-voice integer LCG), shaped by a piecewise-linear `ADSR` envelope whose times are in samples and whose levels are Q15.

The **WAV writer** emits the standard 44-byte canonical PCM header (`RIFF`/`WAVE`/`fmt `/`data`, format 1, 16-bit, little-endian) followed by interleaved samples, serialized **byte by byte by hand** (never `memcpy` of a host-endian struct) and with no timestamps or other non-deterministic fields ‚Äî so the same buffer always encodes to byte-identical bytes. `--audio-render out.wav` renders a fixed deterministic scene (fully headless: no window/GPU) and is pinned by a byte-exact golden, `tests/golden/audio/scene.wav` ‚Äî the **audio WAV golden category** ‚Äî verified on the Windows side (the bytes are identical on both platforms by construction).

## VFX Layer ‚Äî CPU particle / emitter (`engine/vfx/particles.*`)

`engine/vfx/particles.*` (Slice CC) is an **authorable CPU particle / VFX emitter** ‚Äî the VFX analogue of the data-driven material graph, and **distinct** from the fixed `gpu-particles` compute fountain (`shaders/particles.comp.hlsl`, one hard-coded GPU sim). Like `engine/audio` and `engine/physics` it is **pure C++ above the seam** with **zero `vk*`/`MTL*`/`mtl::`/`Backend::` symbols**, compiled into both `hf_core` (ASan-scoped, `vfx_test`) and `hf_engine` (the `--vfx-shot` showcase + the Metal `--vfx` path).

A data-driven `EmitterConfig` describes one emitter as plain values (no handles): `origin`, `spawnRate` (particles/sec, fractional accumulated), `lifetime`, `initVel` + spread, `gravity`, `drag`, a `maxParticles` pool cap (spawns past the cap are dropped, not queued), and **lifetime curves** ‚Äî `size`-over-life and `color`-over-life ‚Äî evaluated by each particle's normalized age. `ParticleSystem::Step(dt)` advances the pool at a **fixed timestep**: it integrates velocity (gravity + drag), retires particles past `lifetime`, and spawns new ones at the configured rate. `BuildBillboards()` turns the live pool into **world-space, camera-facing quad vertices** that the showcase uploads as a dynamic vertex buffer and draws as **additive alpha-blended billboards** ‚Äî reusing the existing dynamic-textured-quad overlay infra (the same pattern the HUD/text overlay and transparency pass use), so it adds **no new RHI seam symbols**.

Determinism is the cross-backend golden contract: a particle emitter normally needs RNG (spawn jitter) and a wall clock (the sim). Both are removed ‚Äî the velocity-spread jitter is drawn from a **fixed-seed integer LCG** (the same `1664525 / 1013904223` constants the audio Noise voice uses ‚Äî no `std::rand` / `<random>`), and the sim runs a **fixed number of fixed-`dt`** `Step()` calls (no clock). So the same `EmitterConfig` + seed + step count yields identical particle state ‚Üí identical billboards ‚Üí byte-identical pixels on both backends, and two runs are bit-identical. Unit-tested (`vfx_test`) and golden-captured (`--vfx-shot` / `--vfx`, the `vfx` golden); the showcase reports a fixed `emitters:1, alive:960, spawned:1800`.

## Networking / Replication Layer (`engine/net/`)

`engine/net/snapshot.*` (Slice BQ) is a **deterministic state-replication core** ‚Äî the networking depth pivot's first slice ‚Äî and, like `engine/audio` and `engine/physics`, it is **pure CPU above the seam** with **zero RHI/graphics-backend symbols** (no `vk*`/`MTL*`/`mtl::`/`Backend::`), compiled into both `hf_core` (ASan-scoped, unit-tested via `replication_test`) and `hf_engine` (the live `--net-shot` showcase). It deliberately ships **no transport**: there are no sockets/UDP/TCP. The authority's serialized byte stream is fed straight into the replica through an in-process "perfect channel", so the focus is the **snapshot/delta wire format and its determinism**, not the network plumbing (a future slice can layer real transport on the same API).

- **Replicated state.** The minimal unit is `RepEntity { id, position, orientation, flags }`; the roll-game's player maps to one entity (its physics body pose) and each uncollected pickup to another, with `kFlagPlayer` / `kFlagPickup` bits so a renderer can tint without a side table. A full `Snapshot` stores its entities in a fixed ascending-`id` order, so the serialized bytes are deterministic.
- **Snapshot + delta.** The authority emits periodic full **keyframes** and, between them, **deltas** that encode only what changed since the last applied snapshot; the replica reconstructs **bit-identical** state by applying the keyframes + deltas in order. The wire format is fixed little-endian with a fixed field order; floats are written as their **raw IEEE-754 bit pattern** (LE `memcpy` of the 4 bytes) ‚Äî because authority and replica are the *same build*, this round-trips a float bit-exactly (a future cross-machine slice can swap in fixed-point quantization behind the same API).
- **Determinism + the showcase.** The authority sim is the deterministic roll-game stepped at the fixed timestep, so the same run yields the same snapshots and the replica reconstructs identical state every time. `--net-shot` steps the authority, streams its bytes into the replica, asserts the replica matches the authority at the capture tick, and **renders the replica's reconstructed scene** through the existing lit/shadowed path (the `net` golden) ‚Äî proving the replicated state is renderable, not just equal. It reports a fixed `replicaMatch:true, savings:43.5%` line (`fullBytes:37233, deltaBytes:21045`), the byte savings of delta replication over sending full snapshots every tick.

### Simulated transport + client interpolation (`engine/net/transport.*`)

Slice BU layers the **hard** part of networking on top of the BQ snapshot core ‚Äî an **imperfect channel** and a **client jitter buffer with interpolation** ‚Äî still **without any real sockets** (real UDP/TCP transport remains a future slice). Like `snapshot.*`, `transport.*` is **pure CPU above `engine/math` + `engine/net`** with **zero RHI/backend symbols**, compiled into both `hf_core` (ASan-scoped, unit-tested via `net_transport_test`) and `hf_engine` (the live `--netsim-shot` showcase).

- **`SimChannel` ‚Äî a deterministic, seeded, socket-free channel.** `Send(tick, bytes)` enqueues an opaque serialized `Snapshot` with `deliverTick = tick + latencyTicks`, except a **seeded loss draw** `< dropRate` drops it (and a **reorder draw** `< reorderRate` perturbs its `deliverTick` by `+[2 .. 2+reorderSpread]` so it genuinely arrives behind the next in-stream packet). All randomness is a **pure integer LCG** (`lcg = lcg*1664525 + 1013904223`, Numerical-Recipes constants) seeded by `lossSeed` and advanced per `Send` ‚Äî **no real RNG, no clock**, so the same `ChannelConfig` + the same `Send` stream yields a bit-identical delivered/dropped/reordered sequence. `Deliver(tick)` returns every queued packet whose `deliverTick <= tick` in delivery order (the queue is a `std::map` keyed by `(deliverTick, seq)`, so ordering is sorted + deterministic with stable `seq` tie-breaks). Counters (`Sent`/`Delivered`/`Dropped`/`Reordered`) feed the showcase stat line + tests.
- **`ClientView` ‚Äî jitter buffer + interpolation.** Delivered packets are deserialized and buffered keyed by tick (`std::map<int, Snapshot>`; out-of-order/duplicate ticks are fine, newest wins, `latestReceivedTick` tracks the max). The client renders at `renderTick = latestReceivedTick ‚àí interpDelay` (default **2 ticks** behind the leading edge) and produces an **interpolated** snapshot via BQ `Interpolate` (lerp position + nlerp/slerp orientation) between the two buffered ticks that **bracket** `renderTick`. Because the render point trails by ‚â•2 ticks, a single **dropped** snapshot is a one-tick hole spanned by its neighbours and interpolated **across** ‚Äî the smoothing hides the drop, no pop. The documented **edge policy** is **hold-last**: at/above the newest buffered tick it holds the newest (never **extrapolates** past the last received snapshot, so it is bounded and never drifts); at/below the oldest it holds the oldest.
- **The showcase.** `--netsim-shot` runs the AX roll-game as the authority over the full scripted track, sends each tick's serialized snapshot through a fixed lossy `SimChannel` into a `ClientView`, and **renders the client's interpolated scene** through the existing lit/shadowed path (the `netsim` golden). It reports a fixed `delivered:226, dropped:42, reordered:29, converged:true` (`converged` asserts the interpolated client tracks the authority despite the lossy channel). Two-run byte-identical (seeded LCG + fixed config).

### Client prediction + server reconciliation (`engine/net/prediction.*`)

Slice BY completes the networking trilogy (`BQ replication ‚Üí BU transport+interp ‚Üí BY prediction+reconcile`). Interpolation (BU) trades **latency** for smoothness ‚Äî it renders the *remote* view a few ticks in the past. Prediction does the opposite for the **local** player: it shows your own input **immediately** instead of waiting a full round-trip. Like `snapshot.*`/`transport.*`, `prediction.*` is **pure CPU above `engine/math` + `engine/net` + the roll-game sim** with **zero RHI/backend symbols** (the seam baseline stays at the 2 `rhi_factory` lines), compiled into both `hf_core` (ASan-scoped, unit-tested via `prediction_test`) and `hf_engine` (the live `--netpredict-shot` showcase).

- **`PredictedClient` ‚Äî predict ahead, then reconcile.** `PredictTick(input)` applies a local `InputCmd` to the client's predicted copy of the sim **right away**, advancing ahead of the last server-acknowledged frame and retaining the input in an ordered buffer. When an authoritative frame arrives, `OnAuthoritative(authState)` **rewinds** the client to that acknowledged state and **replays** every still-unacknowledged buffered input on top of it, so the corrected predicted state re-converges to where the authority will be.
- **`AuthState` carries the full `RigidBody`.** For the rewind/replay to be **bit-exact**, the authoritative frame can't carry only the replicated snapshot pose ‚Äî it carries the player's **full `RigidBody`** (position **and velocity**), so replaying the buffered inputs from the rewind point reproduces the identical trajectory (a velocity-free rewind would diverge on the first integrated step).
- **A real misprediction, then a correction.** The test/showcase is deliberately non-trivial: the authority runs the same sim **plus a scripted server-only impulse** the client has no way to predict, manufacturing a **genuine misprediction** (`maxMisprediction > 0`) that reconciliation then corrects back to the authority's true state. Determinism holds because the scripted inputs + scripted server-only impulse + the seeded BU channel + fixed latency are all bit-stable.
- **The showcase.** `--netpredict-shot` / `--netpredict` drives authority (with the server-only impulse) ‚Üí a delayed BU channel ‚Üí the predicted+reconciled client, and **renders the reconciled client scene** lit + shadowed (the `netpredict` golden). It reports `predTicks:251, reconciles:247, maxMisprediction:0.0655, finalError:0.0250, converged:true` ‚Äî byte-identical cross-backend (shared pure-C++ core), two runs deterministic.

## Editor Layer (`engine/editor/`)

Beyond the deterministic logic modules (`picking`, `gizmo`, `introspect`), the editor layer hosts the **docked ImGui editor shell** (Slice BT): a Scene Hierarchy / Inspector / Stats UI laid out around a central scene Viewport.

- **The pure-CPU panel-data seam (`editor/editor_panel_data.*`).** The architectural move here is the same one used throughout the engine: a **deterministic, backend-free, ImGui-free seam** between *what each panel displays* and *the Dear ImGui calls that draw it*. `BuildPanelData(registry, resources, state)` computes the three panels' data ‚Äî `HierarchyRow`s (one per drawable entity, label = mesh name + view-order index), an `InspectorData` view of the selected entity (`TransformC`/`MaterialC`/`MeshC`), and `StatsData` counts ‚Äî touching **only** the ECS `Registry`, opaque `SceneResources` named pointers (never dereferenced ‚Äî the same contract as `introspect`/`scene_io`), and `math`. It carries **zero** `vk*`/`MTL*`/`rhi`/ImGui symbols, so it lives in `hf_core` and is unit-tested **headlessly** (`editor_panels_test`) by asserting the data, not pixels. The persistent `EditorState.selectedEntity` is **clamped into range and written back** each call (matching the live `--fly` behaviour), so two calls on an identical registry/state yield byte-identical `PanelData`.
- **Deterministic, code-driven docked layout.** The vendored Dear ImGui is **1.91.8, the non-docking master branch ‚Äî there is no `DockBuilder` API**. So the docked layout is **not** loaded from a machine-dependent `imgui.ini` (`io.IniFilename = nullptr`); instead `editor::DockLayout` describes it as **fixed split ratios + fixed dock-node ids** (`leftRatio=0.22`, `rightRatio=0.28`, `leftBottomRatio=0.45`, with the Viewport in the central node, Hierarchy + Stats in the left column, Inspector on the right). Because the layout is a deterministic **value** and ImGui geometry is CPU-built, the panel geometry is **byte-stable run-to-run and identical across backends** ‚Äî which is exactly what makes a golden image of an ImGui UI possible. `BuildEditorUI` (`editor_panels.*`) issues the ImGui widgets from the `PanelData`; the RHI-only `ImGuiRenderer` (`imgui_renderer.*`) draws ImGui's vertex/index/command buffers through the seam (no backend symbols), so the **same** editor chrome renders on Vulkan (`--editor-shot`) and Metal (`--editor`).
- **The showcase.** `--editor-shot [objIndex]` / `--editor` renders the default Slice-F scene with the docked editor chrome over it and a FIXED selected entity, captures (the `editor` golden), and prints a deterministic, two-run byte-identical `editor: {selected:9, entities:10}` stat line.

### Live edit-ops ‚Üí scene_io round-trip (`engine/editor/edit_ops.*`)

Slice BX closes the editor's **author ‚Üí save ‚Üí reload** loop. An `EditOp` (e.g. translate, set-material) **mutates the live ECS `Registry` in place**, then the edited scene is **serialized through the existing `scene_io` writer** and **reloaded** into a fresh registry, and the reloaded state is asserted **bit-identical** to the in-memory edited state ‚Äî proving an edit *survives a save/load round-trip exactly* (no silent loss or reordering across serialization). It follows the same backend-free seam discipline as `editor_panel_data`/`introspect`: it touches **only** the ECS `Registry`, the opaque `scene_io` text format, and `math`, carrying **zero** `vk*`/`MTL*`/`rhi`/ImGui symbols, so it lives in `hf_core` and is unit-tested **headlessly** (`edit_ops_test`) by asserting the registry/round-trip equality, not pixels. The deterministic edit script makes the result two-run byte-identical. Golden-captured (`--editor-edit-shot` / `--editor-edit`, the `editor_edit` golden); the showcase applies a fixed edit script and reports `editor-edit: {edits:2, entity:8, saved:true, reloadMatch:true}`.

## The validation-clean invariant

As of the render-graph barrier work, **Vulkan-validation-cleanliness is a permanent gate**, not a one-off check. Every `--*-shot` showcase is run under the **Khronos validation layer** with both **core** and **synchronization** validation enabled (`VK_LAYER_KHRONOS_validation`, `VK_LAYER_PATH` pointed at the layer's package bin dir, sync validation switched on). The bar is **zero** `VUID-*`, `SYNC-HAZARD-*`, `UNASSIGNED-*`, or `[ERROR]` lines across all showcases. The synchronization-validation half is specifically the oracle for the render-graph barrier solver: if the solver ever placed a missing or wrong barrier, sync validation would flag a hazard. The only tolerated diagnostics are benign `[WARNING: Performance]` notices ‚Äî notably the depth-only shadow pipelines binding a full vertex layout produce "vertex attribute at location N not consumed by vertex shader" ‚Äî which are expected and documented, not errors.

The validation layer is **not** a link-time dependency; it is pulled in via Conan (`vulkan-validationlayers` in `conanfile.py`, debug builds) purely to ship `VkLayer_khronos_validation.{dll,json}` so the gate can actually load the layer on a box where it is not installed system-wide. Without that dependency the oracle would be silently inactive and a validation regression could slip through unnoticed.
