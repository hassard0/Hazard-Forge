# Hazard Forge — Architecture

## The RHI Seam

The central architectural decision in Hazard Forge is the **Rendering Hardware Interface seam** — a layer of pure C++ abstract interfaces in `engine/rhi/rhi.h` that separates all engine and application code from any GPU backend.

### The hard rule

> No `vk*` type, no `MTL*` type, and no backend header may appear in any above-seam module — `engine/rhi/` (the seam itself), `engine/scene/`, `engine/math/`, `engine/render/`, `engine/material/`, `engine/ui/`, `engine/audio/`, `engine/game/`, or the rest. Backend symbols are confined to `engine/rhi_vulkan/` and `engine/rhi_metal/` respectively. (`engine/hal/` is the platform layer: it touches SDL's `SDL_Metal_*` to vend an opaque `CAMetalLayer*` as a `void*`, but pulls in no Metal/Obj-C GPU types.)

The rule is checkable with a single `grep` for `vk[A-Z]|MTL|Metal` over the above-seam dirs: every match is a comment, prose, or a substring false-positive (`kMetallic`, the `Backend::Metal` enumerator name). The only genuine backend-symbol CODE above the seam is the **two-line `rhi_factory` dispatch** (the `CreateMetalDevice` forward-declaration + the `Backend::Metal` → `mtl::CreateMetalDevice` call) — the single sanctioned place the factory names a backend. This baseline is re-checked each consolidation pass and enforced by convention in every slice plan.

### The seam surface (`engine/rhi/rhi.h`)

All resource types are opaque base classes:

```
IRHIDevice        — device factory; owns swapchain, frame loop, descriptor infrastructure
ISwapchain        — surface + images; resize
ICommandBuffer    — per-frame draw recording (BeginRenderPass, BindPipeline, Draw, ...)
IPipeline         — compiled graphics PSO from GraphicsPipelineDesc
IBuffer           — vertex / index / uniform buffer
ITexture          — device-local image + sampler descriptor (base for render targets too)
IRenderTarget     — sampleable offscreen color (+depth) image; inherits ITexture
```

`GraphicsPipelineDesc` is a small, flat descriptor that encodes what the pipeline needs to know about shaders, vertex layout, depth, push constants, descriptor set membership, and a set of mode flags that has grown additively slice by slice — `fullscreen`, `depthOnly`, `pbrMaterial`, `usesEnvironment`, `usesJointPalette`, `depthWrite`, `alphaBlend`, `cullNone`, `lineList`, `fragmentPushConstants`, `usesLightClusters` — all in terms of engine-defined enums and types, no backend types. Every new flag **defaults to the value that leaves all pre-existing pipelines byte-for-byte unchanged**, which is what keeps the committed goldens stable as features land.

There is also a compute path (`IComputePipeline` + `Dispatch`) used by the GPU particle system and the **GPU-driven culling** pass, on the same seam. Newer per-frame binding entry points have been added additively to the command interface — e.g. `BindLightClusters` (clustered/Forward+ light data) and `BindReflectionProbe(ITexture& probeAtlas)` (the baked reflection/irradiance probe atlas) — each with a default no-op body so existing pipelines and goldens are unaffected.

Three later seam additions follow the same defaulted-no-op discipline, so backends without the path still link and pre-existing goldens are byte-for-byte unchanged:

- **Indirect draw** — `DrawIndexedIndirect(IBuffer& args, ...)` consumes a `VkDrawIndexedIndirectCommand` / `MTLDrawIndexedPrimitivesIndirectArguments` produced on the GPU by the cull compute pass (frustum-cull + ordered compaction), so the survivor draw never round-trips to the CPU. The `vk*`/`MTL*` indirect calls live only in the backend dirs.
- **Resource-state transitions** — `TransitionResource(state prev → next)` lets the render-graph barrier solver request a transition declaratively. On Vulkan it lowers to a `vkCmdPipelineBarrier2` (`VkImageMemoryBarrier2` with the matching layout + stage/access masks); Metal's tracked-hazard model makes it a no-op.
- **Secondary / parallel recording** — passes record into per-thread secondary command buffers that are replayed in deterministic creation order (`vkCmdExecuteCommands` on Vulkan; an `MTLParallelRenderCommandEncoder`'s sub-encoders on Metal). This is what makes a 1-worker render and an N-worker render byte-identical.
- **OIT revealage blend (Slice CO)** — a single additive **pure-interface** field, `GraphicsPipelineDesc::oitRevealageBlend` (a `bool`, default `false`), in `engine/rhi/rhi.h`. It is the only knob the Weighted-Blended-OIT resolve needs from the seam: when set, the pipeline's revealage target uses the multiplicative `(0, 1−srcAlpha)` blend (vs. the default additive accum blend). It carries **no backend symbol above the seam** — the actual `VkPipelineColorBlendAttachmentState` / `MTLRenderPipelineColorAttachmentDescriptor` blend factors are set **only inside the backend dirs** (`engine/rhi_vulkan/vulkan_pipeline.cpp`, `engine/rhi_metal/metal_pipeline.mm`), and `false` reproduces every pre-CO pipeline byte-for-byte, so existing goldens are unchanged.
- **Compute barriers (Slice CS)** — two additive **pure-interface** entry points, `ICommandBuffer::ComputeToComputeBarrier()` / `ComputeToFragmentBarrier()` (defaulted empty bodies on the base interface), express the froxel inject→integrate→apply hazards. The real `vkCmdPipelineBarrier2` / Metal tracked-hazard barriers are recorded **only inside the backend command buffers**; the COMPUTE→COMPUTE→FRAGMENT chain is SYNC-HAZARD-free under the Khronos sync-validation layer.
- **Compute-stage shadow-map sampling (Slice CX)** — one additive **pure-interface** entry point `ICommandBuffer::BindShadowMapCompute(IRenderTarget&)` plus one additive field `ComputePipelineDesc::sampledShadowMap` in `engine/rhi/rhi.h`, the only seam knobs the volumetric-shadows inject pass needs to read the sun's CSM atlas from a **compute** shader. Both are pure interface (defaulted/empty on the base, no backend symbol above the seam — the real binding lives only in `engine/rhi_vulkan/` + `engine/rhi_metal/`), so the shadow-pass→inject-compute read barrier is SYNC-HAZARD-clean and pre-CX goldens are byte-for-byte unchanged. The rhi_factory dispatch count is still 2.
- **Cubemap render targets (Slice DD)** — the **cubemap render-target capability** the dynamic cubemap-capture probe needs, a set of additive **pure-interface** entry points in `engine/rhi/rhi.h`: a new opaque base `ICubemapTarget : public ITexture` (a cube render target that is *both* a 6-attachment render destination and a sampled hardware `TextureCube`), `IRHIDevice::CreateCubemapTarget(size, format)`, `BeginCubemapFace(ICubemapTarget&, face)` / `EndCubemapFace(...)` to render each of the 6 faces, the read-back pair `ReadCubemapFace(...)` / `ReadRenderTarget(...)` (the capture-correctness proof), and `ICommandBuffer::BindCubemapProbe(ITexture&)` to bind a captured cube as the reflection source. Every method has a **defaulted body** on the base interface (so backends without it still link and pre-DD goldens are byte-for-byte unchanged); the real per-face attachment + cube-sampling implementation lives **only** in the backend dirs (`engine/rhi_vulkan/vulkan_cubemap_target.*`, `engine/rhi_metal/metal_cubemap_target.*`). The render graph emits the **capture→sample barrier** (the 6 face passes then a ShaderRead transition), proven SYNC-HAZARD-free under the Khronos sync-validation layer. The rhi_factory dispatch count is still 2. **Slices DE (planar reflections) and DF (CAS) added no RHI** — DE reuses the existing `cullNone` knob for the flipped reflection winding; DF is a fullscreen post pass on the existing path.

Backends are handed out through the factory:

```cpp
// Vulkan (Windows):
auto device = hf::rhi::CreateDevice(hf::rhi::Backend::Vulkan, window);

// Metal headless (macOS, no window):
auto device = hf::rhi::mtl::CreateMetalDeviceHeadless(width, height);

// Metal windowed (macOS): built directly from a CAMetalLayer* (passed as void*), so a windowed,
// presenting Metal device can be constructed WITHOUT pulling in hal/window.h or SDL — used by the
// SDL-free native Cocoa entry (mac_window/) that drives the interactive --fly viewport on macOS.
MetalDevice dev(caMetalLayer /* void* */, width, height);
```

`MetalDevice` therefore has three constructors — windowed-from-`Window&` (SDL HAL), windowed-from-`CAMetalLayer*` (native Cocoa, SDL-free), and headless-offscreen — all behind the same `IRHIDevice` seam.

Both return `std::unique_ptr<IRHIDevice>`. The caller never sees a backend type again.

## Descriptor / Binding Model

Hazard Forge uses a **frequency-based descriptor set layout** adopted in Slice D (lighting):

| Set | Update frequency | Contents |
|-----|-----------------|----------|
| 0 | Per frame | UBO: `viewProj`, directional light, view position, point lights, `lightViewProj`, camera basis for sky; shadow map depth image (binding 1) + shadow sampler (binding 2) |
| 1 | Per material / per draw | Material textures + samplers. The base set is color + normal; the **PBR** path (`BindMaterialPBR`) widens it to the full 5-texture metallic-roughness set (base / metal-rough / normal / emissive / occlusion). The **environment** (HDR IBL) binds on a dedicated slot via `BindEnvironment` so the base material layouts are unchanged. |
| 2 | Per skinned draw | Joint palette (skinning), bound via `SetJointPalette` at a dedicated vertex binding. |
| 3 | Per clustered-lit frame | The clustered/Forward+ light data (Slice AG): the per-cluster `Cluster` array (offset/count) at binding 13, the flat light-index list at binding 14, and the `GpuLight` array at binding 15. Bound via `BindLightClusters` when a pipeline sets `usesLightClusters = true`; otherwise the set is absent and pre-existing pipelines are byte-for-byte unchanged. |

GPU instancing adds a **second per-instance vertex stream** (binding 1, step-per-instance) rather than a descriptor set — four `RGBA32_Float` attributes carry a column-major `float4x4` per instance.

The set-3 cluster set follows the **same dedicated-extra-set pattern** established by the environment (HDR IBL) set and the joint palette: a feature that only some pipelines need binds on its own slot through a single `Bind*` call, so the base material/frame layouts — and therefore every existing golden — are untouched. `BindLightClusters` is the first case of **graphics-stage storage buffers** on the seam: the three set-3 bindings are `STORAGE_BUFFER`s read by the fragment shader (`lit_clustered.frag`), not the uniform buffers used by set 0/1. On Vulkan they are SSBO descriptors; on Metal they map to fragment `[[buffer(13/14/15)]]` via `spirv-cross --msl-decoration-binding` (`engine/rhi_metal/metal_common.h`: `kFragClusterBuf=13`, etc.).

**Set 0** is owned by the device. The device maintains one UBO per frame-in-flight (double-buffered), each with a pre-baked descriptor set that already points at its buffer. `SetFrameUniforms(data, size)` memcpys into the current frame's UBO; `BindPipeline` auto-binds set 0 when the pipeline has `usesFrameUniforms = true`. The CPU never writes a UBO that the GPU is still reading.

**Set 1** is owned per texture / render target. Each `VulkanTexture` and each `VulkanRenderTarget` allocates one descriptor set from a device pool at creation time and updates it with its image view and the device's default sampler. `BindTexture(ITexture&)` binds that set.

**Push constants** carry the per-object model matrix (64 bytes, `mat4`). `PushConstants(data, size)` records a push to the vertex stage.

When `usesFrameUniforms = false` (the post-processing pipeline), the material set is the only set and occupies set index 0. The `BindTexture` path accounts for this shift.

## Per-frame UBO growth

The `FrameData` struct (352 bytes at the time of the skybox slice; `kFrameUboSize` was 512) has grown additively across slices:

- Slice D: `viewProj`, `lightDir`, `lightColor`, `viewPos` — basic per-frame camera + directional light
- Slice H: `ptCount`, `ptPos[3]`, `ptColor[3]` — colored point lights
- Slice I: `lightViewProj` — directional shadow map light-space matrix
- Slice J: `camFwd`, `camRight`, `camUp`, `skyParams` — camera basis for sky ray reconstruction
- Slices AD–AF: the cascaded / spot / point shadow matrices, per-cascade split distances, and the spot/point light parameters — the shadow set's per-frame data. This is what pushed the struct past 512 B, so **`kFrameUboSize` was bumped 512 → 1024** (`engine/rhi_vulkan/vulkan_device.cpp` and `engine/rhi_metal/metal_device.h`, kept in sync); both backends now allocate a 1024-byte per-frame UBO, and `SetFrameUniforms` asserts the upload fits.

The `kFrameUboSize` constant is bumped whenever the struct approaches the current allocation. The C++ `FrameData` struct (mirrored byte-for-byte in `metal_headless/visual_test.mm`) and the HLSL `cbuffer FrameData` in every shader that reads it must be kept in sync by hand — the Metal struct layout is asserted to match the Vulkan sample's. Pass-specific parameters that are NOT per-frame (bloom thresholds, SSAO kernel params, per-object material factors, glass tint/alpha) travel via **push constants**, not the UBO — the vertex push constant carries `mat4 model + float4 material`, and the bloom/SSAO fullscreen passes use the `fragmentPushConstants` flag to read their params in the fragment stage.

### `skyParams` channels — and the per-frame TIME channel (issue #5)

`skyParams` is a single `float4` with **per-pass conventions** (only ever read by the pass whose convention it is, so they never collide):

- The sky / lit / animated-sky path reads `skyParams.x = tan(0.5·fovY)`, `skyParams.y = aspect` (the view-ray reconstruction), and now **`skyParams.z = time` (seconds), `skyParams.w = frameIndex`** — the per-frame **time channel** an animated shader reads. Before issue #5 there was nowhere in the per-frame UBO to read time, so animated sky / water / foliage / VFX shaders were stuck static or had to smuggle time through a per-draw push constant. To animate, a shader just reads:

  ```hlsl
  float time = f.skyParams.z;   // seconds since the sample started
  ```

  The CPU fills it from a `runtime::FixedTimestep` (the deterministic fixed-step clock in `engine/runtime/clock.h`): accumulated time = `steps · Step()`. **Golden-invariance:** every existing showcase fills `skyParams.z = 0` (the `FrameData fd{}` zero-init already gives this), so its render is byte-identical to before; only a sample that *wants* motion advances the clock. The worked example is `shaders/sky_animated.frag.hlsl` + the `--sky-animated-shot` (Vulkan) / `--sky-animated` (Metal) showcase, captured at a fixed `2.0 s` for a reproducible golden.
- Other passes that do NOT render the procedural sky repurpose `skyParams.zw` for their own pass-local data (e.g. `planar.frag` and `lit_contactshadow.frag` use `.zw = (1/width, 1/height)`; `lit_pbr_ibl.frag` uses `.z = env maxLod`; the reflection probes pack a box into `.xyz/.w`). These passes never read the time convention, so there is no conflict.

## Multi-Pass Frame Structure

Frames are described by `render::RenderGraph`: passes declare imported targets (shadow map / scene color / swapchain) as reads/writes and are executed in declaration order through the RHI. The baseline lit frame is three passes; feature showcases add more passes to the same graph (a g-buffer prepass + AO + blur + composite for SSAO; a threshold + 5× down + up + composite chain for bloom into an `RGBA16_Float` target; a sorted alpha-blended pass after the opaque pass for transparency; a debug-line draw after opaque geometry). The capstone scene composes **seven distinct opaque pipelines plus a transparent pipeline** into one scene pass.

The baseline three-pass frame:

```
Pass 0: Shadow pass (depth-only)
  BeginShadowPass(*shadowMap)
    → depth-only command buffer, 2048² depth image, no color attachment
    → drawDepthOnly: shadow pipeline (depthOnly=true), PushConstants(model), DrawIndexed
  EndShadowPass → submit + fence-wait → shadow map transitions to SHADER_READ_ONLY

Pass 1: Scene → offscreen render target
  BeginRenderTargetFrame(*rt)
    → command buffer targeting the RT's color+depth
    → sky pipeline (fullscreen, depthTest=false) → Draw(3)   [background fill]
    → lit pipeline: per-renderable [PushConstants, BindTexture, BindVB/IB, DrawIndexed]
  EndRenderTargetFrame → submit + fence-wait → RT color transitions to SHADER_READ_ONLY

Pass 2: Post pass → swapchain
  BeginFrame → acquires swapchain image
    → post pipeline (fullscreen, usesTexture=true, usesFrameUniforms=false)
    → BindTexture(*rt) → Draw(3)
  EndFrame → submit + present (or capture + BMP in --shot mode)
```

Passes 0 and 1 use dedicated command buffers (`rtCmd_`, `rtPool_`, `rtFence_`) to avoid contention with the per-frame swapchain recorder. Both are submitted synchronously (fence-wait) before the swapchain pass begins, so there are no semaphore dependencies to track.

### Render-graph resource-state tracker + barrier solver

`render::RenderGraph` no longer relies on each pass to hand-place its own barriers. Every imported resource carries a current **state** (e.g. `RenderTarget`, `ShaderRead`, `TransferSrc`, `Present`); each pass declares the state it needs each resource in (read vs write). Before a pass runs, the **barrier solver** diffs each resource's prior state against the pass's required state and, when they differ, emits exactly one transition via `ICommandBuffer::TransitionResource(prev → next)`, then advances the tracked state. On Vulkan that transition becomes an explicit `vkCmdPipelineBarrier2`; on Metal the tracked-hazard model makes it a no-op. The solver is pure host logic over `rhi/` types (no backend symbols) and is unit-tested (`render_graph_test`). Crucially the auto-inserted barriers are **proven hazard-free by the Khronos synchronization-validation layer**: running every showcase under sync validation yields zero `SYNC-HAZARD-*` lines, which is the oracle that the solver places correct, sufficient, and non-redundant barriers.

### Parallel command recording (worker pool)

Within a pass, draws can be recorded **across threads** without changing the output a single bit. A worker pool fans the pass's draw list out to N per-thread secondary command buffers; each thread records its slice into its own secondary, and the pass replays the secondaries **in deterministic creation order** (`vkCmdExecuteCommands` with the array in order on Vulkan; closing an `MTLParallelRenderCommandEncoder`, which commits its sub-encoders in creation order, on Metal). Because the replay order is fixed and independent of which thread finished first, a `--mt --workers 1` render and a `--mt --workers 4` render produce a **byte-identical** image (the `mt` golden + a 1-vs-N hash test, `parallel_record_test`, both enforce this). The ordering guarantee — not thread count — is what preserves determinism.

### Screen-space projected decals (`engine/render/decal.h`)

Screen-space decals (Slice BH) project a texture onto whatever scene geometry lies inside an oriented box volume, **reusing the G-buffer** rather than re-rasterizing geometry. A decal is a box: `BuildDecalTransform` builds its local→world `TRS = Translate(center) · R(eulerXYZ) · Scale(2·halfExtents)` so the local **unit cube** `[-0.5,0.5]³` maps onto the world box, and its **inverse** (world→decal) is uploaded to the shader. In the decal composite pass the deferred fragment's world position is reconstructed from depth **exactly as SSR/SSAO reconstruct view space** (`ReconstructViewPos` + the same yFlip convention), then mapped view→world via the camera `invView`; that world point is transformed into decal-local space, and if it lies `InsideUnitBox` the decal texture is sampled at the top-down `DecalUV` (`local.xz + 0.5`, the decal projects along local `-Y`) and alpha-blended over the lit scene with a smoothstep `EdgeFade` near the box faces.

The box-projection math (`BuildDecalTransform` / `WorldToDecalLocal` / `InsideUnitBox` / `DecalUV` / `EdgeFade`) is **header-only, pure CPU, zero backend symbols**, and is **shared** by both the in-shader composite (`shaders/decal.frag.hlsl`) and the unit test (`decal_test`) — exactly the `render/ssr.h` pattern — so the test pins the same math the GPU runs and the CPU side and shader stay mutually consistent. Golden-captured (`--decal-shot` / `--decal`, the `decal` golden).

### Screen-space global illumination — raw / bilateral denoise / temporal (`engine/render/ssgi.h`)

Screen-space global illumination (Slice BP) adds **one bounce of indirect diffuse lighting in screen space**, and the depth pivot here is *reuse*: `engine/render/ssgi.h` is header-only, pure CPU (no device, no backend symbols) and **builds directly on the SSR machinery** rather than duplicating it. It `#include`s `render/ssr.h` and **re-exports** `ssr::ReconstructViewPos` / `ssr::ViewToScreenUV` (`using ssr::…`), so the SSGI math reconstructs the view-space position + normal from the G-buffer with the **same** functions SSR's reflection ray uses (RH view space, `-Z` forward, view-space linear depth `= -vpos.z`, the `yFlip` sign `-1` on Vulkan / `+1` on Metal). For each pixel it traces `K` cosine-weighted hemisphere rays about `N`, marches each in view space against the G-buffer using the **same march + binary-search refinement as `ssr.frag`**, and on a hit samples the **already-lit HDR scene color** as that ray's incoming radiance; the accumulated mean of the `K` hit radiances is the indirect-diffuse irradiance, which the composite multiplies by albedo and **adds** to the scene. The in-shader march (`shaders/ssgi.frag.hlsl`) copies `ssr.frag`'s `ReconstructViewPos`/`ProjectToUV` verbatim, so the CPU model and the shader stay bit-consistent. `SsgiParams` (`K=16`, `marchDist`, `thickness`, `intensity`) mirrors `SsrParams`. Golden-captured (`--ssgi-shot` / `--ssgi`, the `ssgi` golden); unit-tested (`ssgi_test`) against the shared kernel.

The single-frame `K=16` gather is **noisy**, so Slice BR adds an **edge-preserving bilateral denoise** of the indirect buffer (`render/ssgi.h`: `BilateralWeight` + `SsgiDenoiseParams`). `BilateralWeight` is a single pure function returning a neighbor tap's weight relative to the center as the product of a **spatial** Gaussian (pixel-distance²/`spatialSigma`), a **range** term (indirect-color difference), and a **normal** term (G-buffer normal agreement) — so the blur smooths noise *within* a surface but does not bleed indirect light across depth/normal edges. The denoise pass outputs the edge-aware weighted mean over a radius-2 (5×5) window (`sum/wsum`), and the **same math is mirrored verbatim in `shaders/ssgi_denoise.frag.hlsl`**. It is deterministic (no time/RNG) — two runs are byte-identical — and golden-captured (`--ssgi-denoise-shot` / `--ssgi-denoise`, the `ssgi_denoise` golden). Both SSGI passes carry **zero backend symbols** and follow the established `render/ssr.h` "shared CPU math == shader math" discipline.

The spatial denoise attacks the noise *within* a frame; Slice BV adds the orthogonal axis — **temporal accumulation** — completing the GI trilogy `raw → spatial-denoise → temporal`. The `--ssgi-temporal-shot` path renders the **same** Cornell scene/camera/gather but runs a **fixed `N=8`** loop in which each iteration performs a full SSGI gather with a **different golden-angle-rotated hemisphere kernel** (the `ssgi.frag` `frame` param drives the per-frame rotation) and accumulates a **running mean** of the per-frame indirect buffers into a ping-pong HDR accumulation target before the composite multiplies by albedo and adds to the scene. Averaging eight decorrelated (golden-angle-jittered) gathers converges to a much cleaner indirect-diffuse result than the single `K=16` frame. The accumulation count `N=8` matches `taa::kAccumFrames`; because the showcase camera is **static**, the accumulation needs **no reprojection** (history-reprojection under camera motion is a future slice — this is the fixed-camera, no-reprojection running mean). Deterministic two-run byte-identical (no time/RNG/clock — the jitter is a fixed golden-angle sequence), golden-captured (`--ssgi-temporal-shot` / `--ssgi-temporal`, the `ssgi_temporal` golden), carrying **zero backend symbols**.

### GPU multi-draw-indirect batching (`engine/render/mdi.h`)

GPU multi-draw-indirect (Slice BM) collapses **N distinct objects into a single indirect draw call**. `render::mdi::BuildBatch` is a header-only, pure-CPU builder (no device, no backend symbols) that, from a list of `DrawObject`s, lays out the two GPU buffers one `DrawIndexedMultiIndirect(drawCount=N)` consumes:

1. An **indirect-args** buffer of N `MdiCommand` records. The record is the standard 5×u32 `{indexCount, instanceCount, firstIndex, vertexOffset, firstInstance}` — byte-identical (and `static_assert`-pinned at 20 bytes) to `VkDrawIndexedIndirectCommand` and `MTLDrawIndexedPrimitivesIndirectArguments`, so the packed array uploads straight into a `BufferUsage::Indirect` buffer with `stride = sizeof(MdiCommand)`. `instanceCount` is always 1 and `firstInstance` always 0.
2. A **per-draw storage** buffer of N `PerDraw` records (`{ float model[16]; float material[4]; }`, std430-friendly, 80 bytes). The MDI vertex shader (`shaders/lit_mdi.vert.hlsl`) reads `PerDraw[gl_DrawID]` — the SPIR-V `DrawIndex` built-in, **not** `firstInstance` — so command i and `PerDraw[i]` describe the same object and the `firstInstance` field stays free.

The seam gains two defaulted methods on `ICommandBuffer`: **`DrawIndexedMultiIndirect(argsBuffer, drawCount, stride)`** and **`BindPerDrawData(perDraw)`**. On **Vulkan** these lower to a true `vkCmdDrawIndexedIndirect(argsBuffer, 0, drawCount, stride)` plus binding the `gl_DrawID`-indexed per-draw SSBO; both default to no-ops so passes/backends without MDI are unaffected, and the **Metal golden renders the identical scene per-object** (the per-object reference path). Render-invariance is structural: because the per-object reference path pushes the *same* `PerDraw[i]` values and issues the *same* index slices, the one-MDI-call render is mathematically identical to N per-object draws, and the single builder is the sole source of truth for both arrays. The builder is deterministic (a plain ordered memcpy into two POD arrays), unit-tested (`mdi_test`), and golden-captured (`--mdi-shot` / `--mdi`, the `mdi` golden); the showcase reports `drawCalls:1, refDrawCalls:144` and `mdi==per-draw: BYTE-IDENTICAL`.

### Bindless textures (`engine/render/bindless.* + engine/rhi_vulkan/vulkan_bindless.*`)

Bindless textures (Slice BZ) are the texturing analogue of MDI and complete the GPU-driven story: instead of rebinding a per-material descriptor set before each draw, **all** scene textures are bound **once** as a single large descriptor **array** and selected per-draw by an integer `texIndex`. The above-seam half is `render/bindless.h` — a header-only, pure-CPU **texture-index table / interner** (`BuildBindlessTable`): it assigns each distinct texture a **stable first-insertion index**, de-duplicates repeats, and is the **single source of truth** for which array slot every draw samples. It carries **zero backend symbols** and is unit-tested (`bindless_test`: interning, material-coverage, determinism, ASan-clean).

The descriptor-indexing implementation lives **only below the seam** in `engine/rhi_vulkan/vulkan_bindless.*`. The seam gains an additive, pure-interface trio: **`CreateBindlessTextureSet(span<ITexture*>)`** → an **`IBindlessTextureSet`**, **`BindBindlessTextures(set)`**, and a `usesBindlessTextures` query. On **Vulkan** this is **true bindless**: a runtime sampled-image array (set 4) created with `PARTIALLY_BOUND` + `VARIABLE_DESCRIPTOR_COUNT` + `UPDATE_AFTER_BIND` (the `descriptorIndexing` device features enabled), bound once, with a `texIndex` push constant per draw; `shaders/lit_bindless.frag.hlsl` samples `gTextures[NonUniformResourceIndex(texIndex)]` (DXC emits `SPV_EXT_descriptor_indexing`). Both seam methods **default to no-ops**, so **Metal** inherits the bound path and renders the **identical** multi-texture scene per-material (image backend-identical) — the Vulkan-true / Metal-bound-golden split, exactly as MDI is Vulkan-true / Metal-per-object. `lit_bindless.*` is Vulkan-only and intentionally **not** in the Metal MSL generation list.

Render-invariance is structural: `--bindless-shot` renders the bindless-array path **and** the per-material **bound** reference (same texture, same `LINEAR`/`REPEAT` sampler) and asserts the two captures are **byte-identical** (FNV), so the single index table feeds both paths. The showcase reports `textures:5, draws:100, textureBinds:1, refTextureBinds:5` and `bindless==bound: BYTE-IDENTICAL`, and is golden-captured (`--bindless-shot` / `--bindless`, the `bindless` golden).

### The GPU-driven pipeline — combined pass + culled pass (`engine/render/gpu_driven.h` + `engine/render/gpu_culled.h`)

MDI (BM) and bindless (BZ) are the two halves of a modern GPU-driven renderer; Slices CB and CD **compose** them — and then add the GPU as the culler — into the complete `cull → compact → MDI + bindless` pipeline. Both builders are **header-only, pure-CPU, zero backend symbols**, and are each the **single source of truth** that feeds both the GPU-driven path and its per-object reference, so the two paths can never diverge.

**Fully-GPU-driven combined pass (Slice CB, `render/gpu_driven.h`).** `render::gpudriven::BuildBatch` merges the BM and BZ outputs into one record set. The combined per-draw record `GpuDrivenPerDraw` extends `render::mdi::PerDraw` (model at offset 0, `float4` material at 64) with a `texIndex` at 80, padded to **96 bytes** (std430-friendly); the fully-GPU-driven vertex shader (`shaders/lit_gpudriven.vert.hlsl`) reads `PerDraw[gl_DrawID].{model,material}` and passes `texIndex` **flat** to the fragment, which samples `gTextures[NonUniformResourceIndex(texIndex)]` (`shaders/lit_gpudriven.frag.hlsl`). So **one** `DrawIndexedMultiIndirect(N)` (the BM indirect-args buffer) **plus one** `BindBindlessTextures` (the BZ array, bound once) draw an entire **multi-material** scene — the union of the two earlier capstones. The builder is the sole producer of the commands, the per-draw records, **and** the bindless table; `texIndex` is assigned by first-insertion interning (`render::bindless::Intern`), and the `ITexture*` is treated as an **opaque interning key** (never dereferenced), so the whole builder is unit-testable with no GPU (`gpu_driven_test`). On **Vulkan** the showcase issues one true `vkCmdDrawIndexedIndirect` + one descriptor-indexing array bind; **Metal** inherits the per-object **bound** path and renders the identical scene. Render-invariance is structural — the per-object reference pushes the same model + material and binds the same texture each object's `texIndex` selects — so `--gpudriven-shot` reports `drawCalls:1, textureBinds:1` (vs `refDrawCalls:100, refTextureBinds:101`) and `gpudriven==bound: BYTE-IDENTICAL`, golden-captured (`--gpudriven-shot` / `--gpudriven`, the `gpudriven` golden).

**Fully-GPU-driven-CULLED pass (Slice CD, `render/gpu_culled.h`).** This is the capstone: instead of laying out *all* objects unconditionally (CB), a **compute** pass frustum-culls the instances and **compacts the survivors** itself, then writes the **GPU-decided MDI `drawCount`** that the single indirect draw consumes — the GPU both *decides which objects survive and draws exactly them*. `render::gpuculled::Cull` is the authoritative CPU **mirror** of `shaders/gpudriven_cull.comp.hlsl`, composing four proven slices: it runs the **identical** conservative `render::frustum::SphereOutside` test (AR) on the **identical** world bounding sphere `render::gpu_cull::InstanceWorldSphere` computes (AR), emits the survivors **in source-index order** via the **ordered single-workgroup prefix-sum compaction** (AR — *not* an unordered `atomicAdd` append; the source-order scan is the determinism trick that makes the golden reproducible), and writes each survivor as a `render::gpudriven::GpuDrivenPerDraw` (CB) whose count is the MDI `drawCount` (BM) and whose `texIndex` selects its bindless slot (BZ). The compute pass writes the compacted survivors into the per-draw SSBO + the indirect command, then **one** `DrawIndexedMultiIndirect(drawCount = survivor count)` + **one** bindless bind render exactly them. Render-invariance is structural: the showcase also renders the same survivor set per-object **bound** (CPU frustum-cull → draw only the K survivors, each texture bound) and asserts byte-identity; because the survivor set (the conservative sphere test) and the per-draw values are pure functions of scene + frustum and the cull math is shared with the validated AR reference, the images cannot diverge. `--gpucull-draw-shot` reports `total:144, drawn:107, cpuRef:107, drawCalls:1, textureBinds:1`, a matching `gpucull-draw-hash == ref-hash`, and `gpucull-draw==bound: BYTE-IDENTICAL`, golden-captured (`--gpucull-draw-shot` / `--gpucull-draw`, the `gpucull_draw` golden). Unit-tested (`gpu_culled_test`).

### Hi-Z occlusion culling — completing the GPU-driven culling pipeline (`engine/render/hiz.h`)

Slice CJ adds the **occlusion** half of GPU-driven culling, completing the pipeline that AR (frustum cull → compaction → indirect draw) and CD (compute-cull → MDI + bindless) began: now the GPU culls not just what's **outside the frustum** but what's **hidden behind already-drawn geometry**. A **depth pre-pass** rasterizes the opaque occluders, then `render::hiz` builds a **Hi-Z max-depth pyramid** — a mip chain where each coarser texel holds the **maximum** (farthest) depth of the four finer texels it covers, so a single coarse texel conservatively bounds the depth of a screen region. The occlusion test for a candidate projects its world bounding box to a **screen-space AABB**, picks the Hi-Z mip whose texel size covers that AABB, and compares the candidate's **nearest** depth against the mip's max-depth: if the candidate is entirely farther than the closest occluder in that region it is **fully hidden** and culled. `render/hiz.h` is **header-only, pure-CPU, zero backend symbols** — the pyramid reduction + the conservative screen-AABB test are the single source of truth, mirrored by the GPU path and unit-tested (`hiz_test`).

The test is deliberately **conservative** (it never culls a visible object — only ones provably behind the depth bound), which is exactly what makes it **render-invariant**: the culled objects would have contributed **zero pixels** anyway, so the occlusion-culled render is **byte-identical to the frustum-only render**. `--hiz-cull-shot` runs both and asserts the hashes match: `total:31, frustumKept:31, occluded:24, drawn:7, cpuOccluded:24` and `hiz-cull==frustum-only: BYTE-IDENTICAL`. On **Vulkan** the candidates surviving frustum+occlusion are drawn via the CD MDI + bindless path; **Metal** renders the identical CPU-occlusion-culled survivor subset per-object bound. Golden-captured (`--hiz-cull-shot` / `--hiz-cull`, the `hiz_cull` golden). Combined with frustum culling, the GPU-driven culling pipeline is now **frustum + occlusion**.

### Clustered light culling / Forward+ (`engine/render/cluster.h`)

Slice CL is the **many-lights flagship**: a Forward+ light-culling pass that lets the lit shader touch only the lights affecting a given fragment's screen region. `render::cluster` divides the view frustum into a **`16×9×24` cluster grid** (3456 clusters — `16×9` tiles across the screen, 24 exponential z-slices in depth), computes each cluster's view-space **AABB**, and `AssignLights` tests **every light's bounding sphere against every cluster's AABB** (sphere-AABB overlap), producing a per-cluster `{offset, count}` plus a flat **light-index list** that the lit pass reads (the set-3 storage buffers bound via `BindLightClusters`, exactly as the earlier `render/clustered.h` Forward+ path). The grid construction + sphere-AABB assignment is **header-only, pure-CPU, shared math, zero backend symbols** — the single source of truth for both the GPU compute assignment and the CPU reference, unit-tested (`cluster_test`).

Determinism + render-invariance are proven two ways. First, the **GPU assignment is byte-identical to the CPU reference**: the compute pass and `render::cluster::AssignLights` produce the **same** flat index list (GPU assigned `5606` == CPU ref `5606`, with `maxPerCluster:34` inside the valid `(1, 96)` range), and the per-cluster brim buffer is two-run byte-identical (`brimByteIdentical:true`). Second, the **clustered shading is byte-identical to brute-force**: shading only each cluster's assigned lights produces the same image as shading every one of the 96 lights per fragment (`clustered==brute-force: BYTE-IDENTICAL`). The showcase reports `lights:96, clusters:3456, maxPerCluster:34, avgPerCluster:1.62, brimByteIdentical:true`; golden-captured (`--clustered-lights-shot` / `--clustered-lights`, the `clustered_lights` golden), two-run byte-identical.

### Data-driven post-process stack (`engine/render/post_stack.*`)

The post-process stack (Slice BN) makes the final post chain **data-driven**: a `PostStack` is an **ordered** list of `PostEffect`s applied in sequence to the resolved scene color. The effect kinds are **ACES `Tonemap`**, **`ColorGrade`** (per-channel lift/gamma/gain), **`ChromaticAberration`** (radial RGB split, strength in pixels), **`Vignette`**, and **`FilmGrain`** (additive luminance noise from a fixed 32-bit integer hash of the *integer* pixel coord — no time/frame input, so golden-stable). The per-effect math lives in `engine/render/post_stack.h` and is **shared** by both the in-shader stack (`shaders/post_stack.frag.hlsl`) and the unit test (`post_stack_test`) — the same `render/ssr.h` / `render/decal.h` sharing pattern — so the CPU mirror and the GPU evaluate bit-for-bit identically (the grain hash uses defined 32-bit unsigned wraparound so HLSL `uint` math matches).

The shader encodes the whole chain into a **flat float push-constant stream** to honor the **128-byte guaranteed `maxPushConstantsSize` minimum**: a `float4[7]` (28 floats = 112 B) + an `int4` count (16 B) = exactly **128 bytes**. Each effect contributes `kind` followed by that kind's params, and the shader walks the stream with a running cursor, applying each effect in order — a per-effect `vec4` array would overflow a 128-byte GPU, so the flat-stream layout is the binding-budget constraint that shapes the design. The stack is **data** (`LoadPostStack` parses it from JSON via the vendored single-header parser; an absent/empty `"effects"` array yields a pass-through, an unknown `"kind"` fails the load) so an agent can compose looks. Pure CPU, zero backend symbols; deterministic two-run byte-identical; golden-captured (`--poststack-shot` / `--poststack`, the `poststack` golden) — the showcase's `DefaultShowcaseStack` is the 5-effect cinematic chain tonemap→colorgrade→chromatic→vignette→grain (`count:5`).

### Sky / atmosphere suite — water, depth of field, volumetric clouds, cloud shadows (`engine/render/water.h`, `engine/render/dof.h`, `engine/render/clouds.h`)

Slices CF–CK complete the **sky / atmosphere suite** (sky + water + clouds + cloud shadows) and a companion lens effect. All three follow the established `render/ssr.h` discipline: a **header-only, pure-CPU, zero-backend-symbol** math core that is the single source of truth **shared verbatim** with the in-shader pass, evaluated **deterministically at a fixed time** (no clock / RNG), so a two-run capture is byte-identical and the Metal golden matches the Vulkan render.

- **Water (Slice CF, `engine/render/water.h`).** A **Gerstner-wave** water surface: a fixed set of summed Gerstner waves (`GerstnerDisplace` — per-wave amplitude / wavelength / steepness / direction, evaluated at a fixed `time`) displaces a `gridN×gridN` water mesh on the CPU and the shared shader, and shading adds a **Fresnel-weighted reflect/refract** mix (Schlick Fresnel selecting between a reflected-sky term and a refracted-depth-tinted term). The displacement + normal + Fresnel math is shared with `shaders/water.*` and unit-pinned; the water-mesh pass renders over the scene. Golden-captured (`--water-shot` / `--water`, the `water` golden), reporting `waves:3, time:1.3, gridN:128`, two-run byte-identical.
- **Depth of field (Slice CG, `engine/render/dof.h`).** A **thin-lens circle-of-confusion (CoC) depth gather**: `CoC(depth)` computes each pixel's blur radius from a thin-lens model (`focalDist` / `aperture` / `maxCoC`) over the reconstructed view-space depth, and a fullscreen pass gathers neighbours weighted by their CoC to composite a focus-plane-sharp / out-of-focus-blurred image. The CoC + gather math lives in the header and is mirrored in `shaders/dof.frag.hlsl`. Golden-captured (`--dof-shot` / `--dof`, the `dof` golden), reporting `focalDist:12.0, aperture:90.0, maxCoC:18.0`, two-run byte-identical.
- **Volumetric clouds (Slice CH, `engine/render/clouds.h`).** A **raymarched cumulus** layer: a fullscreen pass marches a fixed `steps` count through a cloud slab, sampling a procedural density field (`coverage`-thresholded), accumulating **Beer-Lambert** transmittance for self-shadowing and an **Henyey-Greenstein (HG) phase** term for forward sun scattering, then composites the sunlit cloud over the sky. The density / Beer-Lambert / HG math is shared with `shaders/clouds.frag.hlsl` and evaluated at a fixed `time`. Golden-captured (`--clouds-shot` / `--clouds`, the `clouds` golden), reporting `steps:64, coverage:0.42, time:2`, two-run byte-identical. Together with the existing procedural sky and the CF water surface, this completes the sky / atmosphere suite.
- **Cloud shadows on the ground (Slice CK, `engine/render/clouds.h` — the `CloudShadow` addition).** The cloud field casts **dappled ground shadows** by attenuating the **direct sun** on the lit scene. `CloudShadow(worldPos, sunDir, time)` marches a fixed `steps` **sun ray** from each shaded world point up through the same `coverage`-thresholded procedural cloud slab, accumulating **Beer transmittance** along the ray — the fraction of sunlight that survives the cloud — which scales the direct-light term in the lit pass. It reuses the same density field as the CH cloud raymarch (one source of truth for both the sky clouds and their ground shadows), is header-only / pure-CPU / zero-backend-symbol, shared verbatim with the lit shader, and evaluated at a fixed `time`. Golden-captured (`--cloud-shadows-shot` / `--cloud-shadows`, the `cloud_shadows` golden), reporting `steps:24, time:2`, two-run byte-identical. This is the fourth member of the suite (sky + water + clouds + **cloud shadows**).

### DDGI dynamic global-illumination arc (`engine/render/probe_gi.h`, `engine/render/probe_capture.h`, `engine/render/probe_sh.h`)

All seven follow the same `render/ssr.h` discipline — **header-only, pure-CPU, shared-math, zero-backend-symbol** cores shared verbatim with the in-shader pass (the only "vk"/"MTL"/`mtl::` mentions in these headers are seam-discipline doc comments, *not* code symbols), each carrying a built-in bit-exact / byte-identical proof plus a **probeCount=0 / disabled-path no-op proof**. **The entire GI arc adds NO new RHI** — every slice reuses the existing compute-pipeline / SSBO surface, the fragment-stage storage-buffer path (`usesLightClusters` / `BindLightClusters`), and (for DI/DO) the DD cubemap render-target RHI unchanged; `rhi.h` is byte-identical to its DG #20 state and the four additive pure-interface capabilities (CO `oitRevealageBlend`; CS `ComputeToComputeBarrier`/`ComputeToFragmentBarrier`; CX `BindShadowMapCompute` + `sampledShadowMap`; DD cubemap-RT set) remain pure interfaces. (The GI showcases run the lit pass, which samples the per-frame shadow map; Slice DQ added the missing empty shadow-pass to their render graphs so the dummy shadow image is correctly transitioned — a pixel no-op that makes every GI showcase Vulkan-validation-clean, and added them to `verify.ps1`'s validation gate.)

**Cross-backend FP discipline (what makes the GPU==CPU bit-exactness real).** A naive GI compute pass would diverge from a CPU reference because GPUs evaluate transcendentals (`sin`/`cos`/`sqrt`) and fused multiply-adds with different rounding than the host. The GI arc sidesteps that entirely: every transcendental is **host-precomputed once** (the Fibonacci-sphere ray directions, the SH sample directions, the `SHBasis9` weights, the per-sample solid-angle weights) and **uploaded as exact bits** that both the GPU pass and the CPU reference read identically, and every accumulation uses **`fma`** so the multiply-add rounds the same way on both sides. The result: the per-probe SSBO the GPU writes is **bit-identical** (`memcmp`) to the CPU reference *and* bit-identical across Vulkan and Metal — the same contract the `gpu_cull` gpu==cpuRef count proof established, now extended to floating-point GI buffers.

- **Probe-grid ray-trace (Slice DH, `engine/render/probe_gi.h`).** A world-space lattice of irradiance probes, each tracing `kRaysPerProbe = 16` deterministic **Fibonacci-sphere** rays against the scene's **view-space depth field** read back as a **flat `float[w*h]` SSBO** (the G-buffer `.w` channel — not a sampled depth texture, mirroring `froxel_inject`'s analytic reconstruct). It re-exports `ssr::ViewToScreenUV` / `ssr::ReconstructViewPos` (as `ssgi.h` does) so the probe march and the in-shader march share the **same** projection, and `FibonacciSphere` / `TraceRayToDepth` are copied verbatim into `shaders/probe_raytrace.comp.hlsl`; `probe_gi_test` exercises the exact march. **GPU==CPU proof:** the compute ray-hit SSBO is BIT-EXACT (`memcmp`) to `probegi::TraceRayToDepth` over the same flat field. **probeCount=0 proof:** `dimX==0` → `DispatchCompute(0)` → the ray-hit SSBO equals the cleared upload (all `kRayMiss`). The compute→readback barrier is SYNC-HAZARD-free under the sync-validation layer. Only the verified ray-trace data layer (no visible indirect lighting yet). Golden-captured (`--probegi-shot` / `--probegi`, the `probegi` golden), reporting `{probes:256, rays:16, hits:1474}`.

- **Probe SH-encode (Slice DJ, `engine/render/probe_sh.h`).** Each captured probe cubemap is encoded into **3rd-order real spherical harmonics** (9 coefficients per RGB channel) by a **pure compute pass** over a fixed host-precomputed sample set (the cross-backend FP discipline above). **GPU==CPU proof:** the per-probe SH SSBO is BIT-EXACT to the CPU `probesh` reference. **zero-radiance proof:** a zero-radiance cubemap encodes to zero-SH. **probeCount=0 proof:** leaves the SH SSBO byte-identical to its cleared value. ### Virtual-geometry / cluster arc (`engine/render/meshlet.h`, `engine/render/cluster_cull.h`, namespace `hf::render::vg`)

A **virtual-geometry pillar** (the 2nd flagship — virtualized massive-detail geometry) that reuses the engine's already-complete GPU-driven-rendering foundation: per-cluster GPU cull is the per-OBJECT GPU cull (Slices AR/CD) with a geometry cluster as the unit, so the whole cull/LOD front-half needs **no new RHI**. The geometry-cluster namespace is `hf::render::vg` (distinct from `hf::render::cluster`, which is Forward+ *light* clustering). Each slice follows the engine discipline — header-only pure-CPU core copied verbatim into the compute shader, a GPU==CPU / byte-identical proof, a disabled-path no-op, and a Metal golden at DIFF 0.0000.

- **Meshlet/cluster decomposition (Slice DS, `engine/render/meshlet.h`).** A deterministic mesh→cluster partition: triangle centroids are quantized into the mesh AABB and **Morton-sorted** (with the original triangle index as a tie-break for a total order → fully deterministic, no float-order ambiguity), then swept into contiguous clusters of ≤ `kMaxTrisPerCluster` (128) triangles with a conservative per-cluster AABB + bounding sphere. The reordered index buffer draws each cluster as an index sub-range. **Proofs:** partition completeness (the cluster triangles are exactly the original index multiset, `Σ triCount == T`), two-build determinism (`memcmp`), conservative bounds (every cluster vertex inside its AABB + radius — the contract DT/DU rely on), degenerate/empty no-op. Pure integer sort + min/max → bit-identical on every platform; the viz colors each cluster by a deterministic integer hash. Golden-captured (`--meshlet-viz`, the `meshlet_viz` golden = a sphere segmented into flat-colored spatial cluster patches), reporting `{tris:3072, clusters:24, maxTris:128}`. No new RHI, no GPU math.
- **GPU per-cluster frustum cull → indirect cluster draw (Slice DT, `engine/render/cluster_cull.h`).** The DS clusters of an instance grid are culled on the GPU at (instance × cluster) granularity by frustum (each cluster's world bounding sphere, derived via `gpu_cull.h::InstanceWorldSphere`), and the survivors drawn via MDI. `cluster_cull.comp.hlsl` is a near-copy of `gpudriven_cull.comp.hlsl` — the same single-workgroup Hillis-Steele **ordered** prefix-sum compaction (deterministic source-cluster-instance order) with `frustum::SphereOutside` copied verbatim. A surviving cluster → one `MdiCommand{indexCount = triCount·3, firstIndex = triOffset·3, firstInstance = sourceIndex}`. **Proofs:** GPU survivor count == CPU `frustum.h` reference (via `ReadBuffer`); the GPU-compacted-MDI render is BYTE-IDENTICAL to the CPU-culled render; all-clusters-in-frustum == the full unculled draw; two-run/two-process determinism. No new RHI (reuses the compute-compaction + MDI + readback surface). Golden-captured (`--cluster-cull-shot` / `--cluster-cull`, the `cluster_cull` golden = a 4-sphere row with the off-frustum 2 culled away, 48/96 cluster-instances surviving). On Metal the same survivor set renders via the CPU per-cluster bound path (the `gpudriven_cull` convention; the image is backend-identical and the Vulkan side carries the GPU==CPU proof).
- **GPU per-cluster Hi-Z occlusion cull (Slice DU, `engine/render/cluster_cull.h` + `shaders/cluster_hiz_cull.comp.hlsl`).** On top of DT's frustum cull, a cluster fully hidden behind closer geometry is dropped — the per-cluster analogue of the per-object Hi-Z cull (Slice CJ), with `hiz.h::IsOccluded` (the verbatim shared math) tested against each cluster's bounding-sphere AABB over a Hi-Z pyramid built from the depth pre-pass. **Make-or-break proof:** because the Hi-Z is conservative (only culls clusters guaranteed fully occluded), the occlusion-on render is BYTE-IDENTICAL to the frustum-only (DT) render — the culled clusters were invisible anyway — while the survivor count measurably drops (a 5-sphere scene behind an occluder wall: 120 → 48, the 3 hidden spheres' 72 clusters culled). Also `occlusionEnabled = false` == DT byte-identical, and GPU survivor count == CPU reference / GPU-culled == CPU-culled. No new RHI (reuses the DT/CJ depth-prepass + readback + compute-compaction + MDI surface). Golden-captured (`--cluster-hiz-shot` / `--cluster-hiz`, the `cluster_hiz` golden = the occluder wall + the two visible side spheres).
- **Discrete cluster-LOD by screen-space error (Slice DV, `engine/render/cluster_lod.h` + `shaders/cluster_lod_select.comp.hlsl`).** The final core slice: per instance, one of 3 pre-baked discrete LOD meshes (tessellations of the mesh, each DS-cluster-decomposed, with a conservative per-LOD geometric error) is selected by **projected screen-space error** — near geometry draws full detail (LOD0), far geometry a coarser LOD. The selection is kept **GPU==CPU bit-exact** by comparing on **squared** view-distance (no `sqrt`, which is not bit-identical CPU↔GPU) against `(geometricError · projScale / threshold)²` with a host-precomputed projection scalar `screenH / (2·tan(fovY/2))` and `std::fma` — the same discipline the GI buffers use, now for an integer LOD index. **Disabled-path proof:** `forceLod0` (error threshold 0) selects LOD0 for every instance → BYTE-IDENTICAL to the full-detail render. Also distance-monotonic (a farther instance never picks a finer LOD) and the selected set spans ≥ 2 LODs (a real falloff). No new RHI. Golden-captured (`--cluster-lod-shot` / `--cluster-lod`, the `cluster_lod` golden = a row of spheres marching away, fine clusters near → coarse far). Together DS–DV are the recognizable **virtual-geometry cull/LOD front-half** (deterministic decomposition → GPU frustum + Hi-Z cluster culling → screen-error LOD), with zero RHI additions and every stage byte-exact-provable.

### Visibility buffer (`engine/render/visbuffer.h`, `engine/render/visresolve.h`)

Slices DW–DX add a **deferred-geometry rendering architecture**: rasterize cluster + triangle IDs into a screen-size integer buffer, then resolve material/shading per pixel from that buffer — decoupling geometry from shading so shading cost is per-visible-pixel, not per-triangle. This introduces the virtual-geometry arc's **only RHI addition**: a single additive `Format::R32_Uint` enum value (+ its two backend pixel-format mappings + two readback-bpp cases), defaulted-no-op — selected by no existing call site, so every pre-existing render target / pipeline / golden is byte-for-byte unchanged. No new interface method.

- **Vis-buffer render (Slice DW, `engine/render/visbuffer.h`).** The DS–DV cluster MDI draw rasterizes `PackVisId(clusterID, triID) = (clusterID << 7) | triID` into an `R32_Uint` target (depth-tested so the front-most surface wins per pixel; `SV_PrimitiveID` supplies the triangle ID). The written value is a flat integer — no interpolation/FP/blend — so it is inherently bit-exact cross-vendor. **Proof B (bit-exact, vendor-robust, via raw `ReadRenderTarget` memcmp — never depends on matching the hardware rasterizer's edge rules):** every written texel decodes to a valid survivor (`clusterID < survivorCount`, `triID < that cluster's triCount`) and the distinct clusterIDs present are exactly the on-screen subset of the CPU cull's survivors; two renders are byte-identical; and at deterministically-chosen cluster-interior pixels the GPU ID equals the CPU-predicted covering cluster. The image golden CPU-colors the read-back IDs by `hashColor(clusterID)` → identical on both backends by construction. Golden `visbuffer` (the clustered spheres as per-cluster ID patches). (Cross-backend caveats handled: `SV_PrimitiveID` requires MSL 2.2 on Metal — isolated per-shader so other goldens are untouched; and the integer-target clear uses a `0xFF000000` background sentinel that is float32-lossless and decodes outside the valid ID range, since `0xFFFFFFFF` reinterpreted through the float clear path becomes NaN→0.)
- **Deferred material resolve (Slice DX, `engine/render/visresolve.h`).** A fullscreen pass **texel-fetches** the integer vis-buffer (`Texture2D<uint>.Load`, `OpImageFetch` / MSL `texture.read`, no sampler — reusing the existing `BindTexture`, no new RHI), reconstructs the covering triangle's vertices from the cluster/index/vertex SSBOs, computes a **flat** geometric normal + Lambert, and shades — the spheres rendered via the vis-buffer instead of a forward lit pass. **Honest proof scope:** a byte-identical "resolve == forward render" proof is *not* cross-vendor-feasible (hardware-rasterizer barycentric/fill-rule divergence the FP discipline can't close — even flat shading depends on which triangle covers each edge pixel). So DX proves **ID-provenance** (the resolve reads the correct IDs at interior pixels — integer equality inherited from DW) + **GPU==CPU resolve-math** bit-exact (the flat-normal + Lambert at interior pixels matches the `ResolveFlatShade` CPU mirror, `std::fma` + host-precomputed light), with cross-backend guaranteed by the same-shader **image golden** and a forward-render **smoke bound** (resolve-vs-forward ≤ ~1 LSB, explicitly not 0.0000). Golden `visresolve` (the flat-shaded faceted spheres lit from the vis-buffer). A runtime cluster-DAG simplifier is the deferred stretch.
- **Software rasterization (Slices SW1–SW2, `engine/render/swraster.h` + `shaders/swraster.comp.hlsl`).** The alternative path that *produces* the visibility buffer for sub-pixel / tiny triangles where a hardware rasterizer is inefficient: a deterministic **integer edge-function** scan-converter that packs `depth|id` (depth high bits, the same `PackVisId` low bits) into a per-pixel buffer and resolves the front-most surface by a **`min` over the 32-bit key**. SW1 is the pure-CPU reference (`swraster.h`): int64 edge functions, a watertight **top-left fill rule** (a shared cluster edge covers each boundary pixel exactly once — no gap, no double-cover), and host-snapped fixed-point screen verts. SW2 is the GPU compute version (`swraster.comp.hlsl`, one thread per triangle) writing the same packing into a `RWStructuredBuffer<uint>` via **`InterlockedMin`** — **no new RHI** (the existing `InterlockedAdd` storage-buffer family; no storage image, no 64-bit atomic). **The make-or-break proof:** the GPU vis-buffer is BIT-IDENTICAL to the CPU `swraster.h` reference at *every* pixel (full-frame `ReadBuffer` memcmp) — guaranteed because the GPU consumes the same integer screen verts (zero GPU FP), runs the verbatim integer edge math (vendor-independent coverage), and the `InterlockedMin` is commutative (thread-race cannot change the result). This is *stronger* than the hardware vis-buffer's interior-only oracle — there is no edge fuzz. Goldens `swraster` (CPU) and `swraster_gpu` (a clustered sphere, watertight, depth-occluded, plus a sub-pixel triangle the hardware path would miss). (Cross-toolchain note: the int64 edge math compiles under DXC for the Vulkan SPIR-V path but not glslang's HLSL frontend, so `swraster.comp` is Vulkan-only and Metal renders the identical result via the CPU `swraster.h` path — the `cluster_cull.comp` convention; the Vulkan side carries the GPU==CPU proof.) Slice SW4 then **feeds the SW vis-buffer into the deferred resolve**: a fullscreen int32 "unpack" blit (`swraster_resolve_blit.frag`) strips the depth high-bits and writes the low-16 `visId` into the `R32_Uint` target the DX resolve already reads, so `visresolve.frag` is reused byte-for-byte — completing a full software-raster → deferred-shade pipeline (golden `swraster_resolve`, a Lambert-lit software-rasterized sphere; the integer vis-buffer + blit are cross-backend bit-identical, the float Lambert resolve is the same per-backend-golden + interior-bit-exact bar as DX). The HW-large/SW-small hybrid dispatch is the remaining deferred stretch.

### Virtual shadow maps (`engine/render/vsm.h`, namespace `hf::render::vsm`)

Slices VA–VC add a **clipmap-organized directional Virtual Shadow Map** — virtual-geometry-scale shadowing reduced to its byte-exact-provable core: a huge virtual shadow resolution sparsely backed by physical pages, allocated only where the visible receivers need them. It reuses the existing shadow-atlas RHI (`CreateShadowMap` + `SetViewport` tiling + `SetShadowMap`) and the flat-SSBO + readback discipline wholesale, so the whole arc needs **no new RHI**. The page table, allocation, and per-pixel page selection are all integer-deterministic → bit-exact GPU==CPU; the shadowed image is a normal lit golden.

- **Page table + page-needed marking (Slice VA, `engine/render/vsm.h`).** A clipmap of `levels` rings, each covering `level0WorldExtent · 2^L` world units across `virtualPagesPerSide²` pages, centered on the camera. A compute pass marks which virtual pages the visible receivers need: per receiver, `level = SelectClipmapLevel(distToCamera)` — selected by an **integer threshold-ladder** against host-precomputed `level0WorldExtent·2^L` bounds, *not* a GPU `log2`, so the level (and thus the whole resident set) is bit-identical cross-backend — then project to `(px,py)` and mark `resident[pageId]=1`. **Proof:** the GPU-marked resident set equals the CPU `vsm.h` reference byte-for-byte (`ReadBuffer` memcmp, integer set, no FP); two runs identical; `markingEnabled=false` → empty. Golden `vsm_pages` (a top-down clipmap viz, resident pages `hashColor`-lit over concentric rings, CPU-colored from the bit-exact set → identical both backends by construction).
- **Physical-page depth rendering (Slice VB, `engine/render/vsm.h`).** A deterministic allocator assigns each resident virtual page a physical atlas tile (an integer virtual→physical indirection table, bit-exact), and each resident page's shadow casters are rendered into its tile via the CSM `SetViewport` atlas loop, each with that page's clipmap-level ortho sub-frustum (`PageWorldOrtho`). **Proof:** the indirection table is two-run byte-identical and matches the CPU `AllocatePhysicalPages` reference; `markingEnabled=false` → empty atlas. (Since a `CreateShadowMap` depth atlas has no color image for `ReadRenderTarget`, the VB *golden* renders depth-as-grayscale into a color RT — a viz-only trick; the actual shadow sampling in VC uses a real depth atlas.) Golden `vsm_atlas` (the physical tiles, each holding its page's caster-depth silhouettes).
- **Lit-pass indirection sample (Slice VC, `shaders/lit_vsm.frag.hlsl`).** A sibling lit pass shadows each receiver pixel through the VSM: compute its clipmap level + virtual page (the VA math copied verbatim), look up the physical tile in the indirection SSBO, project into that page's light-space, and sample the tile's depth with tile-clamped 3×3 PCF — the standard CSM depth-compare semantics, with the tile origin fetched from the page table instead of a cascade index. The VSM atlas binds as a depth-only shadow map via the existing `SetShadowMap`; the indirection SSBO via the existing `BindLightClusters` fragment-storage path — **no new RHI**. **Make-or-break no-op:** `vsmEnabled=0` forces the shadow factor to a literal `1.0` → the render is BYTE-IDENTICAL to the unshadowed lit; **page-lookup proof:** at interior receiver pixels the `(level, pageId, tile)` the shader resolves equals the CPU `vsm.h` reference (integer, bit-exact); `vsmEnabled=1` casts coherent shadows (differs from A), and a single-clipmap-level VSM is within a small epsilon of the existing CSM shadow (an equivalence smoke, not a byte match). Golden `vsm_shadow` (the lit scene with VSM-cast shadows). Per-page caching is the deferred stretch.

### Runtime virtual texturing (`engine/render/vt.h`, namespace `hf::render::vt`)

Slices VT1–VT5 are a complete **Runtime Virtual Texturing** core (the literal RVT technique): a virtual texture far larger than VRAM, sampled through a virtual→physical page indirection table driven by per-pixel **feedback**. It is structurally the **same machinery as the VSM clipmap arc applied to texture pages** — a virtual page table → a feedback/marking pass → physical tile allocation + indirection → (later) a sampled lookup → per-page caching — and reuses the VSM page-table/allocator discipline near-verbatim, so the whole arc needs **no new RHI**. UV-space page math has no clipmap-snap/world-projection, making RVT structurally *simpler* than VSM; the page IDs / residency sets / indirection tables / procedural page content are pure integer, so **every slice of the VT1–VT5 core is cross-backend bit-identical** (the strict zero-differing-pixel posture, not the float-smoke bar) — only the deferred bilinear/3D-textured refinement introduces float.

- **Page-needed feedback / marking (Slice VT1, `engine/render/vt.h`).** The virtual texture is a **mip pyramid**: `PageId(mip, px, py) = mipPageOffset(mip) + py·pagesPerSide(mip) + px`, where `pagesPerSide(m) = virtualPagesPerSideMip0 >> m` and the per-mip offset is a host **prefix-sum** of `pagesPerSide(k)²` (unlike VSM's flat `level·vpps²`, since mips differ in size) — a bijection with `UnpackPageId`. A compute pass marks which pages a set of (UV, mip) sample-requests need: `VtPageId` quantizes `floor(u·pagesPerSide(mip))` → `feedback[pageId]=1` (an order-independent integer set, no atomics). The one float→int boundary (the UV quantize) is eliminated entirely by **host-snapping** each request to integer `(mip,px,py)` before upload (the `swraster.h` discipline), so the GPU does zero floating point and the feedback set is trivially bit-exact. Mip selection uses the VSM integer threshold-ladder `SelectMipLevel` (no `log2`). **Proof:** the GPU feedback set equals the CPU `MarkFeedbackPages` reference byte-for-byte (`ReadBuffer` memcmp); `feedbackEnabled=false` → empty; two runs identical. Golden `vt_feedback` (a per-mip page grid, resident pages `hashColor(pageId)`-lit — CPU-colored from the integer set → identical both backends by construction).
- **Physical tile-pool allocation + indirection (Slice VT2, `engine/render/vt.h`).** A finite physical tile pool (`VtTilePool`, `tilesPerSide²` tiles) and a **deterministic ascending-priority allocator** (`AllocatePhysicalTiles`, the VSM `AllocatePhysicalPages` analog): walk pageId ascending, each resident page takes the next sequential tile while the pool has room, else `kNoTile` (non-resident or overflow). Ascending pageId is mip-major = **finest-mip-first** priority, so the nearest/highest-detail pages win the pool when it overflows. Allocation is inherently sequential, so the GPU mirror is a single-thread compute scan (`shaders/vt_alloc.comp.hlsl`) — **no new RHI**. **Proof:** the GPU indirection table equals the CPU `AllocatePhysicalTiles` reference byte-for-byte; `allocated = min(resident, capacity)`, tile indices unique and in-range, the first `capacity` resident pages by pageId are the allocated ones; `allocEnabled=false` → all `kNoTile`; two runs identical. Golden `vt_alloc` (the per-mip indirection viz: allocated pages tile-colored, overflow-resident a dim color, non-resident dark — bit-identical both backends).
- **Procedural page generation into the physical atlas (Slice VT3, `engine/render/vt.h` + `shaders/vt_pagegen.comp.hlsl`).** GENERATE each allocated page's content into its physical tile — the analog of VSM-VB's physical-page render, writing procedural texture color instead of caster depth, kept pure-integer. `PageTexel(pageId, mip, lx, ly)` is a deterministic per-texel generator (an FNV/avalanche hash of `(pageId, mip)` for the per-page base color, modulated by an 8×8 integer checkerboard + a coarse gradient over the page-local `(lx, ly)`; every channel floored into `[0x40,0xFF]` so it never collides with the `kAtlasClear = 0xFF101010` sentinel). A compute pass (one thread per atlas texel, a race-free map via a host-built `tilePageId` reverse table) fills the physical atlas SSBO; pure shifts/xor/add/mul → no float. **Proof:** the GPU atlas equals the CPU `GeneratePhysicalAtlas` reference byte-for-byte; exactly the allocated tiles are non-clear and each is fully written; `genEnabled=false` → all clear; two runs identical. Golden `vt_atlas` (the physical atlas: each allocated tile a distinct tinted patterned page, unallocated dark — bit-identical both backends).
- **Material-pass sample through the indirection (Slice VT4, `engine/render/vt.h` + `shaders/vt_sample.comp.hlsl`).** The round-trip that makes RVT a usable texture: `SampleVirtualTexel(vx, vy, mip)` computes the virtual page → looks up `indirection[pageId]` → offsets into the physical atlas via `PhysicalTileOrigin` → reads the texel. **Design call: NEAREST (integer) sampling**, so the sample path stays bit-exact and the golden is cross-backend bit-identical — bilinear filtering and perspective-correct UV on real geometry are an explicit deferred float refinement (the round-trip is proven first, then the filter). A compute pass reconstructs the virtual texture's mip-0 image in virtual-UV space (one thread per virtual texel); a non-resident page samples `kVtMiss`. **Proof:** the GPU image equals the CPU `ReconstructVirtualImage` reference byte-for-byte; every resident virtual texel reads back exactly what VT3 wrote (`== PageTexel`, the indirection is a correct virtual→physical map); page-lookup matches the CPU at interior texels; `sampleEnabled=false` → all miss; two runs identical. Golden `vt_sample` (the reconstructed virtual texture: resident pages textured at their UV location, non-resident pages the miss color — visually distinct from VT3's physical-atlas layout, bit-identical both backends).
- **Per-page caching across frames (Slice VT5, `engine/render/vt.h` + `shaders/vt_cachegen.comp.hlsl`).** Skip regenerating physical tiles whose page content is unchanged — the analog of VSM-VD's `PageContentKey`/`VsmPageCache`. `PageContentKey(pageId, mip, contentVersion)` is an FNV hash of the page's content identity; a per-tile cache stores `(key, valid)`; a tile regenerates iff its key changed. `PageTexel` becomes `PageTexelV(…, contentVersion=0)` (version 0 reproduces VT3's bytes exactly — the VT3/VT4 goldens are the guard). The cache-aware compute sibling early-returns for cached tiles, leaving the **persistent** atlas SSBO untouched. **Proof (the VSM-VD SHA-equality lineage, here direct `memcmp`):** Pass 2 re-runs the same scene → all cache hits, atlas byte-identical to Pass 1 (cached == fresh); Pass 3 bumps one page's content version → exactly one tile regenerates and the partial-cached atlas is byte-identical to a full uncached regen (the cache is transparent); two runs identical. Golden `vt_cache` (the cache-status viz: green-washed cached tiles + the one red regenerated tile + dark unallocated — bit-identical both backends). The whole VT1–VT5 core adds **no new RHI** and every slice is cross-backend bit-identical; the bilinear-filter / textured-3D-geometry float refinement is the deferred next step.

### GPU isosurface meshing (`engine/render/mc.h`, namespace `hf::render::mc`)

Slices MC1–MC3 begin a **GPU Marching-Cubes isosurface mesher** — extract a triangle mesh from a scalar (voxel/SDF) field, the foundation of real-time procedural & destructible geometry (a capability with no mainstream-engine core-flagship equivalent). It decomposes into exactly the integer primitives this engine has golden-proven cross-backend, so the whole core is **cross-backend bit-identical** (the strict zero-differing-pixel bar) and adds **no new RHI**. The SDF field is generated host-side by a deterministic pure function (the `heightmap.h::Height` discipline — no RNG/clock/disk).

- **Per-cell case classification (Slice MC1, `engine/render/mc.h` + `shaders/mc_classify.comp.hlsl`).** A `VoxelField` of `int32` quantized scalars; a locked 0–7 corner numbering (`kCornerOffset`) + 0–11 edge numbering. A compute pass classifies each voxel CELL into its 8-bit Marching-Cubes case index: `CaseIndex = Σ ((corner[i] > isovalue) ? 1 : 0) << i` — eight pure integer compares, no transcendental, the cross-backend crux. **Proof:** the GPU case-index buffer equals the CPU `ClassifyCells` reference byte-for-byte; hand-checked known cells (`empty=0x00`, `full=0xFF`, `corner0=0x01`); disabled → all-zero; two runs identical. Golden `mc_classify` (a grid of Z-slices through a sphere field, `hashColor(caseIndex)` per surface cell — the surface cells form concentric rings, bit-identical both backends).
- **Per-cell triangle count (Slice MC2, `shaders/mc_count.comp.hlsl`).** The canonical 256-case triangle table (`kTriTable`, Bourke/Lorensen-Cline — it matched the locked corner/edge numbering directly) and its derived `kTriCount`. The compute pass looks up each cell's triangle count and accumulates the grand total via `InterlockedAdd` — an integer atomic that is commutative → order-independent → deterministic → cross-backend bit-identical (the autoexposure-histogram precedent). **Proof:** per-cell counts AND the atomic total equal the CPU reference (5240 triangles for the test sphere); the table is self-consistent (`kTriCount` derived = `kTriTable`, empty/full = 0); disabled → zero; two runs identical. Golden `mc_count` (the surface cells colored by triangle count).
- **Prefix-sum compaction + triangle emission (Slice MC3, `shaders/mc_scan.comp.hlsl` + `shaders/mc_emit.comp.hlsl`).** A single-thread exclusive prefix-sum of the per-cell counts (the VT2 single-thread-scan pattern — the allocation order is sequential, so one thread keeps it GPU==CPU bit-exact) gives each cell its write offset; then one thread PER CELL emits its triangles into preallocated vertex+index SSBOs at disjoint ranges (race-free, no atomics). Vertices are cell-edge **midpoints** in half-grid integer units (`EdgeMidpoint` = the two corner grid-coords summed → no division, no float); a triangle soup with identity indices (cross-cell vertex dedup deferred). **Proof:** the GPU prefix-offsets AND the full vertex+index buffers (15720 verts for the sphere) equal the CPU `MarchCells` reference byte-for-byte; a hand-checked cell's first vertex; disabled → empty; two runs identical. Golden `mc_emit` (a deterministic 2D orthographic projection of the generated mesh — the sphere surface as a filled-disk point cloud, bit-identical both backends). The whole MC1–MC3 core reuses the autoexposure `InterlockedAdd`, the VT2 single-thread scan, and the swraster fixed-point lattice.
- **Fixed-point interpolated vertices (Slice MC4, `EdgeInterp`).** Replace the cell-edge midpoint with the ACTUAL isosurface crossing: `t = (iso - s0)/(s1 - s0)` computed as a TRUNCATING integer divide into a `1/256` edge lattice (the swraster `kSub` trick) — `t = ((iso - s0) * kSub) / (s1 - s0)`, clamped, `vertexPos = Ca*kSub + (Cb - Ca)*t`. The same truncating integer divide on C++/HLSL/MSL stays bit-identical (int32, no overflow for this field), so the smooth-surface mesh is still GPU==CPU byte-exact and cross-backend zero-diff. **Proof:** the interpolated vertex+index buffers equal the CPU reference byte-for-byte (15720 verts); every vertex lies on its edge and on the isosurface to lattice precision (`max |field - iso| ≤ 1`). Golden `mc_interp` (a smoother sphere projection than `mc_emit`). MC1–MC4 are the strict cross-backend bit-identical integer core.
- **Lit render + smooth normals (Slices MC5–MC6).** MC5 RENDERS the extracted mesh as a lit 3D surface through the EXISTING terrain lit-mesh pipeline (no new shader, no new RHI): the fixed-point mesh → float world positions (`vert/kSub`) + flat per-face normals → `BindVertexBuffer`/`DrawIndexed`. MC6 swaps the flat normals for per-vertex SDF FIELD-GRADIENT normals (`-∇f` via central differences) for smooth Gouraud shading. These two are the arc's FIRST FLOAT slices — a perspective-lit raster render is cross-vendor float, so they are held to the **float bar** (the visresolve/SW4 discipline): the golden is baked on Metal, the gate is Metal-render == Metal-golden DIFF 0.0000 (deterministic) + mesh provenance (the rendered geometry IS the MC4 bit-exact mesh) + visual parity, with the Vulkan-vs-Metal delta the documented cross-vendor smoke (NOT zero, the engine's ~55–60-mean float-render baseline). Goldens `mc_render` (faceted lit sphere) + `mc_normals` (smooth lit sphere). This completes the 6-slice GPU Marching-Cubes flagship — a voxel/SDF field meshed on the GPU (bit-exact) and rendered as a lit 3D surface, with zero new RHI across the whole flagship.

### Deterministic fixed-point physics (`engine/sim/fpx.h`, namespace `hf::sim::fpx`)

Slices FPX1–FPX6 are a **complete deterministic fixed-point rigid-body simulator** (flagship #6) — the first physics in the engine that is BIT-IDENTICAL across Vulkan/Metal/compilers AND frame-to-frame/run-to-run reproducible. The existing `engine/physics/` is float (held to a "visually identical at rest" bar); `engine/sim/fpx.h` is an additive, parallel, Q16.16-fixed-point reimplementation. The motivation is concrete and a genuine capability gap: the engine already ships snapshot replication + client-prediction, but float physics diverges across machines, so true cross-platform **lockstep / rollback netcode** (peers re-simulate from inputs alone to identical state) is impossible with it — exactly the limitation mainstream float physics has. A fixed-point solver is the missing primitive, and FPX5 lands the proof (replica==authority bit-for-bit from inputs alone, on two platforms) — the arc's beyond-mainstream headline. The whole arc adds **no new RHI** and **no new shader beyond FPX1–FPX4's compute passes**; the integer/compute outputs are strict cross-backend bit-identical (the zero-differing-pixel bar), and only the final FPX6 render is float (the visresolve-bar).

- **Q16.16 integrator (Slice FPX1, `engine/sim/fpx.h` + `shaders/fpx_integrate.comp.hlsl`).** A `fx = int32` Q16.16 scalar with `fxmul(a,b) = (int64(a)*b) >> 16` (int64 intermediate); `FxVec3`; `FxISqrt`/`FxLength` (integer floor-sqrt copied from `mc.h::ISqrt` — no `std::sqrt`, no `<cmath>` on the bit-exact path); `FxBody`/`FxWorld`; `IntegrateStep` (semi-implicit Euler + a ground floor-clamp). The compute pass consumes **host-snapped Q16.16 integers and does zero floating point**, so it is GPU==CPU bit-exact by construction (the swraster/VT1 host-snap discipline). **Cross-toolchain note:** `fxmul`'s int64 compiles under DXC (the Vulkan path) but glslc (the Metal HLSL→SPIR-V→MSL frontend) cannot parse `int64_t`, so `fpx_integrate.comp` is Vulkan-only and the Metal showcase runs the CPU `IntegrateStep` (byte-identical by construction — the `swraster.comp` convention). **Proof:** GPU bodies == CPU reference byte-for-byte after K steps; disabled no-op; two runs identical; a hand-checked closed-form fall. Golden `fpx` (the settled bodies, CPU-colored from the integer read-back → identical both backends).
- **Integer broadphase (Slice FPX2, `shaders/fpx_pair_{count,scan,emit}.comp.hlsl`).** An integer-AABB sweep generates the deterministic candidate collision-pair list: `AabbOverlap` is six **int32 compares** (no int64, no products) — so unlike FPX1 these shaders MSL-generate natively and run as a true GPU pass on both backends. The count→prefix-sum→emit compaction reuses the MC3/VT2 pattern (per-body count, a single-thread exclusive scan, per-body emit at the offset). **Proof:** the per-body counts, prefix offsets, and the full `(i,j)` pair list all equal the CPU `BuildPairs` reference byte-for-byte. Golden `fpx_pairs` (the pair-adjacency matrix — a banded structure of the grid neighbors).
- **PBD collision response (Slice FPX3, `shaders/fpx_solve.comp.hlsl`).** A position-based-dynamics positional solver resolves ground + sphere-sphere contacts over the FPX2 pairs: move bodies apart along the contact normal by their inverse-mass-weighted share of the penetration, K Gauss-Seidel iterations in a fixed order. `fxdiv = (int64(a) << 16) / b` (truncating Q16.16 divide) and `FxNormalize` (via `FxISqrt`) keep the impulse-free math fully integer. The solver is **single-thread** (`[numthreads(1,1,1)]`) — sequential Gauss-Seidel is order-dependent, so one thread guarantees bit-exactness (the VT2 single-thread pattern); it is int64 → Vulkan-only + the Metal showcase runs the CPU `StepWorld`. **Proof (the make-or-break):** the bodies after K integrate+solve steps equal the CPU reference byte-for-byte; the residual-overlap count is deterministic (bit-exact, not necessarily zero after K iterations); a hand-checked contact splits by the exact Q16.16 inverse-mass shares; two runs identical. Golden `fpx_solve` (the settled bodies).
- **Integer quaternion orientation (Slice FPX4, `shaders/fpx_orient.comp.hlsl`).** Promotes the sim to full 6-DOF: `FxQuat{x,y,z,w}` with `FxQuatMul` (the Hamilton product, each term an int64 `fxmul`), `FxQuatNormalize` (length via `FxISqrt` of the int64 Q32.32 sum-of-squares, `fxdiv` each component — no `std::sqrt`), `FxRotate`, and `IntegrateOrientation`/`IntegrateBodyFull` (a body's rotation integrated from its angular velocity). `orient`/`angVel` are **defaulted** on `FxBody` so FPX1–FPX3 and their GPU packs stay byte-identical. One thread per body (independent), int64 → Vulkan-only + the Metal showcase runs the CPU `IntegrateBodyFull`. **Proof:** GPU == CPU byte-for-byte over K steps; the `|q|-1` normalization drift is ~zero; a hand-checked analytic spin. Golden `fpx_orient` (per-body orientation gizmos).
- **Lockstep + rollback (Slice FPX5, pure CPU — `FxCommand`/`ApplyCommand`/`SimTick`/`SnapshotWorld`/`RestoreWorld`/`RunLockstep`/`RunRollback`).** The arc's headline, and a capability mainstream float physics cannot provide. A peer fed the **input command stream alone** (no full-state sync) re-derives the authority's exact state — `RunLockstep`'s `replica == authority` is bit-for-bit. `RunRollback` saves a snapshot, advances with a *wrong* predicted input, then `RestoreWorld` + re-simulates with the corrected input to the exact authoritative state (with the mispredicted path differing first — a positive+negative control proving the rollback fixed a real divergence). No new shader/RHI (a CPU harness over the existing fixed-point sim). **Proof:** the five-body 24-tick scenario re-simulates byte-identically on Vulkan-Windows and Metal-Mac (the cross-platform lockstep pixel compare is zero-differing-pixel); snapshot round-trip and determinism. Golden `fpx_lockstep` (the converged state).
- **Lit 3D render (Slice FPX6, render-only — `FxToFloat`/`FxBodyTransform` + `BuildPileWorld`/`StepPileToSettled`).** The money-shot: the bit-exact fixed-point state rendered as a lit 3D scene. A settled pile (a ring of static corral spheres confining a dynamic drop — the FPX3 PBD is frictionless, so the corral is what yields a recognizable mound) is built and stepped by the **existing integer sim**, then each settled `FxBody` becomes a float model matrix via `FxBodyTransform = translate(pos/kOne)·rotate(the float-normalized quat of orient)·scale(radius/kOne)` — the single host float crossing, render-only, out of the bit-exact path. The bodies render as lit 3D **instanced** spheres through the **existing instanced lit pipeline reused verbatim** from `--instanced-shot` (`lit_instanced.vert` + `lit.frag` + `scene::InstanceTransformLayout` + the frame UBO + sky/shadow/post) — no new shader, no new RHI. This is the arc's one **float** slice (the visresolve-bar like the MC mesher's render): the sim is bit-exact but the final raster/shade is float, so the golden is Metal-baked and the bar is Metal-determinism (two-run DIFF 0.0000) + provenance (every instance transform is the bit-exact `FxBody` state) + visual parity, with the cross-vendor delta the documented float baseline. **Proof:** `{bodies, settled, shaded}` provenance, two-render byte-identical determinism, a coherent lit pile, and an empty-world no-op. Golden `fpx_render` (the lit 3D pile). **Completes flagship #6.**

### Deterministic GPU navmesh + pathfinding (`engine/nav/navmesh.h`, namespace `hf::nav`)

Slices NAV1–NAV6 build a **Recast/Detour-style navmesh + pathfinder** — the AI pillar mainstream engines ship and this engine lacked (a total greenfield). It is built almost entirely in **host-snapped integer** arithmetic, so every build stage is BIT-IDENTICAL across Vulkan/Metal (the strict zero-differing-pixel bar, like the marching-cubes mesher and the fixed-point broadphase), and — paired with the FPX5 lockstep determinism — yields **bit-identical cross-platform replayable pathfinding**, which mainstream float pathfinding cannot guarantee. `engine/sim/fpx.h` (float physics aside, `FloorDiv`/`FxISqrt` reused read-only) and `engine/physics/` are untouched; the whole flagship adds **no new RHI** and stays int32-native (Metal-MSL) except the deferred int64 distance/cost stages. The pipeline is the engine's home turf — the `mc.h`/`fpx.h` **count→scan→emit** compaction and `[numthreads(1,1,1)]` **single-thread serial** mirrors, stage after stage.

- **Integer heightfield span rasterization (Slice NAV1, `shaders/nav_raster_{count,scan,emit}.comp.hlsl`).** Input world triangles are host-snapped to an int32 voxel grid and rasterized, per voxel column, into solid `Span{ymin,ymax,area}` lists — the count→scan→emit compaction (one thread per column counts covering triangles via an integer 2D edge-function sign test, a single-thread exclusive prefix-sum reserves per-column ranges, one thread per column emits each covering triangle's integer y-span). All int32 → MSL-native. `RasterizeTriangleSpans` is the CPU reference the GPU buffer is `memcmp`'d against. The inclusive edge test double-covers shared triangle edges (deterministic, merged away by the column-local `MergeColumnSpans` finalize — the standard Recast rasterize-then-merge). **Proof:** GPU `{colCount,colOffset,spans}` == the CPU reference byte-for-byte; two runs identical. Golden `nav_spans` (the heightfield debug viz, CPU-colored from the integer read-back → strict zero-diff cross-backend).
- **Walkable filter + integer distance field (Slice NAV2, `shaders/nav_filter.comp.hlsl` + `nav_distance.comp.hlsl`).** `FilterWalkableSpans` marks each span's `area` walkable (clearance-above ≥ agent height, integer compares) and derives a per-column walkable mask + surface height; `IsConnected` is the 4-neighbour max-step (`|Δsurface| ≤ climb`) predicate. `BuildDistanceField` is an integer two-pass chamfer (cardinal weight 2, diagonal 3 — no `sqrt`, no int64) seeded from non-walkable/border cells, traversing only walkable+connected neighbours so the distance is **geodesic** over the walkable surface. The chamfer's two sweeps are sequential → `[numthreads(1,1,1)]` single-thread (bit-exact); the filter is per-column (one thread each). **Proof:** GPU walkable[] + dist[] == the CPU reference byte-for-byte; obstacles (a different surface height, not connected) become distance seeds. Golden `nav_distfield` (the heat-ramp distance field, strict zero-diff).
- **Watershed region generation (Slice NAV3, `shaders/nav_region.comp.hlsl`).** `BuildRegions` partitions the walkable distance field into connected basins by an integer **watershed**: a level-descending flood (from `maxDist` down to 1) that GROWs existing regions into each level's cells (fixed-point expansion, ascending-cellId scan, lowest-region-id tie-break, `IsConnected`-gated) then SEEDs new regions for the rest (ascending-cellId order). The make-or-break determinism is by construction: every ordering decision is pinned and the whole watershed runs `[numthreads(1,1,1)]` single-thread, so the irregular flood wavefront is bit-exact CPU↔GPU↔both backends (a signed level counter — the unsigned `down to 1` would wrap). **Proof:** GPU region[] == the CPU reference byte-for-byte; ≥2 connected regions partition the walkable cells (the obstacles pinch the ground into basins); two runs identical. Golden `nav_regions` (the hash-colored region partition, strict zero-diff).
- **Contour tracing + integer polygonization (Slice NAV4, `shaders/nav_contour.comp.hlsl` + `nav_polygonize.comp.hlsl`).** `TraceContours` walks each region's boundary into a closed integer loop (lowest-cellId start, pinned clockwise turn order); `SimplifyContour` is an integer Douglas–Peucker (a perpendicular-distance-squared test, cross-multiplied — no division, no sqrt); `BuildPolyMesh` ear-clips each simplified contour into convex polygons (clip the lowest-index valid ear via the integer cross-product orientation + point-in-triangle tests) with per-edge cross-poly adjacency (the A* graph). Three sequential algorithms, each `[numthreads(1,1,1)]` single-thread with every order pinned → bit-exact. **Proof:** GPU contour verts + poly indices + adjacency == the CPU reference byte-for-byte; closed contours, consistent adjacency. Golden `nav_polymesh` (the contours + triangulated polys, strict zero-diff). (Simplifications, documented: one walkable surface per column; outer-contour-only — interior region holes are not carved; triangles are the convex polys.)
- **Integer A* pathfinding (Slice NAV5, `shaders/nav_astar.comp.hlsl`).** `FindPath` runs A* over the NAV4 poly adjacency graph — the flagship's headline: a fully INTEGER cost graph (edge cost = heuristic = Manhattan distance between integer poly centroids, a true metric → admissible + consistent → optimal, no sqrt/int64) with a deterministic integer frontier (pop lowest `f=g+h`, tie → lowest poly id) → a poly corridor. Deterministic start/goal (largest connected component, lowest-id start, farthest-Manhattan goal). `[numthreads(1,1,1)]` single-thread → bit-exact. This is the **deterministic, replayable, cross-platform-bit-identical pathfinding** that pairs with FPX5 lockstep — mainstream float pathfinding cannot guarantee it bit-for-bit. **Proof:** GPU corridor + cost == the CPU reference byte-for-byte; the corridor is an adjacency-connected path start→goal; two runs identical. Golden `nav_path` (the corridor over the navmesh, strict zero-diff). (Per-region components — inter-region portals are a documented future refinement.)
- **Lit 3D render (Slice NAV6, render-only — `NavVertToWorld`/`PolyMeshToRenderMesh`/`PathToWorldPolyline`).** The capstone: the bit-exact integer navmesh + A* corridor rendered as a lit 3D scene — a lit ground, the walkable polygons as a translucent per-region overlay (the existing transparent/glass pass), and the corridor as a bright always-on-top debug line with start/goal markers — through the **existing lit + transparent + debug-line pipelines reused verbatim** (no new shader, no new RHI). The integer voxel coords cross to float once host-side (`coord/(float)scale`), render-only; the navmesh build/pathfind stay integer. This is the arc's one **float** slice (the visresolve-bar): the golden is Metal-baked, the bar is Metal-determinism (two-run DIFF 0.0000) + provenance (the geometry is the bit-exact NAV1–NAV5 navmesh) + visual parity, with the cross-vendor delta the documented float baseline. **Proof:** `{regions, polys, corridor, shaded}` provenance, two-render determinism, a coherent lit navmesh + path, an empty no-op. Golden `nav_render` (the lit 3D navmesh + corridor). **Completes flagship #7.**

### Deterministic GPU cloth (`engine/sim/cloth.h`, namespace `hf::sim::cloth`)

Slices CL1–CL6 add a **Position-Based-Dynamics cloth solver in Q16.16 fixed-point** — BIT-IDENTICAL across Vulkan/Metal/compilers AND lockstep/rollback-replayable, the first deterministic deformable simulation no major engine ships (mainstream cloth systems are float and non-deterministic). It is a near-verbatim **FPX twin**: cloth is `engine/sim/fpx.h`'s Q16.16 rigid-body toolbox generalized from sphere-pairs to a particle lattice with a distance-constraint graph, so it directly extends the FPX5 lockstep/rollback determinism headline from rigid to deformable bodies. `engine/sim/fpx.h` (reused read-only) and `engine/physics/` are untouched; the whole flagship adds **no new RHI**, with the same int64/glslc split FPX proved (the int64 integrate/solve shaders are Vulkan-only + Metal runs the byte-identical CPU reference; the int32 graph-build shaders are MSL-native).

- **Q16.16 integrator + grid build (Slice CL1, `shaders/cloth_integrate.comp.hlsl`).** A `ClothParticle{pos, prev, vel (FxVec3); invMass; flags}` (std430 int32, the `FxBody` discipline; flags bit0 = PINNED) over a fixed W×H lattice; `IntegrateParticles` is semi-implicit Euler under gravity (copied verbatim from `fpx::IntegrateStep`, one thread per particle, independent → bit-exact). int64 `fxmul` (gravity·dt overflows int32, the FPX1 reason) → the shader is Vulkan-only + the Metal showcase runs the CPU integrate. **Proof:** GPU particles == the CPU reference byte-for-byte after K steps; pinned corners held; two runs identical. Golden `cloth_integrate` (the free-falling lattice, strict zero-diff).
- **Distance-constraint graph (Slice CL2, `shaders/cloth_edge_{count,scan,emit}.comp.hlsl`).** `BuildConstraints` enumerates the cloth's `Constraint{i, j, restLen, kind}` edges — structural (axis neighbours), shear (cell diagonals), bend (2-away) — each owned by its lower linear index, `restLen` the host-snapped integer rest distance. Laid out by the count→scan→emit compaction (the FPX2/MC3 pattern), **pure int32 → MSL-native** (a true GPU pass on both backends). **Proof:** GPU `{counts, offsets, edges}` == the CPU reference byte-for-byte; counts match the lattice formula; two runs identical. Golden `cloth_edges` (the constraint graph, struct/shear/bend color-coded, strict zero-diff).
- **PBD constraint solver (Slice CL3, `shaders/cloth_solve.comp.hlsl`).** `SolveDistanceConstraint` projects each edge — `pen = FxLength(pos[j]-pos[i]) - restLen; n = FxNormalize(d);` move the endpoints together/apart by their inverse-mass-weighted share (`fxdiv`) of `pen` — the **verbatim generalization of `fpx::SolveContact` from a sphere-pair to a graph edge**. `StepCloth` integrates then runs K Gauss-Seidel passes over the edges in a fixed order; pinned particles (invMass 0) never move, the sheet drapes. Gauss-Seidel is order-dependent → the GPU pass is `[numthreads(1,1,1)]` single-thread (bit-exact), int64 → Vulkan-only + Metal runs the CPU `StepCloth`. **Proof (the make-or-break):** GPU particles after K integrate+solve steps == the CPU reference byte-for-byte; the per-edge residual is deterministic (cohesive drape, ≈0.025/edge — PBD is iterative so the residual is deterministic but non-zero, the FPX3 caveat); a hand-checked single constraint splits by the exact Q16.16 inverse-mass share; two runs identical. Golden `cloth_solve` (the draped lattice, strict zero-diff). (A single-thread serial dispatch has a GPU-watchdog/TDR duration ceiling, so the showcase sizes its iteration count under it.)
- **Integer collision (Slice CL4, `shaders/cloth_collide.comp.hlsl`).** `CollideSpheres` projects each particle out of a set of `fpx::FxBody` sphere colliders (and the ground plane) — if it is inside a sphere, snap it to the surface along the outward normal — the same positional projection as the solve. The cloth and the rigid sphere are the **same Q16.16 world units** (`SphereFromBody(fpx::FxBody)`), so this is the engine's first deformable-meets-rigid integer interaction: the cloth drapes over the sphere. Folded into `StepClothCollide` after the constraint passes; int64 → Vulkan-only + Metal runs the CPU reference. **Proof:** GPU == the CPU reference byte-for-byte; no particle ends inside the sphere (0 penetrating, within an LSB epsilon — the projection is deterministic but not analytically perfect, the FPX3 caveat); the cloth visibly drapes over the sphere; two runs identical. Golden `cloth_collide` (the cloth draped over the sphere, strict zero-diff).
- **Lockstep + rollback (Slice CL5, pure CPU — `ClothCommand`/`SimClothTick`/`SnapshotCloth`/`RestoreCloth`/`RunClothLockstep`/`RunClothRollback`).** The arc's headline, and the direct analog of the FPX5 netcode harness. A peer fed the **input command stream alone** (wind/pin commands, no full-state sync) re-derives the authority's exact cloth state — `RunClothLockstep`'s `replica == authority` is bit-for-bit. `RunClothRollback` saves a snapshot, speculatively advances with a *wrong* predicted input, then `RestoreCloth` + re-simulates with the corrected input to the exact authoritative state (with the mispredicted path differing first). No new shader/RHI (a CPU harness over the bit-exact cloth step). **Proof:** the 16×16 / 24-tick scenario re-simulates byte-identically on Vulkan-Windows and Metal-Mac (the cross-platform pixel compare is zero-differing-pixel); snapshot round-trip and determinism. Golden `cloth_lockstep` (the converged state). **This is the flagship's beyond-mainstream result: deterministic, rollback-able, bit-identical-cross-platform *deformable*-body cloth — extending the FPX5 rigid-body headline to cloth, which mainstream float cloth systems cannot provide.**
- **Lit 3D render (Slice CL6, render-only — `ClothVertToWorld`/`ClothToRenderMesh`).** The capstone: the bit-exact draped cloth rendered as a lit 3D surface. The W×H integer particle lattice becomes a float triangle mesh (a vertex per particle at `pos/kOne`, two triangles per cell quad, per-vertex normals averaged from the adjacent faces — the single host float crossing, render-only), rendered through the **existing lit-mesh pipeline reused verbatim** from `--mc-render-shot` (`lit.vert`+`lit.frag`, `scene::MeshVertexLayout`, the frame UBO, sky + shadow + post) — no new shader, no new RHI. This is the arc's one **float** slice (the visresolve-bar): the golden is Metal-baked, the bar is Metal-determinism (two-run DIFF 0.0000) + provenance (the mesh is the bit-exact CL1–CL5 cloth) + visual parity, with the cross-vendor delta the documented float baseline. **Proof:** `{particles, tris, shaded}` provenance, two-render determinism, a coherent lit drape, an empty no-op. Golden `cloth_render` (the lit 3D cloth draped over the sphere). **Completes flagship #8.**

### Deterministic GPU fluid (`engine/sim/fluid.h`, namespace `hf::sim::fluid`)

Slices FL1–FL6 add a **Position-Based-Fluids solver in Q16.16 fixed-point** — BIT-IDENTICAL across Vulkan/Metal/compilers AND lockstep/rollback-replayable, completing the deterministic-sim trilogy (rigid `fpx` → cloth → fluid). It is the **density-constraint twin of the cloth solver**: PBF replaces SPH's divide-heavy pressure EOS with a unilateral density constraint `C_i = ρ_i/ρ0 − 1` solved by the same Gauss-Seidel position-projection the cloth uses, so the whole flagship reuses the `fpx.h` Q16.16 substrate, the FPX2 grid-hash, and the cloth CL1–CL6 mold. mainstream float particle fluids are non-deterministic — a fixed-point bit-exact fluid is a genuine first. `engine/sim/fpx.h`/`cloth.h` (reused read-only) and `engine/physics/` are untouched; **no new RHI** across the flagship.

- **Q16.16 particle integrator (Slice FL1, `shaders/fluid_integrate.comp.hlsl`).** A `FluidParticle{pos, prev, vel, invMass, flags}` pool (the `ClothParticle` packing); `InitBlock` is a dam-break particle block; `IntegrateFluid` is semi-implicit Euler under gravity (copied from `fpx::IntegrateStep`, one thread per particle, independent → bit-exact). int64 `fxmul` → Vulkan-only + the Metal showcase runs the CPU integrate. **Proof:** GPU particles == the CPU reference byte-for-byte after K steps; two runs identical. Golden `fluid_integrate` (the falling/settling block, strict zero-diff).
- **Grid-hash neighbor search (Slice FL2, `shaders/fluid_cell_{count,scan,emit}.comp.hlsl` + `fluid_neighbor_{count,scan,emit}.comp.hlsl`).** A uniform spatial-hash grid at cell-size `h` (the PBF radius): `BuildCellTable` buckets particles into cells (CSR, count→scan→emit), `BuildNeighborList` gathers each particle's candidate neighbours from the 27-cell stencil with a **pure-int32 per-axis `|dx| < h` reject** (the exact radial cull is deferred to FL3's kernel, which is 0 beyond `h`). All int32 → MSL-native (a true GPU pass on both backends, the FPX2/CL2 precedent). **Proof:** GPU cell-table + neighbor-list == the CPU reference byte-for-byte; two runs identical. Golden `fluid_neighbors` (the per-particle neighbor-count heat viz, strict zero-diff).
- **Density + λ (Slice FL3, `shaders/fluid_density.comp.hlsl` + `fluid_lambda.comp.hlsl`).** The PBF make-or-break, de-risked by a **host-snapped kernel LUT**: `BuildKernelTable` precomputes the poly6 `W(r,h)` and spiky `∇W(r,h)` over `r²` bins as Q16.16 constants (the one build-time sqrt — monotone/non-negative by construction, so the in-shader gather has no divide and no monotonicity hazard). `ComputeDensity` gathers `ρ_i = Σ_j W(r_ij)` over the FL2 neighbours; `ComputeLambda` computes the scaling factor `λ_i = −C_i/(Σ|∇C|² + ε)` (the one `fxdiv`). The squared distance and the divide are int64 → Vulkan-only + Metal runs the CPU reference; per-particle independent → multi-thread (no single-thread TDR). **Proof:** GPU `ρ`/`λ` == the CPU reference byte-for-byte; the density is coherent (interior dense, surface sparse, `ρ̄ ≈ ρ0`); two runs identical. Golden `fluid_density` (the density heat field, strict zero-diff). (The binned LUT is a deterministic step-function fidelity simplification, the FPX3/cloth caveat shape.)
- **PBF density-constraint solve (Slice FL4, `shaders/fluid_dp.comp.hlsl` + `fluid_apply.comp.hlsl` + `fluid_collide.comp.hlsl`).** The Macklin–Müller position-based-fluids solve: each particle gathers its neighbours' λ to a position correction `Δp_i = (1/ρ0) Σ_j (λ_i + λ_j) ∇W(r_ij)`, the correction is applied, then particles are projected out of the ground/`fpx::FxBody` sphere colliders (the cloth CL4 collision, reused). Unlike the cloth's order-dependent Gauss-Seidel, **PBF is Jacobi — every particle's Δp reads the frame's λ and is independent → the shaders are `[numthreads(64,1,1)]` multi-thread with NO single-thread TDR/GPU-watchdog ceiling** (the CL3 ceiling does not apply). int64 (the Δp accumulate + the divide) → Vulkan-only + Metal runs the CPU reference. **Proof:** GPU particles after K solve iterations == the CPU reference byte-for-byte; the peak over-compression is relieved (the constraint does work — `peakSolved < peakFree`, the documented open-ground PBF residual caveat: the summed |ρ−ρ0| can grow as the block spreads with no container, so peak-compression is the honest metric); no particle ends inside the sphere; two runs identical. Golden `fluid_solve` (the settled/spread fluid, strict zero-diff).
- **Lockstep + rollback (Slice FL5, pure CPU — `FluidCommand`/`SimFluidTick`/`SnapshotFluid`/`RestoreFluid`/`RunFluidLockstep`/`RunFluidRollback`).** The trilogy's third netcode headline (after FPX5 rigid + CL5 cloth): a peer fed the **input command stream alone** re-derives the authority's exact fluid state (`RunFluidLockstep`'s `replica == authority` bit-for-bit); `RunFluidRollback` snapshots, speculatively advances with a wrong predicted input, then `RestoreFluid` + re-simulates to the exact authoritative state. A CPU harness over the bit-exact fluid step — no new shader/RHI. **Proof:** the scenario re-simulates byte-identically on Vulkan-Windows and Metal-Mac (zero-differing-pixel cross-platform); snapshot round-trip + determinism. Golden `fluid_lockstep` (the converged state). **The beyond-mainstream result: a deterministic, rollback-able, bit-identical-cross-platform *fluid* — mainstream float particle systems cannot provide this. Completes the deterministic-sim trilogy (rigid → cloth → fluid).**
- **Lit 3D render (Slice FL6, render-only — `FluidVertToWorld`/`FluidToRenderInstances`).** The capstone: the bit-exact fluid rendered as lit 3D instanced spheres (a droplet per particle at `pos/kOne`, scaled by the droplet radius — the single host float crossing, render-only), through the **existing instanced-lit-sphere pipeline reused verbatim** from `--fpx-render-shot` (`lit_instanced.vert`+`lit.frag`, the per-instance transform buffer, the frame UBO, sky + shadow + post) — no new shader, no new RHI. The arc's one **float** slice (the visresolve-bar): Metal-baked golden, the bar is Metal-determinism (two-run DIFF 0.0000) + provenance (the instances are the bit-exact FL1–FL5 fluid) + visual parity, the cross-vendor delta the documented float baseline. **Proof:** `{particles, instances, shaded}` provenance, two-render determinism, a coherent lit droplet mass, an empty no-op. Golden `fluid_render` (the lit 3D droplet pool over the collider sphere). **Completes flagship #9 — the ninth flagship.**

### Deterministic GPU granular/sand (`engine/sim/grain.h`, namespace `hf::sim::grain`)

Slices GR1–GR6 add a **Position-Based granular (sand/soil) solver in Q16.16 fixed-point** — the **4th member of the deterministic-sim family** (rigid `fpx` → cloth → fluid → grain), adding the one physics the trilogy never modeled: **dry Coulomb friction / shear (the angle-of-repose)** — what makes sand pile into a self-supporting cone instead of spreading flat. Bit-identical CPU↔Vulkan↔Metal AND lockstep/rollback-replayable; mainstream engines ship no deterministic granular solver (float soil/RBD piles are non-deterministic), so a fixed-point bit-exact sand pile is a genuine first. It is the **frictional twin of the fluid solver**: it reuses the FL2 grid-hash neighbour search, the FL4 **Jacobi multi-thread** position solve, the `fpx.h` Q16.16 substrate, and the FPX6/FL6 instanced-lit render — adding the tangential-friction primitive at GR4. `engine/sim/fpx.h`/`cloth.h`/`fluid.h` (reused read-only) and `engine/physics/` are untouched; **no new RHI** across the flagship — the TENTH flagship, complete (6/6).

- **Q16.16 grain integrator (Slice GR1, `shaders/grain_integrate.comp.hlsl`).** A `GrainParticle{pos, prev, vel, invMass, radius, flags}` pool (the `FluidParticle` packing **plus a first-class `radius` field** → 48-byte std430); `InitGrainBlock` is a dropped grain block; `IntegrateGrains` is semi-implicit Euler under gravity with a **radius-aware ground rest** (the grain's surface rests on the floor: `pos.y < groundY + radius`), one thread per grain, independent → bit-exact. int64 `fxmul` → Vulkan-only + the Metal showcase runs the CPU integrate. **Proof:** GPU == the CPU reference byte-for-byte after K steps; two runs identical. Golden `grain_integrate` (the settling grain block, strict zero-diff). (Without inter-grain contact a fully-settled block collapses to one floor plane, so the golden is a mid-fall state — contact arrives at GR3.)
- **Grid-hash neighbor search (Slice GR2, `shaders/grain_cell_{count,scan,emit}.comp.hlsl` + `grain_neighbor_{count,scan,emit}.comp.hlsl`).** The candidate contact-pair list over the grain pool via a bounded dense grid at cell-size `hSearch` (the contact search radius, `≥` the contact diameter): `BuildGrainCellTable` buckets grains (CSR, count→scan→emit), `BuildGrainNeighborList` gathers the 27-cell stencil with a **pure-int32 per-axis `|dx| < hSearch` reject** (the exact radial overlap cull deferred to GR3's solve). All int32 → **MSL-native (a true GPU pass on both backends** — the FL2/FPX2/CL2 precedent, the flagship's strongest cross-vendor proof). **Proof:** GPU cell-table + neighbor-list == the CPU reference byte-for-byte; two runs identical. Golden `grain_neighbors` (the per-grain neighbor-count heat viz, strict zero-diff).
- **Frictionless contact projection (Slice GR3, `shaders/grain_contact_dp.comp.hlsl` + `grain_contact_apply.comp.hlsl` + `grain_collide.comp.hlsl`).** Resolves grain-grain overlap + ground/`fpx::FxBody` colliders by an inverse-mass-weighted **normal** push-apart: each grain accumulates `Δp_i = Σ_j (w_i/(w_i+w_j))·pen·n̂_ij` over its GR2 neighbours with `pen = (r_i+r_j) − |p_i−p_j| > 0` (the FL4 density-solve structure with a contact term, the cloth distance-constraint per pair). Like FL4 it is **Jacobi — every grain reads the iteration-start positions into a separate `dp[]` → `[numthreads(64,1,1)]` multi-thread with NO single-thread TDR ceiling** (the CL3 limit does not apply — the decisive FL4 reuse). The radius-aware colliders snap the grain surface to the floor / sphere surface. int64 → the 3 contact shaders Vulkan-only + Metal runs the CPU `StepGrainContact` (the GR2 neighbour passes, rebuilt each step, stay MSL-native). **Proof:** GPU == the CPU reference byte-for-byte after K steps; the overlap is relieved (`penAfter < penBefore` — Jacobi is iterative so the residual penetration is deterministic-but-nonzero, the honest non-penetration metric, the FL4 caveat shape); no grain ends below the floor or inside a collider; two runs identical. Golden `grain_contact` (a **loose frictionless pile that spreads wide/flat** — GR4's friction is what makes it a self-supporting cone, strict zero-diff).
- **Tangential Coulomb friction (Slice GR4, `shaders/grain_friction.comp.hlsl` — the new physics).** The one physics the deterministic-sim trilogy never modeled. After the GR3 normal correction, project out the inverse-mass-weighted **tangential** relative displacement, clamped to the Coulomb cone `μ·pen`: `Δx_t = Δx_rel − (Δx_rel·n̂)n̂` (with `Δx_rel = (p_i−prev_i) − (p_j−prev_j)`), `static` (|Δx_t| ≤ μ·pen) cancels all slip so a grain stays on a slope, `kinetic` clamps the slip to the cone — the standard Unified-Particle/PBD friction. So a poured pile holds a **self-supporting angle-of-repose cone with no container**. A second Jacobi sub-pass per iteration (normal then friction), `[numthreads(64,1,1)]`, int64 → Vulkan-only + Metal CPU (reuses `grain_contact_apply`/`grain_collide`; the GR2 passes stay MSL-native). **Proof:** GPU == the CPU reference byte-for-byte; the pile holds a slope (`MeasureGrainRepose` → a deterministic, two-run-identical, within-μ-band slope — the emergent angle of repose is iterative so it is a within-band metric, NOT an exact degree, the FL4 caveat shape); a `μ = 0` control equals the GR3 frictionless spread (proving friction does work — the slope contrast was 2.5×). Golden `grain_friction` (a self-supporting heap holding a slope, strict zero-diff). (A perfectly axis-aligned lattice collapses radially with zero tangential slip — a small deterministic stagger breaks the symmetry so friction engages.)
- **Lockstep + rollback (Slice GR5, pure CPU — `GrainCommand`/`SimGrainTick`/`SnapshotGrain`/`RestoreGrain`/`RunGrainLockstep`/`RunGrainRollback`).** The trilogy's **4th netcode headline** (rigid FPX5 → cloth CL5 → fluid FL5 → grain GR5): a peer fed the **input command stream alone** re-derives the authority's exact grain state (with friction) bit-for-bit; `RunGrainRollback` snapshots, mispredicts, restores, re-simulates to the exact authoritative state. A CPU harness over the bit-exact `StepGrainFriction` — no new shader/RHI. **Proof:** the scenario re-simulates byte-identically on Vulkan-Windows and Metal-Mac (zero-differing-pixel); rollback corrected a mispredict to the authority exactly; snapshot round-trip + determinism; the command stream displaced the pile non-trivially. Golden `grain_lockstep` (the converged state). **The beyond-mainstream result: a deterministic, rollback-able, bit-identical-cross-platform *granular* sim — mainstream engines have no deterministic granular, let alone replayable.**

Golden `boids_neighbors` (a neighbor-count heat field).
- **The full flock step (Slice BD3, `shaders/boids_flock.comp` — separation + alignment + cohesion).** Over the BD2 neighbor list, accumulate the three Reynolds rules — separation (away-sum), alignment (toward the neighbors' mean velocity), cohesion (toward their mean position) — and integrate. **The means are integer divides by the neighbor count** (the vendor-identical `/` truncation — the GPU==CPU crux). Golden `boids_flock`.

Golden `boids_path` (the crowd streaming start→goal along the corridor). This pairs the engine's two integer pillars — nav A* + the crowd sim.
- **Lockstep + rollback (Slice BD5, pure CPU — `RunBoidsLockstep`/`RunBoidsRollback`).** The FPX5/VH5/AC5 twin over `StepFlockPath` with an input **perturbation stream** (`BoidsCommand` — a velocity kick). **The BD simplification: the snapshot is just the agent world** — `SteerPath` is stateless, the grid is rebuilt each tick, and the corridor is a const input, so there is *no mutable extra* (unlike VH5's hinge axes / AC5's tick — the stateless steering pays off). No GPU/shader/RHI. **Proof:** authority == replica bit-identical; rollback corrected the mispredict exactly; a real divergence before rollback; two-run determinism. Golden `boids_lockstep`. **A 256-agent crowd that's rollback-netcode-safe — which mainstream float crowd systems cannot provide.**

Golden `boids_render` (the lit 3D crowd). **Completes flagship #18 — the eighteenth flagship.**

### Deterministic convex rigid-body contacts (`engine/sim/convex.h`, namespace `hf::sim::convex`)

Slices CX1–CX6 add the **first bit-identical, lockstep-replayable convex rigid-body contact solver**: boxes that tumble, stack, and interlock byte-for-byte across CPU/Vulkan/Metal — the one guarantee mainstream float physics cannot make. It is the **deepest single moat-extension** of the sim suite: every prior rigid sim shares one caveat — *contacts are sphere-sphere with no inertia tensor and no torque-from-contact* (repeated verbatim in seven headers). CX lifts that ceiling. `convex.h` is an **additive sibling** that `#include`s `fpx.h` read-only and reuses its Q16.16 toolbox, broadphase, Gauss-Seidel solver shape, and quaternion math; it **demonstrates** the upgraded contact model rather than retrofitting it (`fract`/`vehicle`/`joint` stay byte-frozen — wiring it back is a future composition). **No new RHI across the whole flagship**; the new compute shaders are int64 → Vulkan-GPU + a byte-identical Metal CPU reference (the FPX3 split), and the render capstone reuses the instanced-lit pipeline verbatim — the NINETEENTH flagship, complete (6/6).

- **The box-box SAT overlap test (Slice CX1, `shaders/convex_sat.comp.hlsl`).** A deterministic integer **Separating-Axis Test** over a pair of oriented boxes (`FxBody` pose + `FxBox` half-extents): 15 candidate axes in a fixed order — A's 3 world face normals, B's 3, and the 9 edge-edge crosses `FxCross(axA_i, axB_j)` — each normalized (a near-zero edge cross is skipped deterministically), the projected-radius overlap test computed, and the minimum-penetration axis + depth (or separation) returned with a strict-`<` lowest-index tie-break. Introduces `FxMat3` (the 3×3 inertia-tensor type, used in CX3), `FxDot`/`FxCross` (the `FxRotate` internal cross promoted), and `FxBox`. int64 (world-scale products) → Vulkan-only + Metal CPU reference. **Proof:** GPU `SatResult[]` == CPU `BoxSat` byte-for-byte (12 pairs / 8 overlapping); the overlap classification matches a known-truth brute force; a far-apart pair separates. Golden `convex_sat` (a top-down SAT diagnostic). **Caveat:** boxes only (15-axis box-box SAT, not general convex/GJK/EPA).
- **The contact manifold (Slice CX2, `shaders/convex_manifold.comp.hlsl`).** Turns the SAT axis into the actual 1–4 contact **points**. The *face* case (axis 0–5) does reference/incident-face **Sutherland-Hodgman** clipping — clip the incident box's face against the reference face's four side planes (fixed corner / plane / iteration order), keep the penetrating vertices, reduce to ≤4 (deepest + clip-order). The *edge-edge* case (axis 6–14) is a single closest-points-between-segments contact. int64 → Vulkan + Metal CPU ref. **Proof:** GPU manifold == CPU byte-for-byte; every overlapping pair yields ≥1 point, every separated pair 0, every contact point inside both boxes' slabs, every depth ≥0; a deep face-face pair yields a 4-point patch. Golden `convex_manifold`. **Caveat:** the 4-point cap is deepest + clip-order, not the area-maximizing reduction.
- **The angular contact impulse — the new physics (Slice CX3, `shaders/convex_solve.comp.hlsl`).** The first time `angVel` is driven by a contact in the whole engine. The inverse-mass + **inertia-tensor** velocity impulse over the manifold: the box body-space diagonal inverse inertia rotated to world (`WorldInvInertia` = R·diag·Rᵀ — `FxMat3`'s first real use), then a Gauss-Seidel sweep applying `jn = −(1+e)·vₙ / (invMₐ + invM_b + n·((I⁻¹ₐ(rₐ×n))×rₐ) + n·((I⁻¹_b(r_b×n))×r_b))` to **both** linear and angular velocity (`ω += I⁻¹(r×J)`). The manifold normal is sign-corrected A→B once. Velocity-only (position correction is CX4). int64 → Vulkan + Metal CPU ref. **Proof:** GPU resolved bodies == CPU byte-for-byte; an off-center contact imparts non-zero `angVel` (the box gains spin) and removes the approach; a dead-center zero-lever contact yields exactly zero spin. Golden `convex_tumble` (a box dropped on a corner visibly tumbles). **Caveat:** diagonal inertia, non-accumulated Gauss-Seidel (residual not analytic).
- **The full convex step — a settling stack (Slice CX4, `shaders/convex_step.comp.hlsl`).** The host multi-pass per-tick driver assembling CX1–CX3 plus the one piece they lacked — **position de-penetration** (split-by-inverse-mass push-out, so a stack rests instead of sinking): per tick, predict-integrate every dynamic body → all-pairs narrowphase (fixed `i<j` order) → world-level Gauss-Seidel impulse → position de-penetration sweeps. A static floor box + three dynamic boxes settle into a coherent resting tower. Two CX4-level robustness additions (additive, CX1–CX3 frozen): `BoxSatStable` (a face-preference post-filter — a resting stack's edge-cross axis ties the clean face axis by ~1 LSB and its drifting normal walks the stack apart, so prefer a face axis within `kFacePrefEps`; the *favor-face-contacts* technique real engines use), and opt-in `linDamp`/`angDamp` resting-body damping (default off) to bleed the fixed-point Gauss-Seidel residual torque (standard Box2D/Bullet/PhysX practice). int64 → Vulkan drives the chained step; Metal CPU `StepConvexWorldN` byte-identical. **Proof:** GPU final body world == CPU byte-for-byte over 300 ticks; the stack comes to rest, does not interpenetrate beyond slop, stays stacked; a single box rests on the floor (does not sink). Golden `convex_stack`. **Caveat:** all-pairs (small scene), frictionless-normal, linear position correction (stable within a band; a slow fixed-point drift tips the tower far past the showcase window).
- **Lockstep + rollback (Slice CX5, pure CPU — `RunConvexLockstep`/`RunConvexRollback`).** The FR5/BD5 twin over `StepConvexWorld` with a `ConvexCommand` input stream (impulse / set-angVel). The snapshot is just the bodies vector (the boxes are immutable). Two peers fed the input stream alone converge byte-identical; a mispredicted command rolls back from a snapshot and re-sims bit-for-bit. No GPU/shader/RHI — both backends run the identical harness, so the golden is bit-identical by construction. **Proof:** authority == replica bit-identical; rollback corrected == authority; a real divergence before rollback; two-run determinism. Golden `convex_lockstep`. **The cross-platform rollback-replayable convex-contact sim mainstream float physics cannot provide.**
- **Lit 3D instanced render (Slice CX6, render-only — the money-shot).** The capstone: `ConvexToRenderInstances` turns the bit-exact settled stack into a split float instance set — one **oriented cube** per body (`translate(FxToFloat(pos))·rotate(normalize(orient))·scale(2·halfExtents)` via `math::FromTRS`, the static floor vs the dynamic boxes) — drawn through the **existing cube-mesh `lit_instanced` pipeline reused verbatim** (the VH6 cube path) as a lit, shadowed 3D box tower. The per-instance orient makes the box rotation visible (unlike the prior sphere capstones). **Matte** warm-amber boxes on a cool floor to avoid the sky-IBL iridescence. The float visresolve-bar: Metal two-run DIFF 0.0000 + provenance (instances == bit-exact rebuild) + visual parity + an in-band cross-vendor mean (~20/channel). **Proof:** `{bodies, instances, dynamic}` provenance from the bit-exact stack, two-render determinism, a non-trivial shaded pixel count. Golden `convex_render`. **Completes flagship #19 — the nineteenth flagship.**

### Deterministic tangential contact friction (`engine/sim/fric.h`, namespace `hf::sim::fric`)

Slices FC1–FC6 add **Coulomb friction** to the convex box-box contacts — the static-and-kinetic friction cone that makes a box grip a slope, a stack stand without sliding apart, and a sliding box decelerate to rest, all **bit-identical across CPU/Vulkan/Metal and lockstep/rollback-replayable**. Friction is the single most numerically-sensitive part of a contact solver and the corner where mainstream float solvers diverge first machine-to-machine; locking it down deterministically is the natural completion of the convex arc and a genuine moat. It is almost entirely **assembly of frozen pieces**: the friction impulse slots into CX3's `SolveManifoldImpulse` shape (swap the normal for two tangents, clamp to `±μ·jn`), reuses the box inertia tensor (so friction produces torque), reuses the CX5 lockstep harness and the CX6 render bridge, and ports the Coulomb cone from the granular solver's `grain.h` (from position-projection to impulse form). `fric.h` is an **additive sibling** that `#include`s `convex.h` read-only; **no new RHI across the whole flagship** — the new compute shaders are int64 (Vulkan-GPU + a byte-identical Metal CPU reference, the established split), the render reuses the instanced-lit pipeline verbatim, and FC5 is pure CPU. The TWENTIETH flagship, complete (6/6).

- **The tangent basis (Slice FC1, `shaders/fric_basis.comp.hlsl`).** A deterministic, degeneracy-safe integer Gram-Schmidt: from a unit contact normal `n`, pick the least-aligned cardinal axis (smallest `|n.component|`, lowest-index tie-break — so the projection never collapses), project it off `n` and normalize to `t1`, then `t2 = FxCross(n, t1)`. `(t1, t2)` span the friction tangent plane, orthonormal up to fixed-point drift. int64 (`FxNormalize`/`FxCross`) → Vulkan-only + Metal CPU reference. **Proof:** GPU `TangentBasis[]` == CPU byte-for-byte (14 normals); every basis orthonormal within an integer epsilon; the axis-aligned normals (where a naive fixed axis would collapse) are handled. Golden `fric_basis`.
- **The friction-augmented manifold point (Slice FC2, `shaders/fric_points.comp.hlsl`).** For a box-box pair, take the frozen CX2 contact manifold and pair every contact point with its FC1 tangent basis + zeroed warm-start impulse accumulators, into a `FrictionPoint[]` — the per-contact solver state FC3 consumes. The normal is sign-corrected toward B once per pair (the `SolveManifoldImpulse` rule). int64 → Vulkan + Metal CPU ref. **Proof:** GPU `FrictionPoint[]` == CPU byte-for-byte (12 pairs / 8 with contact / 26 points, matching the CX2 manifold); every basis orthonormal; counts match the manifold; accumulators zero; normals point toward B. Golden `fric_points`.
- **The cone-clamped tangent-impulse solver — the new physics (Slice FC3, `shaders/fric_solve.comp.hlsl`).** The Coulomb cone in impulse form. Per Gauss-Seidel sweep, per contact point: first the normal impulse (the CX3 form reproduced verbatim, so `μ=0` is frictionless), then for each tangent `t1`, `t2` the friction impulse `jt = −v_t / k_t` (the same effective-mass form as the normal, using `t`), **clamped to the cone** `jt ∈ [−μ·jn, +μ·jn]`, applied to **both** linear and angular velocity through the inertia tensor (so contact drag produces torque). The contact-point velocities are recomputed after each apply (sequential impulse). int64 → Vulkan + Metal CPU ref. **Proof:** GPU resolved bodies == CPU byte-for-byte; a high-μ sliding box has its tangential speed strictly reduced (slide arrested ~72%) and gains spin; a `μ=0` pair is byte-identical to the frictionless `ResolveContactPair` (the normal solve is provably unaltered). Golden `fric_solve`. **Caveat:** isotropic single-μ; the cone clamps against the current-sweep `jn` (the standard sequential-impulse coupled-iteration approximation).
- **The friction-locked world step (Slice FC4, `shaders/fric_step.comp.hlsl`).** The money-physics beat: `StepFrictionWorld` reproduces the CX4 5-pass per-tick shell exactly (predict-integrate → all-pairs narrowphase → world Gauss-Seidel impulse → position de-penetration → orientation) with the **only** change being the FC3 friction solve swapped in for the normal-only impulse. A box released on a tilted static ramp **grips** (high μ — its down-ramp displacement stays tiny) or **slides and decelerates to rest** (low μ); and the settling box stack stands with **angular damping turned off** (`angDamp = kOne`) — real friction physically holds the tower, **retiring the CX4 angular-damping stability aid**. int64 → Vulkan drives the single-thread world step; Metal CPU `StepFrictionWorldN` byte-identical. **Proof (two scenes):** GPU final body world == CPU byte-for-byte over 240 ticks; the high-μ box gripped, the stack rested + stayed stacked + non-interpenetrating at `angDamp = kOne`, the low-μ box slid (a clear margin). Goldens `fric_ramp` + `fric_stack`. **Caveat:** boxes only, all-pairs small scene, linear position correction (stable within a band).
- **Lockstep + rollback (Slice FC5, pure CPU — `RunFricLockstep`/`RunFricRollback`).** The friction world *is* a `convex::ConvexWorld`, so FC5 reuses the CX5 command + snapshot machinery **verbatim** and swaps only the per-tick step to `StepFrictionWorld`. Two peers fed the input command stream alone converge byte-identical; a mispredicted impulse rolls back from a snapshot and re-sims bit-for-bit. No GPU/shader/RHI — both backends run the identical harness, the golden bit-identical by construction. **Proof:** authority == replica bit-identical; rollback corrected == authority; a real divergence before rollback; two-run determinism. Golden `fric_lockstep`. **Friction contacts that are lockstep-replayable — boxes that grip, slide, and stack with deterministic Coulomb friction, re-derived bit-for-bit on two machines from inputs alone, which mainstream float physics cannot provide.**
- **Lit 3D instanced render (Slice FC6, render-only — the money-shot).** The capstone: `FrictionToRenderInstances` delegates to the frozen `convex::ConvexToRenderInstances`, turning the bit-exact friction-settled world into the split float cube-instance set, drawn through the **existing cube-mesh `lit_instanced` pipeline reused verbatim** as a lit, shadowed 3D scene. The distinctive shot is a **matte warm-amber box gripped at rest on a tilted ramp** — held by friction on a slope, impossible in the frictionless solver. **Matte** material to avoid the sky-IBL iridescence. The float visresolve-bar: Metal two-run DIFF 0.0000 + provenance + visual parity + an in-band cross-vendor mean (~42/channel). **Proof:** `{bodies, instances, dynamic}` provenance from the bit-exact friction-settled world, two-render determinism, a non-trivial shaded pixel count. Golden `fric_render`. **Completes flagship #20 — the twentieth flagship.**

### Deterministic warm-started contact caching + sleeping islands (`engine/sim/persist.h`, namespace `hf::sim::persist`)

Slices PS1–PS6 add the **stabilization machinery** that turns the convex+friction box solver from a demo into a solver that converges tighter, rests harder, and scales — the persistent contact cache (warm-starting) and the sleeping-island state machine — **made fully deterministic and bit-identical across CPU/Vulkan/Metal and lockstep/rollback-replayable**. Every mainstream engine warm-starts (re-applies last tick's accumulated contact impulses so the solver converges in fewer iterations) and sleeps (parks low-energy bodies so a thousand-body pile costs nothing at rest), but their impulse caches and energy thresholds are **float and FPU-order-dependent**, so they diverge machine-to-machine — which is exactly why mainstream networked physics is never bit-deterministic. Making the *stabilization* itself deterministic is a genuine moat, and the highest-reuse one in the suite: it **activates the warm-start accumulators `fric.h` already carried** (previously zeroed every build) and reuses the friction step shell, the lockstep harness, and the render bridge wholesale. `persist.h` is an **additive sibling** that `#include`s `convex.h`/`fric.h` read-only; **no new RHI across the whole flagship** — the new physics compute shaders are int64 (Vulkan-GPU + a byte-identical Metal CPU reference, the established split), PS5 is pure CPU, and PS6 reuses the instanced-lit pipeline verbatim. The TWENTY-FIRST flagship, complete (6/6).

- **The contact feature key (Slice PS1, `shaders/persist_key.comp.hlsl`).** The primitive that makes a contact *identifiable across ticks*: an order-normalized integer key `(bodyA, bodyB, axisIndex, featureIndex)` per contact point (bodyA ≤ bodyB so the pair is symmetric), plus a pure-int32 avalanche hash. This is the rare **int32 MSL-native** shader (it runs as a true GPU pass on *both* backends, the strongest proof tier) because the key alone needs no 64-bit math. **Proof:** GPU keys == CPU byte-for-byte (12 pairs / 26 contacts / 26 distinct keys, matching the CX2 manifold); keys order-normalized + distinct contacts distinct; cross-vendor zero-differing-pixel. Golden `persist_key`.
- **The persistent manifold cache (Slice PS2, `shaders/persist_cache.comp.hlsl`).** The structural store: this tick's friction manifold points are matched against last tick's by contact key in fixed order — a matched point **inherits** its prior normal+tangent impulse accumulators, an unmatched point cold-starts at zero, a stale key is evicted deterministically. The cached manifold is built by the int64 friction path → int64 Vulkan-only + Metal CPU reference. **Proof:** GPU == CPU byte-for-byte (`{pairs:12, contacts:22, matched:18, coldStart:4}`); inherited-exact / fresh-cold-start / stale-evicted; store-then-match round-trip; cross-vendor zero. Golden `persist_cache`.
- **The warm-started cone solver (Slice PS3, `shaders/persist_warm.comp.hlsl`).** The solver that *uses* the cache: the accumulated sequential-impulse form (the proper warm-start) — each contact's impulse accumulator is seeded from the cache once, then the cone sweeps apply only **deltas** clamped on the running total (normal ≥ 0, tangent within `±μ·jₙ`). A warm-started step converges measurably tighter at a fixed low iteration count than a cold start. int64 → Vulkan + Metal CPU reference. **Proof:** GPU == CPU byte-for-byte over 240 ticks; the warm solve's residual speed (20422) beats the cold solve's (26942); warm and cold agree within an integer epsilon at convergence (an honest Q16.16-truncation caveat — the determinism that matters, GPU==CPU / two-run / cross-vendor, is bit-exact). Golden `persist_warm`.
- **Sleeping islands (Slice PS4, `shaders/persist_sleep.comp.hlsl` — the new physics).** The money-physics beat: a per-body integer kinetic-energy accumulator with fixed wake/sleep **hysteresis**, and a fixed-order contact-graph **island propagation** (union-find-free) so a resting tower sleeps and wakes **atomically as one island**. A warm-started tower settles, goes quiet, and **sleeps at exactly zero residual** (asleep bodies are frozen, not merely slow); a thrown box striking the pile raises its energy above the wake threshold and the whole island wakes deterministically and topples. int64 → Vulkan single-thread + Metal CPU `StepWarmSleepWorldN` byte-identical. **Proof (two scenes):** GPU == CPU byte-for-byte; the tower sleeps (asleep=3, max speed exactly 0), the thrown box wakes the island atomically; cross-vendor zero. Goldens `persist_sleep` + `persist_wake`. **Caveat:** velocity is frozen on sleep (standard practice), and the thresholds sit above the warm solver's steady-state jitter band — a deterministic heuristic, not a physical truth.
- **Lockstep + rollback (Slice PS5, pure CPU — `RunPersistLockstep`/`RunPersistRollback`).** The netcode headline. Unlike the earlier sims, the warm+sleep world's replayable state is a **triple** — the bodies, the persistent impulse cache, and the per-body sleep state — so `PersistSnapshot` captures all three (a peer that restored only the bodies would warm-start from the wrong impulses or mis-time the sleep and diverge). The frozen command machinery is reused verbatim; only the per-tick step swaps to the warm+sleep step. Two peers fed the input command stream alone warm-start, sleep, wake, and re-derive the entire triple byte-identical; a mispredicted impulse rolls back from a snapshot and re-sims bit-for-bit. No GPU/shader/RHI — both backends run the identical harness, the golden bit-identical by construction. **Proof:** authority == replica bit-identical across bodies+cache+sleep; rollback corrected == authority; a real divergence before rollback; two-run determinism; cross-vendor zero. Golden `persist_lockstep`. **Warm-started, sleeping contacts that are lockstep-replayable — boxes that settle, sleep, and wake, re-derived bit-for-bit on two machines from inputs alone, which mainstream float physics cannot provide.**
- **Lit 3D instanced render (Slice PS6, render-only — the money-shot).** The capstone: `PersistToRenderInstances` delegates to the frozen `convex::ConvexToRenderInstances`, turning the bit-exact warm+sleep converged world into the split float cube-instance set, drawn through the **existing cube-mesh `lit_instanced` pipeline reused verbatim** as a lit, shadowed 3D scene. The distinctive shot is a **matte warm-amber tower toppled from dead rest** — the visible payoff of the warm+sleep moat: the stack settles, sleeps, and is toppled by a wake-impulse, the tilted slabs frozen mid-topple. **Matte** material to avoid the sky-IBL iridescence. The float visresolve-bar: Metal two-run DIFF 0.0000 + provenance (two render calls byte-equal) + an in-band cross-vendor mean (~27/channel — the integer-determinism headline is the earlier slices', this one render is the single float crossing). **Proof:** `{floor:1, boxes:3}` provenance, `{dynamic:3, toppled:true}`, two-render determinism, a non-trivial shaded pixel count. Golden `persist_render`. **Completes flagship #21 — the twenty-first flagship.**

### Deterministic general convex-hull contacts — integer GJK + EPA (`engine/sim/gjk.h`, namespace `hf::sim::gjk`)

Slices GJ1–GJ6 lift the single hardest limitation of the whole contact stack: the narrowphase was **box-only** (oriented-box SAT). This flagship generalizes it to **arbitrary convex polyhedra** — tetrahedra, octahedra, wedges, boxes — via deterministic integer **GJK** (Gilbert-Johnson-Keerthi, for overlap + closest distance) and **EPA** (Expanding Polytope Algorithm, for penetration depth + contact normal), all in Q16.16 fixed point and **bit-identical across CPU/Vulkan/Metal and lockstep/rollback-replayable**. GJK/EPA is the industry-standard narrowphase, but mainstream engines run it in floating point — whose support-dot, simplex-barycentric, and polytope-expansion accumulations are FPU-order- and vendor-dependent, so the simplex they converge to (and therefore the normal and depth they report) diverges machine-to-machine, which is precisely why their networked rigid-body physics is never bit-deterministic. Hazard Forge does the same algorithm with a fixed iteration bound and integer tie-breaks, so the *general convex contact itself* is replayable from inputs alone. It is the **highest reuse-to-novelty** move in the suite: it swaps one box at the bottom of an otherwise frozen tower — the solver, friction, warm-start, sleeping, lockstep, and render layers above the narrowphase are reused verbatim, because the new narrowphase produces the same `convex::ContactManifold` every downstream consumer already speaks. `gjk.h` is an **additive sibling** that `#include`s `convex.h` read-only; **no new RHI across the whole flagship** — the new compute shaders are int64 (Vulkan-GPU + a byte-identical Metal CPU reference, the established split), GJ5 is pure CPU, and GJ6 reuses the instanced-lit pipeline verbatim. The TWENTY-SECOND flagship, complete (6/6).

- **The support function (Slice GJ1, `shaders/gjk_support.comp.hlsl`).** The single primitive GJK and EPA are both built on: `Support(hull, body, dir)` returns the world-space vertex of a convex hull farthest along a direction (an integer argmax of `FxDot` over the hull's local vertices, rotated into world space), and `SupportMinkowski` returns the support of the Minkowski difference A⊖B — the only geometry call the rest of the flagship makes. int64 (world-scale dots) → Vulkan + Metal CPU reference. **Proof:** GPU support vertices == CPU byte-for-byte over a fixed hull set (tetra/box/octa/wedge) × a direction sweep; every returned vertex is a true maximum (brute-force check); cross-vendor zero. Golden `gjk_support`.
- **GJK overlap + closest distance (Slice GJ2, `shaders/gjk_distance.comp.hlsl`).** The GJK simplex evolution: fed only `SupportMinkowski` queries, it evolves a 1–4 point simplex toward the origin (the integer Johnson sub-distance — vertex/edge/face/tetra Voronoi regions by `FxDot`/`FxCross` sign tests, fixed lowest-index tie-breaks, a fixed iteration bound), deciding whether two hulls overlap and, when separated, their closest-point distance + witness points. int64 → Vulkan + Metal CPU reference. **Proof:** GPU == CPU byte-for-byte; the overlap boolean agrees with an independent reference over separated/touching/overlapping pairs; the closest distance is within the fixed-point band; cross-vendor zero. Golden `gjk_distance`. **Caveat:** `Gjk` itself is byte-identical CPU/GPU; the distance is within-band vs a different reference algorithm, not cross-algorithm byte-identical.
- **EPA penetration depth + normal (Slice GJ3, `shaders/gjk_epa.comp.hlsl` — the crux).** Seeded from GJK's terminal overlap simplex, EPA expands a polytope of the Minkowski difference outward to the boundary to find the penetration depth, contact normal, and contact points. EPA's iterative polytope expansion is the single most determinism-sensitive piece in the flagship; it is pinned with a fixed iteration bound, an integer convergence epsilon, a lowest-index closest-face tie-break, consistent face winding (oriented from the seed-tetra centroid, since the origin lies *on* the seed boundary for axis-aligned overlaps), and a fixed horizon re-triangulation order. int64 → Vulkan + Metal CPU reference. **Proof:** GPU == CPU byte-for-byte (the determinism crux); analytic box-into-box depths match within band; the normal is unit and separates (translating one body by depth·normal removes the overlap on re-query); two-run determinism; cross-vendor zero. Golden `gjk_epa`. **Caveat:** EPA termination is deterministic but within-band (best within the bound), not guaranteed analytically optimal.
- **The general-hull world step (Slice GJ4, `shaders/hull_step.comp.hlsl` — the new physics).** The payoff: `StepHullWorld` reproduces the convex 5-pass per-tick shell (predict-integrate → all-pairs narrowphase → world Gauss-Seidel impulse → position de-penetration → orientation) with the **only** change being the box SAT swapped for the GJK/EPA `HullContact` (which returns a `convex::ContactManifold`), plus a hull inverse-inertia (a diagonal tensor, exact for the symmetric canonical hulls). Arbitrary convex polyhedra integrate, collide, and **settle** — a tetrahedron resting on its triangular face, a wedge interlocked against a box, physics the box-only solver literally cannot represent. int64 → Vulkan drives the world step; Metal CPU `StepHullWorldN` byte-identical. **Proof:** GPU final world == CPU byte-for-byte over 240 ticks; the dynamic hulls came to rest (max speed below band) and are held (max penetration within slop). Golden `gjk_settle`. **A note on robustness:** the single-thread world-step dispatch runs the whole 240-tick simulation in one GPU submission (~5s), which marginally exceeds the Windows GPU watchdog (TDR) and intermittently lost the device; the fix splits it into bounded sub-dispatches over the same buffer with a barrier between — bit-identical to one big dispatch, only the dispatch boundary moves. **Caveats:** convex polyhedra only; single-point manifold (a face-resting hull may rock within an angular-damping band); diagonal inertia (off-diagonal dropped — exact for the symmetric test hulls).
- **Lockstep + rollback (Slice GJ5, pure CPU — `RunHullLockstep`/`RunHullRollback`).** The hull world *is* pure data (the `hulls` are immutable geometry, like boxes), so GJ5 reuses the command + snapshot machinery **verbatim** and swaps only the per-tick step to `StepHullWorld`. Two peers fed the input command stream alone converge byte-identical; a mispredicted impulse rolls back from a snapshot and re-sims bit-for-bit. No GPU/shader/RHI — both backends run the identical harness, the golden bit-identical by construction. **Proof:** authority == replica bit-identical; rollback corrected == authority; a real divergence before rollback; two-run determinism. Golden `gjk_lockstep`. **Deterministic, rollback-replayable, bit-identical-cross-platform contacts for *arbitrary convex polyhedra*, re-derived from inputs alone — which mainstream float physics cannot provide.**
- **Lit 3D instanced render (Slice GJ6, render-only — the money-shot).** The capstone: `HullToRenderInstances` triangulates each hull's faces and transforms them by the bit-exact body pose into a float world-space mesh, drawn through the **existing instanced-lit pipeline reused verbatim** as a lit, shadowed 3D scene — the polyhedra rendered as **true polyhedra**, not box proxies. The distinctive shot is a **matte pile of mixed convex shapes** — a rust tetrahedron, an amber octahedron, and a box at rest on a floor under directional light. **Matte** material to avoid the sky-IBL iridescence. The float visresolve-bar: Metal two-run DIFF 0.0000 + provenance (two render calls byte-equal) + an in-band cross-vendor mean (~37/channel — the single float crossing; the integer-determinism headline is GJ1–GJ5's). **Proof:** `{hulls, tris}` provenance, `{dynamic, restedPile}`, two-render determinism, a non-trivial shaded pixel count. Golden `gjk_render`. **Completes flagship #22 — the twenty-second flagship.**

### Deterministic integer broadphase (`engine/sim/broad.h`, namespace `hf::sim::broad`)

Slices BP1–BP6 retire the single most cross-cutting limitation of the rigid-body stack: every solver
(convex/friction/persist/hull) iterated **all i<j body pairs** — the O(n²) "all-pairs small scene" cost. This
flagship adds a deterministic integer **spatial-hash broadphase** — a uniform grid + axis-aligned-box overlap
cull that produces a candidate-pair set — and proves it **bit-transparent**: the broadphase-driven solver returns
a result *byte-identical* to the all-pairs reference, only faster, letting the rigid solvers scale to hundreds of
bodies. Mainstream engines run the broadphase in floating point (float-AABB sort-and-sweep, float-BVH refit),
whose comparisons are FPU-order- and vendor-dependent, so two machines can disagree on *which pairs even enter
the narrowphase* — silently diverging a networked simulation before a single contact is solved. Hazard Forge's
broadphase is the first stage of the pipeline that is *itself* lockstep-exact, not merely the contacts downstream
of it. It is the **highest-reuse** move in the suite: the grid-hash, the integer AABB predicate, the
count→scan→emit compaction, and the all-pairs pair list it is proven equivalent to all already shipped. `broad.h`
is an **additive sibling** that `#include`s the prior headers read-only; **no new RHI across the whole flagship**
— and notably BP1/BP2 are *pure int32*, so they generate native Metal shaders (a true GPU pass on **both**
backends, the strongest proof tier). The TWENTY-THIRD flagship, complete (6/6).

- **The body grid + cell table (Slice BP1, `shaders/broad_cell_{count,scan,emit}.comp.hlsl`).** A uniform grid
  buckets each body's cell, built as a CSR cell table (per-cell count → exclusive prefix-sum → ascending-index
  scatter). Pure int32 (cell quantization + scatter, no products) → **native on both backends**. **Proof:** GPU
  cell table == CPU byte-for-byte; the CSR is a total partition (every body once, ascending within its cell);
  cross-vendor zero. Golden `broad_cell`.
- **The candidate-pair generator (Slice BP2, `shaders/broad_pair_{count,scan,emit}.comp.hlsl` — the crux).** For
  each body, scan the 27-cell neighbourhood and emit the canonical `i<j` AABB-overlapping pairs (a single-thread
  ascending emit, de-duplicated). Pure int32 → native on both backends. **Proof (the spine of the flagship):**
  the grid-emitted pair set is **byte-identical, after a canonical sort, to the all-pairs reference** — the
  broadphase emits the exact pair list the O(n²) scan would, none missed, none duplicated. This is an *exact*
  equivalence (a byte compare), not a within-band metric, and it is self-policing — if a body larger than the
  cell were missed, the proof would fail loudly. Golden `broad_pair`.
- **The broadphase-driven box world step (Slice BP3, `shaders/broad_convex_step.comp.hlsl` — the scaling beat).**
  `StepConvexWorldBP` reproduces the convex 5-pass tick with the all-pairs solve + de-penetration loops replaced
  by iteration over the broadphase pair list — re-built each tick and **sorted (i,j)-ascending** so the
  order-dependent Gauss-Seidel result matches the all-pairs visiting order exactly. A static floor (a body
  spanning many cells) is handled by a small dynamic-vs-static pass, self-policed by the equivalence proof.
  int64 → Vulkan drives the step; Metal CPU reference byte-identical. **Proof:** GPU == CPU; and the **scale
  proof** — a 256-box pile settles byte-identical to the all-pairs reference on the same scene (the broadphase
  changes only performance, not a bit). Golden `broad_convex`. **A note on robustness:** the single-thread
  256-body step runs in bounded sub-dispatches (one tick each) so it never approaches the GPU watchdog timeout —
  the controller's repeated-run gate caught a too-large chunk that intermittently lost the device, and the fix is
  a smaller chunk, provably under the watchdog.
- **The broadphase-driven hull world step (Slice BP4, `shaders/broad_hull_step.comp.hlsl`).** The same for
  arbitrary convex hulls: `StepHullWorldBP` swaps the box AABB for the hull's true world AABB (from support
  queries) and the box narrowphase for the GJK/EPA hull narrowphase. A margin inflates the candidate AABBs into a
  superset covering the sub-pixel integer-GJK within-band contact, keeping the scale proof exact (the extra far
  pairs are no-ops). int64 → Vulkan + Metal CPU reference. **Proof:** GPU == CPU; a mixed-hull pile settles
  byte-identical to the all-pairs hull reference. Golden `broad_hull`.
- **Lockstep + rollback (Slice BP5, pure CPU — `RunBroadLockstep`/`RunBroadRollback`).** Reuses the hull lockstep
  harness verbatim, swapping the step for `StepHullWorldBP`. The broadphase is re-derived each tick from current
  positions, so two peers re-broadphase to the *same* pair list — **the lockstep holds through the broadphase**,
  not merely the narrowphase. No GPU/shader/RHI — both backends run the identical harness. **Proof:** authority ==
  replica bit-identical; rollback corrected == authority; a real divergence before rollback; two-run determinism.
  Golden `broad_lockstep`. **A deterministic broadphase whose candidate set is itself lockstep-replayable —
  mainstream float physics cannot reproduce which pairs enter the narrowphase across machines.**
- **Lit 3D instanced render (Slice BP6, render-only — the money-shot).** `BroadToRenderInstances` delegates to the
  frozen hull render bridge; a heap of dozens of mixed convex polyhedra — stepped by the broadphase-accelerated
  solver — drawn through the **existing instanced-lit pipeline reused verbatim** as a matte, lit, shadowed 3D
  scene. **Proof:** `{hulls, tris}` provenance (two render calls byte-equal), `{dynamic, restedPile}`, two-render
  determinism, a non-trivial shaded pixel count; the float visresolve-bar (Metal two-run DIFF 0.0000 + an in-band
  cross-vendor mean ~46/channel). Golden `broad_render`. **Completes flagship #23 — the twenty-third flagship.**

### Deterministic integer continuous collision detection (`engine/sim/ccd.h`, namespace `hf::sim::ccd`)

Slices CD1–CD6 close the last open assumption in the rigid-body pipeline: that a body never moves more than its
own thickness in a tick. A discrete solver tunnels a fast or thin body straight through geometry — the single
most recognizable gap between a toy solver and a shipping one. This flagship adds **continuous collision
detection** by **conservative advancement**: a deterministic integer time-of-impact that arrests a fast mover
exactly where it first touches, computed by repeatedly querying the closest distance and advancing time by a step
guaranteed not to overshoot. Mainstream engines compute time-of-impact with floating-point root-finding whose
result drifts by platform, compiler, and FMA contraction — so their CCD is explicitly *non-deterministic* and is
disabled in lockstep netcode. This is a **bit-identical integer time-of-impact**: a bullet's stopping point
against a thin wall is the same fixed-point coordinate on every CPU, every GPU, every backend — and it survives
rollback. It is the **highest-reuse** move in the suite: the hardest primitive is already frozen — the convex
narrowphase already returns, for a separated pair, the exact closest-distance vector and witness points, so the
time-of-impact loop is a few dozen lines of new arithmetic wrapping that frozen function, and the swept-AABB
broadphase reuses the #23 grid with zero new broadphase code. `ccd.h` is an **additive sibling** that `#include`s
the prior headers read-only; **no new RHI across the whole flagship**. The TWENTY-FOURTH flagship, complete (6/6).

- **The time-of-impact primitive (Slice CD1, `shaders/ccd_toi.comp.hlsl`).** `ConservativeAdvance` returns the
  integer time-of-impact of a moving convex pair: read the gap from the closest-distance query, advance time by
  `gap / (conservative max closing speed)` — a lower bound on the true impact, so it never overshoots — and
  repeat until the gap closes. The closing-speed bound is the determinism crux: it must include the angular
  contribution (each body's bounding radius × its angular speed) or a rotating pair's gap closes faster than the
  linear bound predicts and the loop overshoots. int64 → Vulkan + Metal CPU reference. **Proof:** GPU
  time-of-impact == CPU byte-for-byte; a straight approach stops at the hand-computed fixed-point instant; a
  rotating approach does not overshoot (the impact pose is non-overlapping); a receding pair reports no impact.
  Golden `ccd_toi`.
- **The swept-AABB broadphase (Slice CD2, `shaders/ccd_swept_{count,scan,emit}.comp.hlsl`).** Each body's box is
  the union of its box at the tick's start and end poses, so a fast mover's sweep overlaps obstacles the
  instantaneous box would miss. The pair generation over the swept boxes reuses the #23 grid + compaction with
  zero new code, and is pure int32 → **native on both backends**. **Proof:** GPU == CPU byte-for-byte; and the
  **superset proof** — the swept-pair set contains every pair the discrete broadphase produces *plus* the
  fast-mover pairs the discrete set misses (a sweep that crosses an obstacle non-overlapping at both endpoints),
  the bit-transparent superset that catches what the discrete one would tunnel past. Golden `ccd_swept`.
- **The substepped CCD world step (Slice CD3, `shaders/ccd_step.comp.hlsl`).** `StepHullWorldCCD` sweeps the
  candidates, computes each pair's time-of-impact, advances all bodies to the *earliest* impact, resolves that
  contact with the frozen solver, then consumes the remaining tick — bounded substep budget. int64 → Vulkan +
  Metal CPU reference; chunked one tick per dispatch so it never approaches the GPU watchdog. **Proof:** GPU ==
  CPU; and the **no-tunnel proof** — in a fast-mover scene the CCD step lands the body on the correct side of the
  wall, while the discrete step on the same scene tunnels it through (the two final states differ, the CCD one
  passes a no-tunnel predicate). Golden `ccd_step`.
- **A bullet through a thin wall stops (Slice CD4, render-only over the reused step).** The headline beat: a fast
  projectile fired at a thin static wall — per-tick travel many times the wall thickness — is arrested at the
  wall face by the CCD step, where the discrete step lets it pass. Reuses CD3's step verbatim (no new shader),
  adding the dedicated scene and the side-by-side discrete control. int64 → Vulkan + Metal CPU reference.
  **Proof:** `{tunneled:false, impactTick}` GPU == CPU; the discrete control on the identical scene reports
  `tunneled:true`. Golden `ccd_bullet`.
- **Lockstep + rollback (Slice CD5, pure CPU — `RunCcdLockstep`/`RunCcdRollback`).** Reuses the lockstep harness
  verbatim, swapping the step for the CCD step. The time-of-impact is re-derived each substep from the current
  positions and velocities, so two peers re-compute the same impact times — **the lockstep holds through the
  swept continuous solve**, not just the discrete step. No GPU/shader/RHI — both backends run the identical
  harness. **Proof:** authority == replica bit-identical; rollback corrected == authority; a real divergence
  before rollback; two-run determinism. Golden `ccd_lockstep`. **A deterministic, rollback-replayable,
  bit-identical-cross-platform continuous-collision solver — which mainstream float CCD, being non-deterministic,
  cannot provide.**
- **Lit 3D instanced render (Slice CD6, render-only — the money-shot).** `CcdToRenderInstances` delegates to the
  frozen hull render bridge; the CCD-settled bullet-wall world — the projectile arrested at the thin wall — drawn
  through the **existing instanced-lit pipeline reused verbatim** as a matte, lit, shadowed 3D scene. **Proof:**
  `{hulls, tris}` provenance (two render calls byte-equal), `{projectile:arrested, tunneled:false}`, two-render
  determinism, a non-trivial shaded pixel count; the float visresolve-bar (Metal two-run DIFF 0.0000 + an in-band
  cross-vendor mean ~45/channel). Golden `ccd_render`. **Completes flagship #24 — the twenty-fourth flagship.**

### Deterministic hull narrowphase hardening (`engine/sim/manifold.h`, namespace `hf::sim::manifold`)

Slices MF1–MF6 harden the general convex-hull narrowphase. The hull contact path (#22) shipped with two documented
approximations: a **single-point** contact manifold and an **axis-aligned-bounding-box diagonal** inertia tensor.
Together they make a hull dropped flat onto a surface **teeter on one point** instead of resting, and they give a
tumbling polyhedron the wrong spin. This flagship replaces both with the physically correct constructions —
**multi-point contact manifolds** generated by deterministic integer face clipping, and a **full convex inertia
tensor** computed by signed-tetrahedron decomposition — and wires them into a stabilized step where a flat-dropped
hull **settles to rest** the single-point step cannot hold. A mainstream float engine gets stable resting and
stacking only by accepting non-determinism in its iterative contact solve; this does it **bit-identically across
CPU, Vulkan, and Metal**, and the stabilized result is lockstep- and rollback-replayable. The design is **additive
by construction**: `manifold.h` is a new sibling header that `#include`s the prior headers read-only and supplies
*hardened alternatives* alongside the frozen functions; the original single-point step and every one of its
committed goldens are byte-unchanged. **No new render RHI across the flagship.** The TWENTY-FIFTH flagship, complete
(6/6).

- **Hull face topology (Slice MF1, pure CPU + render).** The frozen hull is vertices-only; MF1 adds the missing
  primitive — each canonical hull's deterministic, outward-wound **polygon face table** (a box's six quads, a
  tetrahedron's four triangles, and so on), plus the reference/incident face selectors a clip needs, all pure
  integer. A face-colored lit render makes the decomposition visible. **Proof:** exact face counts; every face
  outward-wound; the selectors a pure deterministic function. Golden `mf1_faces`.
- **The multi-point manifold (Slice MF2, pure CPU + render).** Given two overlapping hulls and the penetration
  normal, pick the most-aligned reference face and the most-anti-aligned incident face and **clip** the incident
  polygon against the reference face's side planes by deterministic Sutherland–Hodgman — every keep/drop decision a
  strict integer sign test with no tolerance band, so the variable-length contact set is reproducible to the last
  bit. A box resting flat on a box yields **four** contact points where the old path gave one. **Proof:**
  `{boxOnBox:4, tetraOnFace:3, edgeOnFace:2}`; all points below the reference face; two-run byte-equal; the manifold
  normal agrees with the seed normal. Golden `mf2_clip`.
- **The manifold on the GPU (Slice MF3, `shaders/hull_manifold.comp.hlsl`).** The same clip lifted onto the device,
  one thread per pair, producing the identical 1–4 point manifold. int64 → Vulkan + Metal CPU reference. The
  variable-length point set and the integer tie-breaks are the flagship's hardest determinism surface, and the
  proof is that they reproduce on real GPU hardware. **Proof:** the GPU manifold array == the CPU array
  byte-for-byte over a pair battery; the counts match; the normal matches the narrowphase. Golden `mf3_manifold`.
- **Full inertia + the stabilized step (Slice MF4, `shaders/hull_step_hardened.comp.hlsl` — the new-physics beat).**
  `FxHullInertiaBodyFull` computes a hull's full symmetric inertia tensor by signed-tetrahedron decomposition in
  body-local fixed point (kept local and accumulated wide so the covariance integrals never overflow), then inverts
  it with a deterministic integer adjugate. `StepHullWorldHardened` is the frozen step's five-pass shell with
  exactly two callees swapped — the multi-point manifold and the full inertia — so the frozen solver consumes both
  unchanged. int64 → Vulkan + Metal CPU reference, chunked one tick per dispatch (never near the GPU watchdog).
  **Proof:** the **cube cross-check** — the full tensor reduces *exactly* to the known box-diagonal answer for a
  cube (max error zero), proving it is a strict generalization, not a different model; GPU == CPU byte-for-byte; and
  the headline — under the hardened step a flat-dropped hull **settles** (residual angular speed in a tight band)
  while the frozen single-point step on the same scene leaves it **teetering** (an order of magnitude more residual
  motion). Golden `mf4_stack`.
- **Lockstep + rollback (Slice MF5, pure CPU — `RunHullLockstepHardened`/`RunHullRollbackHardened`).** The hardened
  step's only mutable state is the body vector — the faces, inertia tensors, and per-pair manifolds are re-derived
  each tick from the immutable geometry — so the frozen lockstep harness retargets onto it with no new snapshot
  type. Two peers fed only an input-command stream re-derive the entire hardened world, including the multi-point
  manifolds and the full-inertia response, byte-identical, and a rollback re-sims from a snapshot bit-for-bit. No
  GPU/shader/RHI — both backends run the identical harness. **Proof:** authority == replica bit-identical; rollback
  corrected == authority; a real divergence before rollback; two-run determinism. Golden `mf5_lockstep`. **The
  deterministic stabilization a mainstream float engine disables in lockstep, here replayable as netcode.**
- **Lit 3D instanced render (Slice MF6, render-only — the money-shot).** `HardenedHullToRenderInstances` delegates
  to the frozen hull render bridge; the hardened-settled world — a hull at rest on its support — drawn through the
  **existing instanced-lit pipeline reused verbatim** as a matte, lit, shadowed 3D scene. **Proof:** `{hulls, tris}`
  provenance (two render calls byte-equal); the rendered world is the settled one (residual angular speed in band);
  the integer sim is byte-unmutated by the render; two-render determinism; the float visresolve-bar (Metal two-run
  DIFF 0.0000 + an in-band cross-vendor mean ~44/channel). Golden `mf6_render`. **Completes flagship #25 — the
  twenty-fifth flagship.**

The hardened solver is a within-band iterative solve: "settles" means residual motion inside a fixed band, not
analytic zero. The proven stabilization is a **single hull resting on a support** — the teeter the single-point
manifold could not hold. A *tall* multi-body stack (three or more hulls) still destabilizes in this Gauss-Seidel +
linear position-correction solver as residual jitter accumulates; a robustly stable stack is a deferred refinement
(warm-started contacts / a stiffer stacking solve). Determinism, however, holds for *any* scene — the lockstep
proof (MF5) reproduces a multi-body world bit-for-bit regardless of whether it has settled.

### Warm-started hull contacts + robust stacking (`engine/sim/warmhull.h`, namespace `hf::sim::warmhull`)

Slices WH1–WH6 close the open problem the previous flagship documented: a *tall* hull stack destabilizes because
the non-accumulated impulse solver, re-deriving a fresh impulse each tick, leaks a tiny fixed-point residual torque
that integrates into a spurious spin and topples the tower — previously masked only by a global angular-damping
crutch. This flagship removes the *source* of that torque with **warm starting** (re-applying each contact's
accumulated impulse from the previous frame so the solver converges to a consistent island equilibrium across ticks
rather than restarting each tick) plus **sleeping islands** (a settled island freezes at exactly zero velocity
before the residual can re-accumulate). The result is a **deterministic, bit-identical, lockstep-replayable stable
stack of convex polyhedra** — a capability a mainstream float engine reaches only non-deterministically (its
contact cache and sleep thresholds drift by platform, which is why such engines disable warm starting in lockstep
netcode). `warmhull.h` is an **additive sibling** that `#include`s the prior headers read-only; the frozen
narrowphase, the hardened step, and every one of its goldens are byte-unchanged. The TWENTY-SIXTH flagship,
complete (6/6).

- **The hull contact feature ID (Slice WH1, `shaders/warmhull_key.comp.hlsl`).** Warm starting needs to name a
  contact across frames so this tick's manifold point can inherit last tick's impulse. A box solver names a contact
  by its discrete separating-axis index; a hull manifold has no such index — it comes from a continuous penetration
  normal and a polygon clip whose output-slot order renumbers under sub-pixel motion. The fix is a **geometric
  provenance** key — the reference face plus the incident vertex/edge each clipped point descends from — emitted by
  a tagged clip that mirrors the frozen clip byte-for-byte (its output positions are asserted byte-equal). The key
  is pure 32-bit integer (face/edge indices, shifts, an avalanche hash), so the shader is **native on both
  backends** — the strongest tier, strict-zero cross-vendor. **Proof:** the tagged clip == the frozen clip
  byte-for-byte; distinct contacts hash distinct, and a contact under sub-pixel motion keeps its key; GPU == CPU.
  Golden `wh1_keys`.
- **The keyed manifold + persistent cache (Slice WH2, `shaders/warmhull_cache.comp.hlsl`).** A per-frame cache
  matches this tick's keyed manifold points to last tick's accumulated impulses by key — inheriting a warm match,
  cold-starting an absent one, evicting a departed one. int64 → Vulkan + Metal CPU reference. **Proof:** matched +
  cold == points; a departed key is evicted; GPU == CPU. Golden `wh2_cache`.
- **The accumulated warm solver (Slice WH3, `shaders/warmhull_warm.comp.hlsl`).** The engine of the flagship: an
  *accumulated* normal sequential-impulse solver that primes each point from its cached impulse and applies only
  the per-iteration delta, clamping the accumulated total non-negative — replacing the non-accumulated kernel that
  cannot be warm-started. int64 → Vulkan + Metal CPU reference, chunked one tick per dispatch. **Proof:** at the
  same low iteration budget the warm solver's residual (the windowed angular speed — the residual-*torque* metric,
  not linear) is strictly lower than the non-accumulated one's, and the stack holds with the angular-damping crutch
  switched off; GPU == CPU. Golden `wh3_warm`.
- **Sleeping islands → the stable stack (Slice WH4, `shaders/warmhull_sleep.comp.hlsl` — the new-physics beat).**
  An island of contacting hulls whose energy stays low for a hysteresis window freezes — its velocities zeroed
  exactly, its solve skipped — so the residual torque has nothing left to integrate; an external impulse wakes the

### Deterministic gameplay / netcode product layer (`engine/game/verdict.h`, namespace `hf::game::verdict`)

Slices VD1–VD6 turn the determinism moat into a *product*. Every prior simulation flagship proved a single
subsystem lockstep-replayable; this one is the first artifact that is a *game* — entities, gameplay rules, and a
physics sim — whose ENTIRE world is bit-identical across CPU / Vulkan / Metal and replayable from an input stream

net layer sat on a *float* physics world (same-build determinism only), and each sim re-implemented the
snapshot/equality machinery by hand. Verdict UNIFIES them: a single deterministic game-world abstraction (a fixed

mainstream engine cannot match: ship a rollback-netcode game whose physics AND gameplay are byte-identical

### Hardware ray tracing, deterministically reconciled (`engine/render/rtrace.h`, namespace `hf::render::rtrace`)

Slices RT1–RT6 add **hardware ray tracing** — and do it without giving up the determinism moat. The defining
problem: a mainstream engine's hardware ray tracing is *float* and *vendor-defined* — the GPU's acceleration
structure walks the scene in an unspecified order and commits the closest hit with the driver's own floating-point
arithmetic, so two GPUs (or a GPU and a reference) need not agree bit-for-bit. This engine's whole thesis is the
opposite: **bit-identical across platforms**. RT1–RT6 reconcile the two with one idea — *use the hardware ray
query ONLY as a candidate generator, and keep the closest-hit decision in our own deterministic fixed-point math.*
The result is the differentiator: **hardware ray tracing whose every frame is byte-identical on NVIDIA and Apple,
and identical to a software reference** — RT speed, with a reproducible image no float-only RT pipeline can promise.
`rtrace.h` is one append-only header; the whole render path is **pure-integer Q16.16** (the strictest determinism
tier, strict zero-pixel). The TWENTY-EIGHTH flagship, feature-complete (6/6).

- **The deterministic software reference tracer + the accel-structure RHI seam (Slice RT1).** A pure-integer
  Q16.16 ray tracer — integer primary-ray generation, fx ray/sphere and ray/AABB intersection (the discriminant
  root and the slab divides in `int64`), a brute-force closest-hit by the total order `(t, primIndex)`, and an
  integer Lambert shade. Its GPU twin (`shaders/rt_trace.comp`, one thread per pixel) is proven **byte-identical**
  to the CPU. Critically the search **brute-forces every primitive with NO acceleration-structure cull** — this is
  the *completeness oracle* the later HW slices are checked against. Alongside it lands the additive, defaulted-no-op
  **acceleration-structure RHI seam** (`IAccelStructure`, `CreateBlas`/`CreateTlas`/`SupportsHardwareRayQuery`,
  `BindAccelStructure`) — inert in RT1, leaving every existing golden byte-unchanged. Golden `rt1_trace`, strict-zero
  cross-vendor.
- **Tier-A deterministic hardware ray query (Slice RT2) — the moat proof.** The real Vulkan `VK_KHR_ray_query` +
  `VK_KHR_acceleration_structure` path, wired behind the RT1 seam (capability-gated: a no-RT device degrades to the
  software path, so CI without an RT GPU still passes). The kernel (`shaders/rt_query.comp`) opens an inline
  `RayQuery` over a TLAS of **margin-inflated** procedural AABBs, **drains every candidate** (`Proceed()` without ever
  committing or shrinking `tMax`), runs OUR frozen fx intersection on each, and folds the closest by `(t, primIndex)`
  — **never reading the query's float `t`**. Because the min is over our integer keys, the result is independent of
  the vendor's traversal order and layout. The AABB margin makes the float overlap a strict *superset* of every true
  fx hit, so the hardware path can never skip a hit the no-cull reference finds — and that is checked directly:
  **HW == SW (brute-force) == CPU, byte-identical**, with a zero-missed-candidate proof, on real hardware (an NVIDIA
  RTX 4070). Golden `rt2_query`, strict-zero cross-vendor.
- **The Metal hardware-ray-query twin (Slice RT2b) — cross-platform HW-RT parity.** The same approach on Apple
  silicon: a hand-authored `shaders/rt_query.metal` (the first hand-written `.metal` in the tree) runs a
  `metal::raytracing::intersection_query` — the direct analog of the HLSL `RayQuery` — over an
  `MTLAccelerationStructure` of bounding boxes, draining candidates and reducing in the SAME fx math (`long`
  arithmetic). Validated on an Apple M4: the Metal hardware image is **byte-identical to the Vulkan hardware image
  and to the CPU reference** (golden `rt2_hwquery`, the same bytes as `rt2_query`). The fixed-point-reduction-over-
  hardware-candidates idea now holds bit-for-bit on **two independent RT hardware vendors**.
- **Hard shadows (Slice RT3).** After the primary hit, a SECOND ray query — a shadow ray toward the directional
  light — gates the diffuse term. Occlusion is an *any-hit* test, an order-independent boolean OR over our fx
  intersections, so the hardware path (which may early-out on its first occluder) and the software reference (which
  may scan all) agree exactly — **HW == CPU byte-identical**. Golden `rt3_shadow`, strict-zero.
- **Mirror reflections (Slice RT4).** One-bounce mirror reflections, kept **strictly integer**: an fx reflect
  vector, the reflected hit shaded with the RT3 shadowed shade, composited by a fixed per-surface reflectivity as an
  integer per-channel blend. No float anywhere — so reflections stay strict-zero cross-vendor (the reflective floor
  mirrors the scene, a mirror sphere reflects its neighbours, the shadows intact). Golden `rt4_reflect`.
- **The determinism-envelope + lockstep tie-in (Slice RT5) — the moat ⊗ RT closure.** This slice composes the two
  pillars: a deterministic fixed-point rigid-body simulation, hardware-ray-traced. Two lockstep peers fed only inputs
  produce byte-identical sim states; the hardware-ray-traced image of the authority equals the image of the replica
  equals the CPU reference (the determinism envelope, here exactly zero). **A deterministic physics simulation,
  hardware-ray-traced, is bit-identical between peers and across vendors** — the headline no float-RT engine can
  state. No new shader, no change to the render core (a sim→scene bridge lives in the showcase layer). Golden
  `rt5_simrender`.
- **The lit hero capstone (Slice RT6) — the money-shot.** One polished, higher-resolution (480×360) scene with the
  whole feature set — primary, hard shadows, mirror reflections, a mirror sphere — under an integer **graded sky**.
  Deliberately STRICT INTEGER (the sky is an integer gradient, not float shading) so the flagship's headline holds to
  the last pixel: the entire hardware-ray-tracing pipeline, the hero shot included, is byte-identical HW == CPU and
  strict-zero across vendors. Golden `rt6_hero`. **Completes flagship #28 — the twenty-eighth flagship.**

Honest scope: this is the moat-preserving *Tier-A* (deterministic, procedural/analytic-geometry, strict-integer)
plus a software-reference *Tier-C* golden — exactly the pieces a bit-exact reflections / GI path needs. Deliberately
out of scope here: float HW-triangle ray tracing (the *Tier-B* fidelity path, a later concern), multi-bounce /
path-traced global illumination (the next flagship), glossy/rough reflections, refraction, soft shadows, and AO. The

*the only engine with hardware ray tracing AND a deterministic, cross-platform-identical reference for it.*

### Deterministic Lumen-class global illumination (`engine/render/gi.h`, namespace `hf::render::gi`)

Slices GI1–GI6 add **multi-bounce global illumination** — and keep it *bit-identical across platforms*, which is the
thing the comparison engine's GI cannot be. That engine's flagship GI is float, temporally accumulated, denoised, and
view-dependent; by construction it is not reproducible frame-to-frame, let alone across two GPUs. This engine answers
with the opposite: an **RT-traced, integer spherical-harmonics irradiance probe volume with deterministic multi-bounce
feedback** — the integer GI twin of the (now superseded) float DDGI, re-hosted on the hardware-ray-tracing foundation
(flagship #28) and computed entirely in **pure-integer Q16.16**, so the GI is **byte-identical HW == CPU on NVIDIA and
Apple**. It builds straight on the RT primitives — the probe rays are `rtrace::TraceClosest` over the procedural-AABB
scene (world-space, off-screen-complete, the HW BVH used only as a candidate generator), shaded with the integer
`rtrace::ShadeHitShadowed`. `gi.h` is one append-only header. The TWENTY-NINTH flagship, feature-complete (6/6).

- **The integer RT probe trace (Slice GI1).** A probe grid casts 16 rays per probe via the RT closest-hit, shading
  each with direct light AND an RT shadow ray (per-ray shadowing the screen-space DDGI never had), the radiance kept
  in integer Q16.16. World-space and off-screen-complete — no light lost off-frame. **Proof:** the per-probe radiance
  buffer is byte-identical HW == CPU. Golden `gi1_probe`.
- **The integer SH encode (Slice GI2) — the crux.** The radiance is encoded into a deterministic integer 3rd-order
  SH. Integer SH coefficients are tiny signed fractions and the multi-bounce loop re-injects them, so the dynamic
  range must be exact: the encode accumulates in an **int64 accumulator and quantizes to Q16.16 only after the single
  normalize** ("stay wide, narrow once" — the same discipline as the ray/sphere discriminant). **Proof (the crux,
  settled CPU-first):** the SH stays in range with no underflow to zero (a band-2 coefficient is provably non-zero)
  and no overflow, and is byte-identical GPU == CPU. Golden `gi2_shencode`.
- **The integer trilinear interpolation (Slice GI3).** Any query point trilinearly blends the 8 surrounding probes'
  SH with **integer weights that sum exactly to one** (partition of unity), giving a continuous irradiance field a
  renderer samples. **Proof:** byte-identical GPU == CPU; lattice-point identity (a query at a probe reproduces that
  probe's SH exactly); the weights sum exactly to `kOne`. Golden `gi3_interp`.
- **Integer multi-bounce feedback (Slice GI4) — light that bounces.** Each probe-ray hit's shade gains an indirect
  term — the previous iteration's interpolated irradiance × albedo — fed back a fixed three iterations. The bounce
  energy is a bounded geometric series (albedo < 1), so it stays within the proven SH headroom. **Proof (the second
  make-or-break):** one bounce is byte-identical to single-bounce (a real no-op), and the second bounce is
  *monotonically brighter* for every probe (the bounce adds light and the integer indirect did not truncate to
  zero). Golden `gi4_bounce`.
- **Integer Chebyshev occlusion weighting (Slice GI5) — the leak fix.** The classic probe-volume artifact — a probe
  behind a wall leaking light to a point in front of it — is fixed with a variance-shadow-map Chebyshev visibility
  term, weighting each trilinear corner by the probability the query point is visible to that probe. The distance
  moments come *for free* from the RT trace's hit distance (no separate distance-cube pass the float DDGI needed).
  **Proof:** byte-identical GPU == CPU; occlusion off is byte-identical to GI3; with occlusion on, points behind an
  occluder are measurably attenuated. Golden `gi5_occlusion`.
- **The lit GI hero capstone (Slice GI6) — the money-shot.** A Cornell-style enclosure rendered with the WHOLE
  pipeline — the lit hero (direct + shadows + reflections + graded sky) plus the occlusion-weighted multi-bounce
  indirect at each hit — showing **colored multi-bounce bleed** (the red wall tints the floor, the green wall the
  other side): the colored GI single-bounce direct lighting cannot produce. Strict integer. **Proof:** byte-identical
  HW == CPU; GI off is byte-identical to the no-GI hero (a literal additive +0); and the color-bleed invariant holds
  (the red-wall-adjacent floor's indirect red channel exceeds green and blue, and symmetrically for the green wall).
  Golden `gi6_hero`. **Completes flagship #29 — the twenty-ninth flagship.**

Honest scope: this is the moat-preserving, *strict-integer* GI — diffuse multi-bounce probe GI with occlusion, every
slice byte-identical across vendors. Deliberately out of scope (deferred fast-follows): a float Tier-B fidelity GI for
final-pixel work, glossy/specular GI from probes (specular stays on the mirror-reflection path), temporal accumulation,
real-time per-frame re-bake, world-scale probe streaming, and bounce counts beyond three. One documented determinism
edge: a *grazing secondary ray against a thin slab wall* can fall in the float-BVH's candidate tail (the capstone's
Cornell objects are matte to stay strict-zero; thick geometry and primary/shadow paths are unaffected). The
differentiator the architecture earns: **deterministic, cross-platform-bit-identical Lumen-class global illumination —
the same GI image on NVIDIA and Apple, reproducible frame for frame, which a float/temporal GI cannot promise.**

### Deterministic hull friction + joints (`engine/sim/hullfric.h` + `engine/sim/hulljoint.h`, namespaces `hf::sim::hullfric` / `hf::sim::hulljoint`)

The general convex-hull contact world (warm-started, sleeping, stacking) resolved only the **normal** impulse — hulls
could rest and stack but not **grip**, and the bodies could not be **linked**. This flagship closes the convex-rigid
story by bringing the two capabilities that already existed for boxes — tangential **friction** and **joints** —
to arbitrary convex hulls, and proves the whole thing bit-identical on NVIDIA and Apple and replayable frame for
frame. Both headers are append-only siblings that `#include` the frozen contact/solver headers read-only; no
existing struct layout changes, so every prior golden is byte-for-byte unchanged.

The **friction** half (`hullfric.h`) is built in three steps, each a strict-integer cross-vendor zero-diff golden.
**HF1** tags each hull contact with a tangent **basis** computed once from the continuous EPA contact normal (not a
discrete face axis), and keys a per-contact cache by contact-feature **and** basis axis — the cross-tick stability
crux: because the EPA normal drifts continuously, a naive warm-start would re-inject stale tangent impulses through a
basis flip, so on a basis-axis change the solver **cold-starts the tangents** while keeping the normal warm
(a deterministic, integer-clean safe reset). **HF2** is the warm cone solver — the box friction solver's
accumulated, cone-clamped (`|jt| ≤ μ·jn`) two-axis tangent sweep wired to the hull's full world-space inverse-inertia
tensor, with a measured warm-start lever (the warm residual is below the cold residual). **HF3** assembles the
friction-locked world step: the warm-hull five-pass step with the friction solver swapped in and the friction cache
**persisted across ticks**. Its falsifiable headline is *grip*: a dynamic hull released on a tilted static hull
**rests** at μ>0 (on-ramp speed near zero) but **slides off** at μ=0 — friction the box solver could do, now on
general convex hulls — bit-exact GPU vs CPU over 240 ticks.

The **joints** half (`hulljoint.h`) is the real new value: **composition**. **HF4** is the one-deterministic-step
headline — `JointedHullWorld` carries the hull world plus ball-joint and angular-limit constraint lists plus the
friction cache, and `StepJointedHullWorld` inserts the body-agnostic joint solvers (reused verbatim — no new
constraint math) into the friction step in one pinned order (integrate → joint passes → friction+contact warm solve →
de-penetrate → orient). A four-link hull **chain** hangs **connected** (the largest joint-anchor gap stays small), a
hinged hull **door** holds its angular limit, and joints + friction + contacts compose bit-exact GPU vs CPU; a
control with empty joint lists is byte-for-byte identical to the pure friction step, proving the joints are purely
additive. **HF5** is the netcode headline — a pure-CPU lockstep + rollback harness where two peers re-derive the
*entire* friction+joint world bit-for-bit from the same command stream, and a mispredicted rollback (a wrong impulse
that diverges across the chain *and* the door *and* the resting hull) corrects exactly. Its sharpest proof is that
the friction **cache must be in the snapshot**: dropping it from the restore diverges, including it converges — the
warm contact state is part of the replayable world, not a derived cache. **HF6** is the lit 3D capstone: the bit-exact
settled world (the hanging chain, the hinged door, a hull gripped on a tilted ramp) drawn through the existing
instanced-lit pipeline via a one-line render delegate, matte to stay clean, the integer world unmutated by the render.
It is the flagship's single float slice, so it carries the float bar (per-backend deterministic golden, two-run
zero-diff, with a documented in-band cross-vendor mean rather than strict zero) while HF1–HF5 are strict-integer zero.

The hull friction/joint shaders that need 64-bit integers (the whole-step kernels) are Vulkan-SPIR-V-only, with the
Metal side running the byte-identical CPU reference — the established convention for the deterministic-sim flagships;
each heavy single-thread step is chunked one tick per dispatch so it can never trip the GPU watchdog. Honest scope:
the Gauss-Seidel + fixed-point-divide residual is within-band (the headline is determinism and cross-platform
bit-identity, not analytic exactness), friction is dry Coulomb-cone only, and joints are ball + hinge + cone-limit
(no motors/drives). The differentiator the architecture earns: **deterministic, cross-platform-bit-identical convex
rigid bodies with friction *and* joints, lockstep-replayable — a frictional jointed mechanism two machines re-simulate
byte-for-byte, which a mainstream float physics engine cannot guarantee.**

### Deterministic IK control-rig (`engine/anim/ik.h`, namespace `hf::anim::ik`)

Character animation is the last place a mainstream engine is irreducibly float and non-deterministic: inverse-kinematics
solvers iterate in floating point, and motion matching searches a float feature database. This flagship extends the
determinism moat into that territory — a **bit-identical, cross-platform, lockstep-replayable** IK control-rig in
Q16.16 fixed-point that drives the existing skeleton + skinning pipeline. It is built as one append-only header over
the frozen fixed-point + joint toolboxes; every solver slice is golden-verified at DIFF 0.0000 on both backends.

The make-or-break problem is that IK needs trigonometry the integer engine did not have — the law of cosines needs
`acos`, look-at needs the inverse trig — and the determinism rule forbids runtime transcendentals (they diverge across
compilers and GPUs). The resolution, landed first and in isolation as the **beachhead**, is a **host-snapped angle
lookup table**: `acos`/`sin`/`cos`/`atan2` are evaluated in double precision at build time, rounded to Q16.16, and
looked up at runtime with pure-integer bin indexing + a deterministic fixed-point interpolation — zero runtime
transcendentals, the same recipe the fluid kernel already uses, generalized from per-scene constants to a swept table.
Because the lookup is pure 32-bit integer, its shader lowers to Metal natively and runs a *real* GPU==CPU comparison on
both backends (the strongest determinism tier), with a documented LSB band against the analytic angle (tight in the
interior, wider only at the `acos` sqrt-singularity).

On that primitive the solvers compose. **Two-bone IK** solves the canonical upper/lower limb by the law of cosines
(one table `acos`) and emits the two bone rotations as quaternions, with the end-effector reaching the target inside a
documented sub-LSB-fraction band and a deterministic clamp when the target is out of reach. **FABRIK** generalizes to
an n-bone chain by pure forward-and-backward position reaching (no trigonometry at all — just integer normalize), which
preserves segment lengths by construction; a shortest-arc **look-at** rotation aims a bone at a target (its `acos(dot)`
is always non-negative, which sidesteps a real cross-backend divergence the two-bone work first surfaced: a *negative*
half-angle phase-reduces differently on GPU vs CPU, so the solvers are written to feed the trig tables only
non-negative angles). The **skeleton bridge** wires the solvers into the animation pipeline: forward-kinematics a base
pose to world joint positions, IK-correct a limb sub-chain toward a world target, convert the corrected positions back
to local joint rotations *in the corrected parent frame* (a subtlety the bridge had to get exactly right, since using
the base-frame compounds the error), and write only the chain joints — the un-IK'd joints stay byte-identical to the
base pose. All of this is integer; the base-pose sample and the float skinning-palette read-back are the documented
render-only crossings on either side of the bit-exact solve.

The moat headline is the **lockstep + rollback** slice: two peers fed only an input stream of effector-target moves
("the foot/hand target the player drags") re-derive the entire IK rig state bit-for-bit every tick, and a mispredicted
target is corrected exactly by rolling back to a snapshot and re-simulating — deterministic, replayable character
animation a float engine cannot reproduce. The capstone is the money-shot: the bit-exact IK-corrected pose drives the
existing GPU-skinned character render — a fox plants a foot on a world target through a leg IK chain, rendered lit 3D
through the same skinned pipeline the animation state machine and the ragdoll capstones use, and the same corrected
pose seeds the deterministic ragdoll (animation → IK → ragdoll → recover, all replayable). The render is the flagship's
single float-tier slice (the integer IK is exact; the final raster is float, with a per-backend deterministic golden +
a documented in-band cross-vendor mean). The differentiator the architecture earns: **deterministic,
cross-platform-bit-identical, rollback-replayable inverse kinematics — a character whose IK two machines re-simulate
byte-for-byte, the one part of the animation stack a mainstream float engine leaves non-deterministic.**

### Shadow-atlas tiling (`SetViewport`)

The expanded shadow set (Slices AD–AF) renders **many** light-space depth maps per frame — N directional cascades, a spot map, and the six faces of a point light's cube — without allocating N separate depth textures. Instead the depth pass clears one shadow **atlas** once, then for each sub-map calls a new seam entry point, `ICommandBuffer::SetViewport(x, y, w, h)`, to restrict rasterization to that tile before drawing the geometry for that cascade/face. The lit pass samples the right tile by transforming into the corresponding light-space matrix and offsetting into the atlas region.

`SetViewport` is a defaulted no-op on the base interface, so passes and backends that don't tile an atlas are unaffected (the pre-existing single-shadow-map goldens are byte-for-byte unchanged). On Vulkan it records `vkCmdSetViewport`/`vkCmdSetScissor`; on Metal it sets the encoder viewport + scissor rect. The CSM frustum-split + per-cascade ortho fit, the spot `spotViewProj`, and the point-light 6-face cube view-proj + dominant-axis face/tile mapping are all pure, header-only, unit-tested math (`render/csm.h`, `render/spot.h`, `render/point_shadow.h`) — the goldens prove the GPU side, the unit tests prove the math.

### Shaders are generated, not hand-written

The Metal shaders are **generated from the shared HLSL sources at build time** — there is no hand-written MSL to drift from the canonical shaders. For each shader the `metal_headless` build runs HLSL → SPIR-V (`glslc -x hlsl`) → MSL (`spirv-cross --msl --msl-decoration-binding`), emitting `*.gen.metal`, which `visual_test` compiles at runtime via `newLibraryWithSource:`. The `--msl-decoration-binding` flag maps each resource's SPIR-V binding directly to its Metal `[[buffer/texture/sampler(n)]]` index; the engine's Metal binding constants (`engine/rhi_metal/metal_common.h`) are chosen to match. The only Metal-specific shader adjustments are guarded by `#ifdef HF_MSL_GEN` (two texture-origin V-flips), so the Vulkan SPIR-V is byte-identical.

#### Shared shader includes (`.hlsli`): one source of truth

Shaders that share math pull it from a shared `.hlsli` **include** (a plain `#include "name.hlsli"` resolved relative to the including file). Both compile paths preprocess the include before lowering, so an `.hlsli` needs **no compile entry / no CMake registration** in either the DXC list (`samples/hello_triangle/CMakeLists.txt`) or the `hf_gen_msl` list (`metal_headless/CMakeLists.txt`) — it only has to sit next to the shaders that pull it in (`shaders/`). The two shared includes:

- **`shaders/pbr_core.hlsli`** — the metallic-roughness PBR lighting core (`hfShadePBR` / `hfShadePBRN`, Cook-Torrance + IBL + PCF shadow), reused by the build-time-codegen'd material-graph shaders so the data-driven materials shade identically to `lit_pbr.frag`.
- **`shaders/procedural_sky.hlsli`** (issue #4) — the procedural clear-day sky: **`HFSkyColor(dir, lightDir)`**, the horizon→zenith gradient + dim ground haze + directional sun glow. This is the **single source of truth** for the sky, pulled by **three** sites that previously each hand-duplicated it: the sky pass (`sky.frag`, the view ray), the lit pass's IBL reflection (`lit.frag`'s `SkyColor(R)` — what every metal/glass surface reflects), and the material-graph IBL (`pbr_core.hlsli`'s `hfSkyColor`). Because the sky dome and the IBL reflection now derive from one function, retuning the sky (a sunset palette, a different gradient) updates the reflections in lock-step — change it once. The animated-sky demo (`sky_animated.frag`) layers a time-driven cloud band on the same `HFSkyColor` base.

There is now one more golden than the 264 above: **`sky_animated`** (the issue #5 time-channel demo — `--sky-animated-shot` on Vulkan / `--sky-animated` on Metal), captured at a fixed `2.0 s`.

## Vulkan Backend Notes

- **Vulkan version:** 1.3 dynamic rendering (`VK_KHR_dynamic_rendering`); no `VkRenderPass` / `VkFramebuffer` objects.
- **Synchronization:** Vulkan Synchronization 2 (`vkCmdPipelineBarrier2` / `VkImageMemoryBarrier2`). A `TransitionImage` helper in `vulkan_device.cpp` centralizes the barrier pattern.
- **Memory:** Vulkan Memory Allocator (VMA 3.3) for all images and buffers. Staging uploads use a host-visible VMA buffer, `vkQueueWaitIdle` after the copy (synchronous load-time path).
- **Frames in flight:** `kFramesInFlight = 2` (double-buffered). Per-frame fences + semaphores in `frames_[frameIndex_]`. Per-frame UBOs indexed by `frameIndex_`.
- **Shader pipeline:** HLSL → SPIR-V via DXC (`-spirv`, invoked by a CMake custom command at build time). Shaders are loaded as `.spv` files at runtime via `IShaderModule`.
- **Depth:** swapchain-sized `VK_FORMAT_D32_SFLOAT` image owned by the device, recreated on resize.

## Scene Layer

`engine/scene/` sits above the RHI and below the application:

- `vertex.h` — canonical `Vertex { pos[3], color[3], uv[2], normal[3], tangent[3] }` (stride 56, tangent added for normal mapping) and `MeshVertexLayout()`; a `SkinnedVertex` (stride 88, + joints/weights) and `SkinnedMeshVertexLayout()`; `InstanceTransformLayout()` (stride 64) for the per-instance stream.
- `transform.h` — `Transform { position, eulerRadians, scale }` with `Matrix()` → TRS `mat4`.
- `mesh.h/.cpp` — `Mesh` owns `IBuffer* vertices` and `IBuffer* indices`; static factories `Cube(device)`, `Plane(device)`, `Sphere(device)`.
- `renderable.h` — `Renderable { Mesh*, ITexture*, Transform }`; `scene_io` (load/dump) and `commands` (undoable edits) round-trip the scene; `instance_grid` builds the deterministic instancing field.
- `streaming.h/.cpp` — distance-based scene / asset **streaming** (Slice BD), below.

The scene layer — and the `asset`, `anim`, `physics`, `material`, `ui`, `audio`, `terrain`, `game`, `runtime`, `editor`, and `debug` modules above the seam — has no knowledge of which backend is active. They call `IRHIDevice::Create*` through the seam, which is exactly why the backend-agnostic subset compiles into `hf_core` and runs clean under AddressSanitizer in the pure unit tests. This is the same code the Metal headless target uses to render its reference images.

### Scene / asset streaming (`engine/scene/streaming.*`)

`scene::StreamingWorld` (Slice BD) is **pure CPU above the seam** — zero `vk*`/`MTL*`/`mtl::`/`Backend::` symbols; it depends only on `engine/math` + the stdlib and is compiled into both `hf_core` (ASan-scoped, `streaming_test`) and `hf_engine` (the `--stream-shot` showcase). A large world is divided into a fixed deterministic `N×N` grid of **cells** laid out row-major on the `y=0` plane; `Update(cameraPos)` runs a three-step residency policy each frame:

1. **Desired residency** — a cell within `loadRadius` of the camera wants to be Resident; a cell beyond `unloadRadius` wants to be Unloaded; cells in the **hysteresis band** (`loadRadius < dist ≤ unloadRadius`, with `unloadRadius > loadRadius` enforced) **keep their current state**, which is what prevents load/unload thrash as the camera jitters around a boundary.
2. **Enqueue** to-load (`Unloaded→Loading`) and to-unload (`Resident→Unloading`) by **ascending distance** (nearest first).
3. **Process under a per-frame budget** — up to `loadBudgetPerFrame` loads finish (`Loading→Resident`, synchronously building the cell's procedural renderables) and `unloadBudgetPerFrame` unloads finish (`Unloading→Unloaded`, freeing them) this frame, so a large working set streams in **over** frames rather than in one hitch.

The resident set is a **pure function** of (camera position, radii, budget, prior state). A cell's content is described as backend-free `CellRenderable` data (a primitive `Kind` + model matrix + material + palette index), and the showcase maps that to real GPU meshes/textures — so the streaming module itself stays RHI-free and the unit test stays GPU-free. Driven by a fixed scripted camera path (no live input / RNG / clock) the per-frame `ResidentCellIds` sequence is bit-stable; the `--stream-shot` capture frame reports a fixed `frame:40, resident:24` line and is pinned by the `stream` golden. The "load" here is the **synchronous** construction of a cell's procedural renderables — the MVP models the budget/over-frames behavior, not async disk I/O (that is a future slice).

### Animation state machine — cross-fade FSM (`engine/anim/state_machine.*`)

`anim::StateMachine` (Slice BL) is a **parameter-driven finite state machine with cross-fade blending**, built **purely on top of** the existing skeletal-animation core (`anim::SampleAnimation` / `anim::BlendAnimations`). It is **CPU-only and produces a joint-matrix palette exactly like the existing sampler/blender already do**, so the **unchanged GPU skinning path consumes its output with no new RHI seam and no new shader** — zero `vk*`/`MTL*` symbols.

An `AnimState` is a named clip (an index into a caller-provided `std::vector<Animation>`) with a per-state playback `speed` and a `loop` flag; a `Transition` is a parameter-threshold-gated edge (`paramIndex cmp threshold`, `Cmp::Greater|Less`) with a cross-fade `duration`, where `from == -1` means an **any-state** edge. `Update(dt)` advances a deterministic cursor: when not transitioning it accrues `stateTime += dt·speed(current)` then evaluates the current state's outgoing edges (plus every any-state edge) **in the fixed order they were added — first satisfied wins** — and on a firing edge records the target, seeds `transitionTime = 0`, and starts both states' local times advancing; while transitioning both local times advance, `transitionTime += dt`, and at `transitionTime ≥ duration` the transition **completes** (current = target). `Evaluate()` turns the cursor into a palette: not transitioning → `SampleAnimation(current)`; transitioning → **`BlendAnimations(from, to, weight = transitionTime/duration)`**, reusing the established blend convention.

The FSM output (current state, transition progress, blend weight, final palette) is a **pure function of (state graph, scripted parameter timeline, fixed-dt sequence)** — no input / RNG / wall-clock — so it is bit-stable run-to-run and across backends. Unit-tested (`anim_fsm_test`) and golden-captured (`--anim-fsm-shot` / `--anim-fsm`, the `anim_fsm` golden), scripted to a fixed mid walk→run cross-fade: `state:walk->run, blend:0.53, speed:0.925, step:37`.

## Terrain Layer (`engine/terrain/`)

`engine/terrain/heightmap.*` (Slice BF) is a **procedural terrain generator** that is pure CPU (`engine/math` + the scene `Vertex`/index types + the stdlib only) with **no RHI or backend symbols**. Crucially it is compiled into `hf_core`/`hf_engine` **and** into the standalone `metal_headless` target **from the same translation unit**, so the Windows/Vulkan and Apple/Metal builds generate a **bit-identical** mesh — the cross-backend golden contract (same code → same mesh → same pixels).

- **`Height(x, z)`** is a fixed pure function: `A1*sin(f1·x)*cos(f1·z) + A2*sin(f2·x+P2x)*cos(f2·z+P2z) + A3*ValueNoise(x·nf, z·nf)`, where `ValueNoise` is a hash-lattice value noise (integer-lattice corner values from a 2D integer hash, bilinearly interpolated). It uses no RNG, no clock, and no float-time input, with the constants locked in `heightmap.cpp`, so it is deterministic and locale-independent.
- **`BuildTerrain(n, worldSize, heightScale)`** builds an `n×n` vertex grid over `[-worldSize/2, +worldSize/2]²` in the XZ plane, displacing each vertex by `y = Height(x,z) * heightScale`. **Per-vertex normals come from central finite differences** of `Height` (`N = normalize(cross(dz, dx))`); UVs are the grid coordinates in `[0,1]²`; tangent is `+X` (matching the `Plane`'s convention); and a deterministic height-based **vertex-color tint** (low → grass green, high → rock/snow) reads the relief through the **unchanged** lit shader (which multiplies texture × vertex color), so **no new shader is required**. Indices wind two CCW (viewed from above) triangles per quad so the surface is front-facing under back-face culling. The returned `TerrainMesh` also carries `peak` (max world-space Y) as the deterministic stat.

It is unit-tested (`terrain_test`, which re-implements the locked formula independently and checks structure / counts / index range / corner positions / displaced Y / finite-difference normals / two-call bit-identity) and golden-captured (`--terrain-shot` / `--terrain`, the `terrain` golden); the showcase reports a fixed `n:128, verts:16384, tris:32258, peak:2.0972` line. Because the same TU feeds both the Vulkan sample and `metal_headless`, the generated mesh — and therefore the golden — is identical on both backends.

### Streamed terrain + distance-banded LOD (`engine/terrain/terrain_stream.*`)

`terrain::TerrainStreamWorld` (Slice BJ) is the **open-world capstone** that ties Slice BD (distance-based streaming residency) and Slice BF (procedural heightmap terrain) into a streamed, LOD-selected terrain field. Like both parents it is **pure CPU above the scene** — zero `vk*`/`MTL*`/`mtl::`/`Backend::` symbols, depending only on `engine/math` + `engine/terrain/heightmap` + the stdlib — and is compiled into `hf_core` (ASan-scoped, `terrain_stream_test`), `hf_engine` (the `--terrain-stream-shot` showcase), **and** the standalone `metal_headless` target, so the tile meshes are bit-identical cross-backend.

- **Tile grid.** A fixed `T×T` grid of square **tiles** covers the world, the whole grid centered on the origin; tile `(i,j)` covers a `tileSize×tileSize` XZ region at a fixed world offset (`id = j*T + i`, row-major). Each tile is meshed by `BuildTerrainTile` over **that tile's** region using the **global** `Height(x,z)` — so adjacent tiles meeting at equal LOD share bit-identical edge vertices (seamless; no per-tile randomness).
- **Distance-banded LOD with hysteresis.** A tile's mesh resolution is one of three discrete levels chosen from its center's distance to the camera: `LOD0` (`n=96`, near), `LOD1` (`n=48`, mid), `LOD2` (`n=24`, far). `LodFor` is the pure `≤`-band function; `LodNext` adds **hysteresis** — it refines (toward LOD0) immediately as a tile approaches, but only **coarsens** once the distance is past the band edge by a margin, so a tile sitting on a boundary doesn't flicker LOD. Per-LOD vertex/index counts (`verts == n²`, `indices == (n-1)²·6`) are pinned by the unit test.
- **Residency policy.** `Update(cameraPos)` runs the BD load/unload-radius + hysteresis-band + per-frame-budget policy (nearest-first) to pick the resident tile set, then updates each resident tile's LOD, **rebuilding its mesh** on initial load or whenever its LOD changes.

The resident tile set **and** each tile's LOD are pure functions of (camera position, radii, budget, distance→LOD bands, prior state); driven by a fixed scripted camera path (no live input/RNG/clock) the per-frame `{resident tiles + LODs}` is bit-stable run-to-run and across backends. Unit-tested (`terrain_stream_test`: tile coverage + shared-edge seamlessness, the band/hysteresis functions, the per-LOD counts, residency+LOD determinism, refine-on-approach rebuild) and golden-captured (`--terrain-stream-shot` / `--terrain-stream`, the `terrain_stream` golden); the showcase reports a fixed `frame:45, resident:22, lod0:4, lod1:8, lod2:10` line.

## Material / Shader Graph Layer (`engine/material/`)

Phase 4 adds a **data-driven material system** that authors shaders as node graphs instead of hand-written HLSL, while preserving the "shaders are generated, not hand-written" and golden-stability invariants. It is pure host logic above the seam — no `vk*`/`MTL*`/`Backend` symbols — and is split into five concerns plus a build-time tool:

- **`shader_graph` (`shader_graph.h/.cpp`)** — the pure-CPU graph model: typed nodes (`Float`/`Float2`/`Float3`/`Float4` values on the edges) wired into the PBR fragment inputs through the `PBROutput` sink. The node set is `Constant`, `UV`, `TextureSample`, `Multiply`, `Add`, `Lerp`, and `Fresnel`, the **Slice-AZ expansion** — `Swizzle` (mask over `xyzw`/`rgba`), `MakeFloat3`, `MakeFloat4`, `Dot`, `Normalize`, `Power`, `OneMinus`, and `Saturate` — which together cover vector (de)construction, common math, and remapping, plus the **Slice-BE `NormalMap` node**, enough to author non-trivial materials as graphs. Output types are resolved structurally (`OutputType` propagates the matched input type through `Multiply`/`Add`/`Lerp`, etc.). No GPU types; fully unit-tested (`shader_graph_test`).

  - **The `NormalMap` node + `PBROutput.normal` (Slice BE).** The `PBROutput` sink gained a **5th input slot, `normal`** (tangent-space `float3`, default `(0,0,1)` = no perturbation), **appended last** so the original four inputs (`baseColor`/`metallic`/`roughness`/`emissive`) keep their order and semantics — any graph that leaves `normal` unconnected codegens **byte-identically** to before, which is why every pre-BE golden is unchanged. The `NormalMap` node samples a named normal-texture slot (default `"normalmap"`) at a UV input and outputs `normalize(decode(sample))` with `decode(c) = c*2-1` (the standard tangent-space unpack; the decode is shared with the codegen via `EvalNormalMap`). When (and only when) a graph drives `PBROutput.normal`, codegen emits a perturbed-normal lighting path — the **`hfShadePBRN` variant** of the shared PBR lighting core — which builds a TBN from the interpolated tangent/normal and rotates the decoded tangent-space normal into world space before lighting, instead of the flat-normal `hfShadePBR`. The `mat_normal` golden (`--material-normal`) pins it.
- **`codegen`** — lowers a validated graph to HLSL that drops into the existing `lit_pbr` fragment contract (same descriptor sets / push constants), so a graph material is just another PBR pipeline as far as the render graph and goldens are concerned.
- **`material_loader`** — parses the JSON authoring format (`assets/materials/*.mat.json`) into a `ShaderGraph`.
- **`runtime_compile` (`runtime_compile.h/.cpp`)** — the **runtime path**: it shells out to the `dxc` compiler as a **subprocess** (temp-file IO only; HLSL → SPIR-V), then builds an `IPipeline` from the result. It fails **safe** — returns `nullopt` + an error on any problem, never throws across the boundary — so a bad live edit cannot crash the host.
- **`live_material` (`live_material.h`)** — the in-editor authoring controller: it watches a material file and re-runs the loader → codegen → runtime compile chain on change, hot-swapping the pipeline for live iteration.

**Two production paths, proven equal.** A **build-time codegen tool** (`tools/material_codegen`) runs during the build to bake the showcase materials into committed generated HLSL (`shaders/generated/mat_showcase*.frag.hlsl`), which compile through the normal offline shader pipeline (the fast, shipping path). The **runtime** `runtime_compile` path generates and compiles the *same* graph live (the authoring path). The two are required to render **byte-identically**: `--material-shot` (build-time) and `--material-live-shot` (runtime) produce the same SHA, and the `mat_graph` / `mat_graph2` goldens pin both. This is the material analogue of the HLSL→MSL "generated, not hand-written" rule — the live authoring convenience can never silently diverge from what ships.

A **multi-material scene** (`--material-multi-shot` / `--material-multi`, the `mat_multi` golden) renders three spheres in a row, each shaded by a distinct graph material (`showcase` / `showcase2` / `showcase3`) with one draw per material binding that material's pipeline — demonstrating that several independently-authored graph materials coexist in a single frame, all through the same build-time-generated path.

### Graph introspection — JSON / DOT (`engine/material/graph_introspect.*`)

`material::DescribeGraphJson` / `ToDot` (Slice BI) close the loop on the data-driven material pipeline: having **authored** a graph (`*.mat.json` → `ShaderGraph`) and **rendered** it (codegen / runtime compile), introspection lets an agent — or a future visual node editor — **inspect** it. Over the same in-memory `Graph`/`Node`/`Edge` model the loader and codegen use, it emits a deterministic, machine-readable description of the graph: as pretty-printed **JSON** (`material`, `nodeCount`, `edgeCount`, the resolved `output` PBROutput-sink id, then per-node `id`/`type`/`params`/typed input `ports` with their source-or-default + resolved output type, then the `edges`) or as a **Graphviz DOT** digraph (one labelled node per graph node with the PBROutput sink highlighted, one labelled edge per connection) for visualization.

It mirrors the engine-state introspection (`editor::DescribeEngine`) but for material graphs, and shares its determinism discipline: **HARD RULE** no `vk*`/`MTL*`/`mtl::`/`Backend::` symbols — it is a pure string builder over the CPU graph model (the caller writes the strings), with nodes emitted in a **stable id-ascending order**, ports in declaration order, edges in stored order, floats formatted with the same `%g`-on-the-double convention the engine introspect JSON uses, and LF newlines — so output is byte-identical run-to-run and cross-platform. Exposed via `--material-introspect <mat.json> [out.json]`; unit-tested (`graph_introspect_test`) and pinned by a byte-exact JSON golden: `DescribeGraphJson(load(showcase3.mat.json))` byte-equals `tests/golden/material/showcase3_graph.json` (the test reads the committed golden and compares, so CI checks it too).

## Text / HUD Layer (`engine/ui/`)

`engine/ui/` (`text.h/.cpp`) is a **baked-font text renderer**: it bakes a fixed 8×8 monospace bitmap font (`kFont8x8`, the printable-ASCII range 0x20–0x7E, one `uint8_t[8]` per glyph) into a single RGBA atlas with `BuildFontAtlas`, and lays a string out into a batch of screen-space textured quads in NDC with `LayoutText`. It is **pure CPU with zero RHI / backend symbols** — no `vk*`/`MTL*`/`mtl::`/`Backend::` — and uses no external asset, clock, or RNG, so the same string always produces the same atlas and the same quad batch, and the HUD goldens match on every backend. It is compiled into **both** `hf_core` (ASan-scoped, unit-tested via `text_test`) and `hf_engine`.

The **draw side is additive**: it reuses the existing alpha-blend + sampled-texture paths through a **screen-space overlay pass** (`shaders/text.vert.hlsl` + `text.frag.hlsl`, an `alphaBlend` + `cullNone` pipeline). The atlas is a normal sampled texture and the quads are drawn over the scene *after* post-processing, alpha-blended — so the HUD composites without disturbing any pre-existing pipeline or golden. Two showcases pin it: `--hud-shot` / `--hud` (standalone text, the `hud` golden) and `--game-hud-shot` / `--game-hud` (the game sample with its score HUD overlaid, the `game_hud` golden).

## Audio Layer (`engine/audio/`)

`engine/audio/` is a **deterministic software audio engine** — `mixer.h/.cpp` (the mixer) + `wav.h/.cpp` (a 16-bit PCM WAV writer) — that, like `engine/physics` and `engine/math`, is pure C++ above the seam and compiled into `hf_core` (ASan-scoped, unit-tested) and `hf_engine`. It has **no RHI/graphics-backend symbols**.

The mix path is **integer / fixed-point end to end**: `Q15` (signed 1.15 fixed-point) gains, an `int32` accumulator, and a hard clamp to `int16` — there is **no `float`/`double` anywhere** in `Render()`'s sample loop. This is deliberate: floating-point mixing can differ in the last bit across compilers (MSVC vs Apple clang), which would break a byte-exact cross-platform WAV golden, whereas the integer path is **bit-identical on every platform and every run**. Voices are sine (integer phase accumulator → fixed `int16` wavetable), square (sign of the phase half), or noise (a per-voice integer LCG), shaped by a piecewise-linear `ADSR` envelope whose times are in samples and whose levels are Q15.

The **WAV writer** emits the standard 44-byte canonical PCM header (`RIFF`/`WAVE`/`fmt `/`data`, format 1, 16-bit, little-endian) followed by interleaved samples, serialized **byte by byte by hand** (never `memcpy` of a host-endian struct) and with no timestamps or other non-deterministic fields — so the same buffer always encodes to byte-identical bytes. `--audio-render out.wav` renders a fixed deterministic scene (fully headless: no window/GPU) and is pinned by a byte-exact golden, `tests/golden/audio/scene.wav` — the **audio WAV golden category** — verified on the Windows side (the bytes are identical on both platforms by construction).

## VFX Layer — CPU particle / emitter (`engine/vfx/particles.*`)

`engine/vfx/particles.*` (Slice CC) is an **authorable CPU particle / VFX emitter** — the VFX analogue of the data-driven material graph, and **distinct** from the fixed `gpu-particles` compute fountain (`shaders/particles.comp.hlsl`, one hard-coded GPU sim). Like `engine/audio` and `engine/physics` it is **pure C++ above the seam** with **zero `vk*`/`MTL*`/`mtl::`/`Backend::` symbols**, compiled into both `hf_core` (ASan-scoped, `vfx_test`) and `hf_engine` (the `--vfx-shot` showcase + the Metal `--vfx` path).

A data-driven `EmitterConfig` describes one emitter as plain values (no handles): `origin`, `spawnRate` (particles/sec, fractional accumulated), `lifetime`, `initVel` + spread, `gravity`, `drag`, a `maxParticles` pool cap (spawns past the cap are dropped, not queued), and **lifetime curves** — `size`-over-life and `color`-over-life — evaluated by each particle's normalized age. `ParticleSystem::Step(dt)` advances the pool at a **fixed timestep**: it integrates velocity (gravity + drag), retires particles past `lifetime`, and spawns new ones at the configured rate. `BuildBillboards()` turns the live pool into **world-space, camera-facing quad vertices** that the showcase uploads as a dynamic vertex buffer and draws as **additive alpha-blended billboards** — reusing the existing dynamic-textured-quad overlay infra (the same pattern the HUD/text overlay and transparency pass use), so it adds **no new RHI seam symbols**.

Determinism is the cross-backend golden contract: a particle emitter normally needs RNG (spawn jitter) and a wall clock (the sim). Both are removed — the velocity-spread jitter is drawn from a **fixed-seed integer LCG** (the same `1664525 / 1013904223` constants the audio Noise voice uses — no `std::rand` / `<random>`), and the sim runs a **fixed number of fixed-`dt`** `Step()` calls (no clock). So the same `EmitterConfig` + seed + step count yields identical particle state → identical billboards → byte-identical pixels on both backends, and two runs are bit-identical. Unit-tested (`vfx_test`) and golden-captured (`--vfx-shot` / `--vfx`, the `vfx` golden); the showcase reports a fixed `emitters:1, alive:960, spawned:1800`.

## Networking / Replication Layer (`engine/net/`)

`engine/net/snapshot.*` (Slice BQ) is a **deterministic state-replication core** — the networking depth pivot's first slice — and, like `engine/audio` and `engine/physics`, it is **pure CPU above the seam** with **zero RHI/graphics-backend symbols** (no `vk*`/`MTL*`/`mtl::`/`Backend::`), compiled into both `hf_core` (ASan-scoped, unit-tested via `replication_test`) and `hf_engine` (the live `--net-shot` showcase). It deliberately ships **no transport**: there are no sockets/UDP/TCP. The authority's serialized byte stream is fed straight into the replica through an in-process "perfect channel", so the focus is the **snapshot/delta wire format and its determinism**, not the network plumbing (a future slice can layer real transport on the same API).

- **Replicated state.** The minimal unit is `RepEntity { id, position, orientation, flags }`; the roll-game's player maps to one entity (its physics body pose) and each uncollected pickup to another, with `kFlagPlayer` / `kFlagPickup` bits so a renderer can tint without a side table. A full `Snapshot` stores its entities in a fixed ascending-`id` order, so the serialized bytes are deterministic.
- **Snapshot + delta.** The authority emits periodic full **keyframes** and, between them, **deltas** that encode only what changed since the last applied snapshot; the replica reconstructs **bit-identical** state by applying the keyframes + deltas in order. The wire format is fixed little-endian with a fixed field order; floats are written as their **raw IEEE-754 bit pattern** (LE `memcpy` of the 4 bytes) — because authority and replica are the *same build*, this round-trips a float bit-exactly (a future cross-machine slice can swap in fixed-point quantization behind the same API).
- **Determinism + the showcase.** The authority sim is the deterministic roll-game stepped at the fixed timestep, so the same run yields the same snapshots and the replica reconstructs identical state every time. `--net-shot` steps the authority, streams its bytes into the replica, asserts the replica matches the authority at the capture tick, and **renders the replica's reconstructed scene** through the existing lit/shadowed path (the `net` golden) — proving the replicated state is renderable, not just equal. It reports a fixed `replicaMatch:true, savings:43.5%` line (`fullBytes:37233, deltaBytes:21045`), the byte savings of delta replication over sending full snapshots every tick.

### Simulated transport + client interpolation (`engine/net/transport.*`)

Slice BU layers the **hard** part of networking on top of the BQ snapshot core — an **imperfect channel** and a **client jitter buffer with interpolation** — still **without any real sockets** (real UDP/TCP transport remains a future slice). Like `snapshot.*`, `transport.*` is **pure CPU above `engine/math` + `engine/net`** with **zero RHI/backend symbols**, compiled into both `hf_core` (ASan-scoped, unit-tested via `net_transport_test`) and `hf_engine` (the live `--netsim-shot` showcase).

- **`SimChannel` — a deterministic, seeded, socket-free channel.** `Send(tick, bytes)` enqueues an opaque serialized `Snapshot` with `deliverTick = tick + latencyTicks`, except a **seeded loss draw** `< dropRate` drops it (and a **reorder draw** `< reorderRate` perturbs its `deliverTick` by `+[2 .. 2+reorderSpread]` so it genuinely arrives behind the next in-stream packet). All randomness is a **pure integer LCG** (`lcg = lcg*1664525 + 1013904223`, Numerical-Recipes constants) seeded by `lossSeed` and advanced per `Send` — **no real RNG, no clock**, so the same `ChannelConfig` + the same `Send` stream yields a bit-identical delivered/dropped/reordered sequence. `Deliver(tick)` returns every queued packet whose `deliverTick <= tick` in delivery order (the queue is a `std::map` keyed by `(deliverTick, seq)`, so ordering is sorted + deterministic with stable `seq` tie-breaks). Counters (`Sent`/`Delivered`/`Dropped`/`Reordered`) feed the showcase stat line + tests.
- **`ClientView` — jitter buffer + interpolation.** Delivered packets are deserialized and buffered keyed by tick (`std::map<int, Snapshot>`; out-of-order/duplicate ticks are fine, newest wins, `latestReceivedTick` tracks the max). The client renders at `renderTick = latestReceivedTick − interpDelay` (default **2 ticks** behind the leading edge) and produces an **interpolated** snapshot via BQ `Interpolate` (lerp position + nlerp/slerp orientation) between the two buffered ticks that **bracket** `renderTick`. Because the render point trails by ≥2 ticks, a single **dropped** snapshot is a one-tick hole spanned by its neighbours and interpolated **across** — the smoothing hides the drop, no pop. The documented **edge policy** is **hold-last**: at/above the newest buffered tick it holds the newest (never **extrapolates** past the last received snapshot, so it is bounded and never drifts); at/below the oldest it holds the oldest.
- **The showcase.** `--netsim-shot` runs the AX roll-game as the authority over the full scripted track, sends each tick's serialized snapshot through a fixed lossy `SimChannel` into a `ClientView`, and **renders the client's interpolated scene** through the existing lit/shadowed path (the `netsim` golden). It reports a fixed `delivered:226, dropped:42, reordered:29, converged:true` (`converged` asserts the interpolated client tracks the authority despite the lossy channel). Two-run byte-identical (seeded LCG + fixed config).

### Client prediction + server reconciliation (`engine/net/prediction.*`)

Slice BY completes the networking trilogy (`BQ replication → BU transport+interp → BY prediction+reconcile`). Interpolation (BU) trades **latency** for smoothness — it renders the *remote* view a few ticks in the past. Prediction does the opposite for the **local** player: it shows your own input **immediately** instead of waiting a full round-trip. Like `snapshot.*`/`transport.*`, `prediction.*` is **pure CPU above `engine/math` + `engine/net` + the roll-game sim** with **zero RHI/backend symbols** (the seam baseline stays at the 2 `rhi_factory` lines), compiled into both `hf_core` (ASan-scoped, unit-tested via `prediction_test`) and `hf_engine` (the live `--netpredict-shot` showcase).

- **`PredictedClient` — predict ahead, then reconcile.** `PredictTick(input)` applies a local `InputCmd` to the client's predicted copy of the sim **right away**, advancing ahead of the last server-acknowledged frame and retaining the input in an ordered buffer. When an authoritative frame arrives, `OnAuthoritative(authState)` **rewinds** the client to that acknowledged state and **replays** every still-unacknowledged buffered input on top of it, so the corrected predicted state re-converges to where the authority will be.
- **`AuthState` carries the full `RigidBody`.** For the rewind/replay to be **bit-exact**, the authoritative frame can't carry only the replicated snapshot pose — it carries the player's **full `RigidBody`** (position **and velocity**), so replaying the buffered inputs from the rewind point reproduces the identical trajectory (a velocity-free rewind would diverge on the first integrated step).
- **A real misprediction, then a correction.** The test/showcase is deliberately non-trivial: the authority runs the same sim **plus a scripted server-only impulse** the client has no way to predict, manufacturing a **genuine misprediction** (`maxMisprediction > 0`) that reconciliation then corrects back to the authority's true state. Determinism holds because the scripted inputs + scripted server-only impulse + the seeded BU channel + fixed latency are all bit-stable.
- **The showcase.** `--netpredict-shot` / `--netpredict` drives authority (with the server-only impulse) → a delayed BU channel → the predicted+reconciled client, and **renders the reconciled client scene** lit + shadowed (the `netpredict` golden). It reports `predTicks:251, reconciles:247, maxMisprediction:0.0655, finalError:0.0250, converged:true` — byte-identical cross-backend (shared pure-C++ core), two runs deterministic.

## Editor Layer (`engine/editor/`)

Beyond the deterministic logic modules (`picking`, `gizmo`, `introspect`), the editor layer hosts the **docked ImGui editor shell** (Slice BT): a Scene Hierarchy / Inspector / Stats UI laid out around a central scene Viewport.

- **The pure-CPU panel-data seam (`editor/editor_panel_data.*`).** The architectural move here is the same one used throughout the engine: a **deterministic, backend-free, ImGui-free seam** between *what each panel displays* and *the Dear ImGui calls that draw it*. `BuildPanelData(registry, resources, state)` computes the three panels' data — `HierarchyRow`s (one per drawable entity, label = mesh name + view-order index), an `InspectorData` view of the selected entity (`TransformC`/`MaterialC`/`MeshC`), and `StatsData` counts — touching **only** the ECS `Registry`, opaque `SceneResources` named pointers (never dereferenced — the same contract as `introspect`/`scene_io`), and `math`. It carries **zero** `vk*`/`MTL*`/`rhi`/ImGui symbols, so it lives in `hf_core` and is unit-tested **headlessly** (`editor_panels_test`) by asserting the data, not pixels. The persistent `EditorState.selectedEntity` is **clamped into range and written back** each call (matching the live `--fly` behaviour), so two calls on an identical registry/state yield byte-identical `PanelData`.
- **Deterministic, code-driven docked layout.** The vendored Dear ImGui is **1.91.8, the non-docking master branch — there is no `DockBuilder` API**. So the docked layout is **not** loaded from a machine-dependent `imgui.ini` (`io.IniFilename = nullptr`); instead `editor::DockLayout` describes it as **fixed split ratios + fixed dock-node ids** (`leftRatio=0.22`, `rightRatio=0.28`, `leftBottomRatio=0.45`, with the Viewport in the central node, Hierarchy + Stats in the left column, Inspector on the right). Because the layout is a deterministic **value** and ImGui geometry is CPU-built, the panel geometry is **byte-stable run-to-run and identical across backends** — which is exactly what makes a golden image of an ImGui UI possible. `BuildEditorUI` (`editor_panels.*`) issues the ImGui widgets from the `PanelData`; the RHI-only `ImGuiRenderer` (`imgui_renderer.*`) draws ImGui's vertex/index/command buffers through the seam (no backend symbols), so the **same** editor chrome renders on Vulkan (`--editor-shot`) and Metal (`--editor`).
- **The showcase.** `--editor-shot [objIndex]` / `--editor` renders the default Slice-F scene with the docked editor chrome over it and a FIXED selected entity, captures (the `editor` golden), and prints a deterministic, two-run byte-identical `editor: {selected:9, entities:10}` stat line.

### Live edit-ops → scene_io round-trip (`engine/editor/edit_ops.*`)

Slice BX closes the editor's **author → save → reload** loop. An `EditOp` (e.g. translate, set-material) **mutates the live ECS `Registry` in place**, then the edited scene is **serialized through the existing `scene_io` writer** and **reloaded** into a fresh registry, and the reloaded state is asserted **bit-identical** to the in-memory edited state — proving an edit *survives a save/load round-trip exactly* (no silent loss or reordering across serialization). It follows the same backend-free seam discipline as `editor_panel_data`/`introspect`: it touches **only** the ECS `Registry`, the opaque `scene_io` text format, and `math`, carrying **zero** `vk*`/`MTL*`/`rhi`/ImGui symbols, so it lives in `hf_core` and is unit-tested **headlessly** (`edit_ops_test`) by asserting the registry/round-trip equality, not pixels. The deterministic edit script makes the result two-run byte-identical. Golden-captured (`--editor-edit-shot` / `--editor-edit`, the `editor_edit` golden); the showcase applies a fixed edit script and reports `editor-edit: {edits:2, entity:8, saved:true, reloadMatch:true}`.

## The Agent Experience (AX) product (`engine/editor/introspect.*`, `engine/scene/commands.*`, `engine/scene/scene_io.*`, `engine/runtime/hot_reload.*`, `engine/game/verdict.h`)

The engine's deepest differentiator is not any single subsystem — it is that **everything is deterministic, headless,
and golden-verifiable**, which makes the engine *drivable by an agent* in a way a GUI-bound, stateful, non-deterministic
engine structurally cannot match. This product layer turns that latent property into an explicit, versioned **Agent
SDK**: a stable contract an agent codes against, plus the read / author / reload / replay surfaces it needs — every
one of them proven by the engine's own byte-exact golden harness (the product dogfoods the determinism moat). All six
surfaces are pure-CPU and backend-agnostic, so their goldens are byte-compared once on the host (no per-backend bake),
exactly like the engine-state JSON.

- **The versioned contract.** `--introspect` already emitted a machine-readable dump of the whole engine state; the
  AX layer adds an `"agentApi"` block (`editor::DescribeAgentApi`) that is the SDK contract: a `schemaVersion`, the
  enumerated `capabilities` (`observe` / `mutate` / `author` / `replay` / `hot-reload`), the command protocol surface
  (each op with its capability and a typed argument schema), and a `contentHash`. The crucial design choice is that
  **`schemaVersion` governs SHAPE only** — the structure of the contract — while a `contentHash` (a stable FNV-1a over
  the volatile showcase/feature/command lists) summarizes the *content*. So as later work keeps adding showcases, an
  agent pinned to `schemaVersion:1` sees the `contentHash` move (cheap drift detection) without the contract structure
  ever churning. This property held across all six slices of this layer: each new capability changed the `contentHash`
  and left `schemaVersion` at 1. `--agent-api` emits just that block; it is byte-golden (`agent_api.json`), and it is
  byte-identical to the same sub-object inside the full `--introspect` (`default_scene.json`).
- **The query protocol (read).** The scriptable command bus (`scene::RunCommands`) gains a `query` op — a selective,
  addressable, deterministic JSON read of one entity (optionally narrowed to named fields, emitted in a fixed
  canonical order), the scene stats, or the ordered entity list — addressed by the same view-order index the mutate
  ops use, with an out-of-range index returning a deterministic error object rather than crashing. `RunQueries`
  returns a request/response array (`query_responses.json`), the read half of the SDK's request/response loop.
- **The declarative authoring loop.** `scene::CanonicalizeScene` composes the existing scene loader and writer into a
  round-trip: a declarative scene **spec** (which may be written non-canonically — keys in any order, optional fields
  omitted) is loaded and re-emitted in the engine's single **canonical** form. The guarantee an agent codes against is
  a **fixed point**: canonicalizing the canonical output yields byte-identical text, so there is exactly one canonical
  scene text for a given scene, reached in one step (`authored_scene.json`); a spec naming an unknown resource is
  rejected deterministically.
- **The deterministic hot-reload.** `runtime::ApplyReload` lifts the live editor's file-watch-and-swap into a headless,
  golden form: load scene A, observe the watched path "change" to B through the `FileWatcher` (whose stat source is
  **injectable**, so change-detection is deterministic with no real-filesystem timing race), reload, and assert the
  reloaded world is **byte-identical to a cold load of B** — no stale residue from A survives (`reload_trace.json`).
  "Reload(B) is indistinguishable from a fresh process that loaded B" is the load-equivalence guarantee.
- **Record → replay → assert-determinism (the moat productized).** Built directly on the frozen lockstep/snapshot
  primitives of the gameplay/netcode layer: `DigestSnapshot` fingerprints the whole heterogeneous world in its
  identity-keyed canonical order (stable under entity-handle churn), and a **replay file** (`canonical.replay`) stores
  a command stream plus the expected final-world digest. `--replay-verify` re-runs the stream and asserts the digest
  is bit-identical — and a one-command tamper moves the digest, so the replay is a genuine regression artifact.
  "Replay this stream → bit-identical" is a guarantee only a deterministic sim can offer.
- **The determinism-stress fuzzer (the capstone).** `RunDeterminismStress` sweeps every rollback/snapshot point over
  a fixed stream and asserts that rolling back at *any* point recovers the authority world bit-exactly
  (`allCorrected == points`), emitting a per-point PASS matrix (`stress_report.json`). A divergence flag — true only
  where an injected misprediction genuinely perturbs the world, false in the no-perturbation control — keeps the sweep
  non-vacuous. A determinism regression anywhere in the rollback machinery flips a `corrected` bit and breaks the
  golden. It is the deterministic, golden-locked equivalent of a fuzzer: bounded, seeded, reproducible.

The differentiator the architecture earns: **a versioned, golden-verifiable Agent SDK — author a scene as text and get
back the same bytes, replay a command stream and get back the same world, stress every rollback point and prove it
recovers — an agent-first product surface a stateful, non-deterministic engine cannot expose as a guarantee.**

## The validation-clean invariant

As of the render-graph barrier work, **Vulkan-validation-cleanliness is a permanent gate**, not a one-off check. Every `--*-shot` showcase is run under the **Khronos validation layer** with both **core** and **synchronization** validation enabled (`VK_LAYER_KHRONOS_validation`, `VK_LAYER_PATH` pointed at the layer's package bin dir, sync validation switched on). The bar is **zero** `VUID-*`, `SYNC-HAZARD-*`, `UNASSIGNED-*`, or `[ERROR]` lines across all showcases. The synchronization-validation half is specifically the oracle for the render-graph barrier solver: if the solver ever placed a missing or wrong barrier, sync validation would flag a hazard. The only tolerated diagnostics are benign `[WARNING: Performance]` notices — notably the depth-only shadow pipelines binding a full vertex layout produce "vertex attribute at location N not consumed by vertex shader" — which are expected and documented, not errors.

The validation layer is **not** a link-time dependency; it is pulled in via Conan (`vulkan-validationlayers` in `conanfile.py`, debug builds) purely to ship `VkLayer_khronos_validation.{dll,json}` so the gate can actually load the layer on a box where it is not installed system-wide. Without that dependency the oracle would be silently inactive and a validation regression could slip through unnoticed.
